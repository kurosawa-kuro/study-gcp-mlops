## 段階的 PR 分割戦略

一気に 30 ファイル動かすと PR レビュー不能 + 衝突祭りになるので、**6 つの独立 PR** に分割します。各 PR は単独で `pytest` と `terraform validate` が通る状態を維持します。

### PR #1: 足場作り (低リスク・破壊なし)

新ディレクトリとワークスペース登録のみ。**既存コードは一切動かさない**。

**作業内容:**
- `ml/` ディレクトリとサブフォルダを全作成 (空の `__init__.py` 配置)
- 各 `pyproject.toml` を新規作成 (依存は現状の `jobs/pyproject.toml` / `pipelines/pyproject.toml` からコピー)
- ルート `pyproject.toml` の `[tool.uv.workspace].members` に `ml/embed`, `ml/train`, `ml/serve`, `ml/sync` を追加
- `[tool.pytest.ini_options].pythonpath` と `testpaths` を新パス込みで更新
- `uv lock` 再生成

**検証:** `uv sync --all-packages --dev` が通る。既存 224 tests は全通過。

### PR #2: 独立性の高い周辺から移動

依存が少ない `meili_sync` と `pipeline_trigger` から着手。

**作業内容:**
- `jobs/src/meili_sync/entrypoints/sync_cli.py` → `ml/sync/src/sync/meili_sync.py`
- `jobs/pyproject.toml` の `[project.scripts]` から `meili-sync = "meili_sync.entrypoints.sync_cli:main"` を削除
- `ml/sync/pyproject.toml` に `meili-sync = "sync.meili_sync:main"` を追加
- `functions/pipeline_trigger/` → `ml/trigger/`
- `infra/modules/vertex/main.tf` の Cloud Function source 参照パスを `functions/pipeline_trigger/` → `ml/trigger/` に変更
- `tests/functions/test_pipeline_trigger.py` の import を更新

**検証:** `pytest tests/functions/` が通過、`terraform validate` 成功、`gcloud functions deploy` 相当の source zip が新パスから生成できる。

### PR #3: serve/ 分離 (reranker 推論サーバー)

最も独立性が高い推論サーバー。`training` package から切り離す。

**作業内容:**
- `jobs/src/training/entrypoints/reranker_server.py` → `ml/serve/src/serve/reranker_server.py`
- `jobs/containers/reranker/` → `ml/serve/container/`
- Dockerfile 内の `COPY` パスを新構造に合わせる
- `.github/workflows/deploy-reranker-image.yml` の trigger paths を `ml/serve/**` + `common/src/common/schema/**` に変更
- `ml/serve/pyproject.toml` に LightGBM + FastAPI 依存を記述
- `scripts/checks/layers.py::RULES` から旧 `training.entrypoints.reranker_server` エントリを削除、新 `serve.reranker_server` を追加

**検証:** `deploy-reranker-image.yml` のローカル再現 (または `docker build` 単体) が成功、`tests/arch/test_import_boundaries.py` 通過。

### `scripts/checks/layers.py::RULES` の書き換え例

現状 (抜粋):
```python
RULES = {
    "jobs/src/training/entrypoints/rank_cli.py": {
        "allowed": ["common", "training"],
        "banned": ["app"],
    },
    "jobs/src/training/services/rank_trainer.py": {...},
    "pipelines/src/pipelines/property_search/train_pipeline.py": {...},
    ...

```

移行後:
```python
RULES = {
    "ml/train/src/train/cli.py": {
        "allowed": ["common", "train"],
        "banned": ["app", "embed", "serve"],
    },
    "ml/train/src/train/trainer.py": {...},
    "ml/train/pipeline/train_pipeline.py": {...},
    "ml/embed/src/embed/cli.py": {...},
    "ml/embed/server/encoder_server.py": {...},
    "ml/serve/src/serve/reranker_server.py": {...},
    ...
}
```

新たに **embed ↔ train ↔ serve 間の相互依存を banned にする** 境界ルールを追加することで、将来の混線を AST 検査で防げます。

## リスクと緩和策

| リスク | 緩和策 |
|---|---|
| Vertex CPR Dockerfile の `COPY` が壊れて Endpoint が起動しない | PR #3, #4 で `docker build` + local run を必ず検証してから merge |
| `common/embeddings/` 参照が app 側に残っていて `/search` が ImportError | grep で事前確認 (現時点の app は `VertexEndpointEncoder` 経由で Endpoint 呼出なので直接依存はないはずだが要確認) |
| Terraform `functions/pipeline_trigger` パス変更で Cloud Function source が再 zip → 意図しない再デプロイ | 実 GCP 未投入の今なら影響なし。投入後に移行するなら `terraform plan` で差分確認 |
| `deploy-*-image.yml` の trigger paths 誤記で CI が発火しない | PR merge 後に dummy commit で確認 |
| 224 tests の中に旧パス文字列をハードコードしてるものがある | PR #6 で grep + 個別対応 |
