# 仕様と設計

## 共通前提

- **目的**: GCP Vertex AI プリミティブを使った MLOps パイプライン構築の技術検証・ベストプラクティス確立
- **題材**: 不動産ハイブリッド検索 (クエリ文 + フィルタ → 物件ランキング上位 N 件)
- **入力**: `raw.properties` (BigQuery) に物件マスタが入っている前提。行動ログは `mlops.search_logs` / `mlops.feedback_events` を Pub/Sub 経由で流し込む
- **担当範囲**: 特徴量加工 → 埋め込み生成 → 候補抽出 + 再ランク → サービング → 監視 → 再学習
- **開発環境**: WSL
- **本番環境**: GCP (`mlops-dev-a` / `asia-northeast1`)
- **言語**: Python 3.12
- **API**: FastAPI + Pydantic (Vertex Endpoint のプロキシとして動作)
- **IaC**: Terraform 1.9+
- **シークレット管理**: Doppler (値投入のみ、Terraform は container 管理、当面未配線)
- **CI/CD**: GitHub Actions + Workload Identity Federation

## `vertex` — Phase 3 bq-first への Vertex AI レイヤ後付け

### コンセプト

- Phase 3 (`bq-first`) の **検索コア (Meilisearch BM25 + BigQuery `VECTOR_SEARCH` + RRF + LightGBM LambdaRank)** を Port/Adapter 経由で継承
- encoder / reranker の推論を **Vertex AI Endpoint** へ外部化、学習オーケストレーションを **Vertex AI Pipelines (KFP)** へ移行
- Model 管理は **Vertex AI Model Registry**、特徴量は **Vertex Feature Group** (BQ Dataform を wrap)、監視は **Vertex Model Monitoring v2**
- **差分工事**: Phase 3 の純粋ロジック (`services/ranking.py::run_search` の RRF → rerank → top-K 骨格) は書き換えない
- BigQuery `VECTOR_SEARCH` と Meilisearch on Cloud Run は **Phase 3 から継続** (Matching Engine 化は別スコープ)

### アーキテクチャ

```
raw.properties (upstream ETL)
  └─ Dataform ─> feature_mart.properties_cleaned
                 feature_mart.property_features_daily  (ctr / fav_rate / inquiry_rate)
                                 + assertions (quality / freshness)
                                 └─> Vertex AI Feature Group (property_features × 7)

  └─ Vertex AI Pipelines `embed_pipeline` (日次 03:30 JST)
       load_properties → ModelBatchPredictOp(encoder Endpoint) → write_embeddings
       └─ feature_mart.property_embeddings (768d FLOAT64 REPEATED + VECTOR INDEX)

  └─ Vertex AI Pipelines `train_pipeline` (週次 日曜 04:00 JST / アラート起動)
       load_features → (Vizier 分岐) → train_reranker → evaluate
                                                        └─ dsl.Condition(pass) → register_reranker
       └─ Vertex AI Model Registry (property-reranker, staging → production alias + canary)
       └─ mlops.training_runs (NDCG@10 / MAP / Recall@20 ※ Registry と dual write)

        └─ Cloud Run Service `search-api` (FastAPI、Vertex Endpoint プロキシ、2Gi)
          ├─ /search   — VertexEndpointEncoder.embed → Meilisearch(BM25) + VECTOR_SEARCH → RRF fusion
          │             → build_ranker_features → VertexEndpointReranker.predict (Phase 6+)
          │             └─ Pub/Sub "ranking-log"    ─> BQ mlops.ranking_log
          ├─ /feedback └─ Pub/Sub "search-feedback" ─> BQ mlops.feedback_events
          └─ /jobs/check-retrain
              (Cloud Scheduler 04:00 JST → retrain-trigger → Cloud Function pipeline-trigger)

        └─ Cloud Run Service `meili-search`
          └─ GCS FUSE mount `/meili_data` (index 永続化)

        └─ Vertex AI Endpoints × 2
          ├─ property-encoder-endpoint  (CPR container、ME5 checkpoint を AIP_STORAGE_URI から load)
          └─ property-reranker-endpoint (CPR container、model.txt を GCS から load)

  └─ Vertex AI Model Monitoring v2 (Endpoint に attach、日次)
       FeatureDriftSpec (property-side 7 列) + PredictionOutputDriftSpec
       └─ Pub/Sub "model-monitoring-alerts" → BQ mlops.model_monitoring_alerts
            └─ Eventarc → Cloud Function pipeline-trigger → train_pipeline(enable_tuning=True)

  └─ monitoring/validate_feature_skew.sql (バックアップ)
       Scheduled Query として保持、Vertex Monitoring 停止時のフォールバック
```

### 特徴量加工層

- **ツール**: Dataform (Phase 3 から継続)
- **方式**: SQL + JavaScript、GitHub 連携でバージョン管理
- **スケジュール**: Dataform scheduled executions (日次 03:00 JST)
- **品質保証**: `assertions` で NULL 率・範囲 (ctr / fav_rate / inquiry_rate) チェック + freshness チェック
- **BQ テーブル設計**: partition (`event_date`) + clustering (`property_id`) 必須
- **Vertex Feature Group**: `property_features_daily` を wrap し、property-side 7 列 (`rent` / `walk_min` / `age_years` / `area_m2` / `ctr` / `fav_rate` / `inquiry_rate`) を登録。クエリ時算出の `me5_score` / `lexical_rank` / `semantic_rank` は非登録
- **Training-Serving Skew 対策**: 学習側 SQL と推論側 Python + infra + Feature Group を **6 ファイル同一 PR 原則** で lockstep 維持

### 埋め込み層

- **モデル**: `intfloat/multilingual-e5-base` (768d、fp32 約 1.1GB)
- **実行基盤**:
  - **物件側バッチ**: Vertex AI Pipelines `embed_pipeline` の `ModelBatchPredictOp` が encoder Endpoint を呼ぶ
  - **クエリ側オンライン**: Cloud Run Service `search-api` が `VertexEndpointEncoder` 経由で encoder Endpoint を呼ぶ
- **プロンプト規約**: `"passage: ..."` / `"query: ..."` の prefix を必ず付ける (ME5 の正しい利用法)
- **モデル配置**: GCS `gs://mlops-dev-a-models/encoders/multilingual-e5-base/{version}/` → CPR 起動時 `AIP_STORAGE_URI` から download → sentence-transformers ロード
- **初回投入**: KFP 外の 2 段手動セットアップ (`scripts/local/setup/upload_encoder_assets.py --apply` → `scripts/local/setup/setup_encoder_endpoint.py --apply`)
- **差分更新**: `property_embeddings.text_hash` (sha256(title+description)) が変わった行だけ再エンコード (Phase 3 の logic をそのまま移植)

### 候補抽出層

- **Stage 1a lexical**: Meilisearch (BM25) で top-200 抽出 (Phase 3 から継続)
- **Stage 1b semantic**: BigQuery `VECTOR_SEARCH` + 属性フィルタで top-200 抽出 (Phase 3 から継続。Matching Engine 化は別スコープ)
- **Stage 2 fusion**: Python 側 RRF (k=60) で rank list を融合し top-100 を確定
- 物件側フィルタ: 家賃上限 (`max_rent`) / 間取り (`layout`) / 徒歩分 (`max_walk_min`) / ペット可 (`pet_ok`) / 築年数 (`max_age`)
- VECTOR INDEX: `CREATE VECTOR INDEX ... OPTIONS(index_type='IVF', distance_type='COSINE')` (Terraform 未対応のため手動 DDL; `docs/04_運用.md STEP 16`)

### 再ランク層

- **モデル**: LightGBM `lambdarank` (group = `request_id`)、Vertex AI Endpoint `property-reranker-endpoint` にサーブ
- **特徴量** (10 次元): `rent` / `walk_min` / `age_years` / `area_m2` (物件基本) + `ctr` / `fav_rate` / `inquiry_rate` (行動統計 28 日) + `me5_score` / `lexical_rank` / `semantic_rank` (クエリ時シグナル)
- **ラベル Gain**: `inquiry=3 / favorite=2 / click=1 / none=0` (強いアクションが勝つ)
- **評価指標**: NDCG@10 (主)、MAP、Recall@20
- **ハイパラ**: `num_leaves=31 / learning_rate=0.05 / min_data_in_leaf=50 / lambdarank_truncation_level=20 / ndcg_eval_at=[5,10,20]` (Vizier `enable_tuning=True` 時は探索)
- **推論経路**: `search-api` の `VertexEndpointReranker.predict(instances)` が Endpoint を叩く。特徴量作成は API 側 (`common/feature_engineering.py::build_ranker_features` と lockstep)
- **Phase 4 rerank-free MVP**: Endpoint がカナリア未投入でも `/search` は候補抽出結果 (`final_rank = lexical_rank`) を返せる

### サービング層

- **ランタイム**: Cloud Run Service (Gen2) `search-api` (Vertex Endpoint プロキシとして機能)
- **フレームワーク**: FastAPI + Uvicorn
- **モデル配置**: コンテナ同梱せず、lifespan で `VertexEndpointEncoder` / `VertexEndpointReranker` クライアントを DI するだけ (torch / lightgbm 依存は API workspace から削除)
- **Cloud Run 設定**:
  - `--cpu 2 --memory 2Gi` (encoder / reranker が Vertex Endpoint へ外部化されたため Phase 3 の 4Gi → 2Gi に縮小)
  - `--concurrency 80`
  - `--min-instances 1` (コールドスタート回避)
  - `--max-instances 10`
  - `--cpu-boost`
  - `--execution-environment gen2`
  - `--no-allow-unauthenticated`
- **Dockerfile**: マルチステージ + `uv` パッケージマネージャ
- **認証**: IAM ベース、サービスアカウント分離 (9 SA + WIF = 10)
- **エンドポイント**:
  - `POST /search` — encoder Endpoint → Meilisearch(BM25) + `VECTOR_SEARCH` → RRF fusion → (Phase 6+) reranker Endpoint → 上位 N 件
  - `POST /feedback` — click / favorite / inquiry イベント publish
  - `POST /jobs/check-retrain` — Cloud Scheduler が 04:00 JST に叩く
  - `GET /healthz` `/readyz` `/livez` (Cloud Run 予約回避用に `/livez` alias を用意)

### CPR (Custom Prediction Routine) コンテナ

2 本の FastAPI サーバを実装 (`AIP_HEALTH_ROUTE` / `AIP_PREDICT_ROUTE` / `AIP_HTTP_PORT` 準拠):

- **encoder** (`pipeline/data_job` + `ml/data` 実装)
  - 起動時 `AIP_STORAGE_URI` から ME5 checkpoint を download → `E5Encoder.load` (`ml/data` 実装を再利用)
  - 入力 `{"instances": [{"text": "...", "kind": "query"}]}` → 出力 `{"predictions": [[768d]]}`
- **reranker** (`ml/serving` 実装)
  - 起動時 `model.txt` を GCS から download → `lgb.Booster(model_file=...)`
  - 入力 10 次元 float / 出力 scalar score

### KFP パイプライン

- **embed_pipeline** (日次): `load_properties → batch_predict_embeddings → write_embeddings`
  - text_hash 差分で再エンコード対象を絞る (Phase 3 `embedding-job` ロジックを component として移植)
  - `ModelBatchPredictOp` で encoder Endpoint を呼ぶ
- **train_pipeline** (週次 + アラート起動): `load_features → (vizier 分岐) → train_reranker → evaluate → dsl.Condition(pass) → register_reranker`
  - `train_reranker` は `ml/training/trainer.py::train` を container_component で wrap
  - `evaluate` で NDCG@10 閾値 PASS なら Model Registry 登録 + Endpoint カナリアデプロイ (`traffic_split={new:10, current:90}` → promote で 100%)

### 監視層

- **ランキング / フィードバックログ**: Pub/Sub → BigQuery Subscription (Phase 3 から継続)
- **ドリフト検知**: Vertex AI Model Monitoring v2 が Endpoint に attach、FeatureDriftSpec (property-side 7 列) + PredictionOutputDriftSpec を日次評価、Pub/Sub `model-monitoring-alerts` に通知
- **バックアップ**: `monitoring/validate_feature_skew.sql` の mean-drift 実装を Scheduled Query として保持 (Vertex Monitoring 停止時のフォールバック兼 SQL アドホック可用性)
- **可視化**: Looker Studio (無料版) + Vertex AI コンソール (lineage / ML Metadata)
- **アラート**: Cloud Monitoring のログベースメトリクス (5xx レート / p95 レイテンシ)
- **精度追跡**: `mlops.ranking_log` + `mlops.feedback_events` を join してオフライン NDCG を再計算、Vertex Experiments にも dual write

### 再学習トリガー層 (3 経路、すべて Cloud Function `pipeline-trigger` に集約)

- **(a) 日次スケジュール**: Cloud Scheduler (04:00 JST) → `/jobs/check-retrain` → Pub/Sub `retrain-trigger` → Eventarc → Cloud Function → `train_pipeline(enable_tuning=False)`
  - トリガー条件 (OR):
    1. `feedback_events` の新規行 > `NEW_FEEDBACK_ROWS_THRESHOLD=10_000` (直近 7d)
    2. 精度低下: 現在の NDCG@10 が 1 週間前から `NDCG_DEGRADATION=0.03` 絶対値で低下
    3. 定期: `STALE_DAYS=7` 日以上学習が走っていない
- **(b) ドリフト起動**: Vertex Monitoring 閾値超え → Pub/Sub `model-monitoring-alerts` → Eventarc → Cloud Function → `train_pipeline(enable_tuning=True)` (Vizier ON)
- **(c) セーフティネット**: PipelineJobSchedule (cron) — `embed_pipeline` 03:30 JST 日次 / `train_pipeline` 04:00 JST 日曜

### 実験管理

- **Vertex AI Experiments**: Registry / Pipelines / ML Metadata と自動連携、コンソールで lineage 追跡
- **W&B**: 可視化・アドホック比較 (Phase 1〜3 から継続)
- **Dual write**: `ml/data/loaders/ranker_repository.py::save_run` が BQ `mlops.training_runs` INSERT + `aiplatform.log_params` / `aiplatform.log_metrics` を同時に実行

### CI/CD

- **トリガー**: GitHub push to main
- **チェック**: `ruff` + `ruff format --check` + `mypy strict` + `pytest` (並列実行)
- **ビルド**: Docker → Artifact Registry (api image / encoder CPR image / reranker CPR image / 旧 training / embedding job image は段階削除予定)
- **デプロイ**:
  - `app/**` / `common/**` 変更 → `gcloud run deploy search-api`
  - `ml/embed/pipeline/**` / `ml/train/pipeline/**` / `ml/pipeline_utils/**` 変更 → KFP コンパイル + Pipeline Template 登録
  - `ml/embed/**` 変更 → encoder CPR image 再ビルド + `gcloud ai endpoints deploy-model`
  - `ml/train/**` 変更 → trainer image 再ビルド (KFP `train_reranker` component が picked up)
  - `ml/serve/**` 変更 → reranker CPR image 再ビルド + `gcloud ai endpoints deploy-model`
  - `definitions/**` 変更 → Dataform API へ compilationResults POST
  - `ml/trigger/**` 変更 → `gcloud functions deploy`
  - `infra/**` 変更 → `terraform apply` (main マージ時)
- **認証**: すべて WIF (SA Key 禁止)
- **シークレット注入**: Doppler → Cloud Run 環境変数 (未配線、当面は `--set-env-vars` 直書き)

### 採用 / 不採用の根拠

- ✅ **Vertex AI Pipelines (KFP)** — Cloud Run Jobs からの置換、ML Metadata と自動連携
- ✅ **Vertex AI Endpoints × 2** — encoder / reranker の外部化、search-api のメモリを 4Gi → 2Gi に縮小
- ✅ **Vertex AI Model Registry** — `staging` / `production` alias + カナリアデプロイ
- ✅ **Vertex AI Feature Store (Feature Group)** — Dataform テーブルを wrap、parity invariant の 6 番目
- ✅ **Vertex AI Experiments + ML Metadata** — W&B と dual write
- ✅ **Vertex AI Vizier** — `enable_tuning=True` 時のみ LambdaRank ハイパラ探索
- ✅ **Vertex AI Model Monitoring v2** — Endpoint drift 検知
- ⚠️ **BigQuery `VECTOR_SEARCH`** — Phase 3 から継続。Matching Engine (Vertex AI Vector Search) 化は別スコープ (遅延 p50 < 200ms 実績、IaC 手動 DDL 定着)
- ⚠️ **Meilisearch on Cloud Run** — Phase 3 から継続。Vertex に BM25 相当なし
- ❌ **BQML** — LambdaRank 非対応
- ❌ **Vertex AI Text Embeddings API** — ME5 を自前で Endpoint にロードする方針 (embedding 手数料を抑制)
- ❌ **Cloud Run Jobs `training-job` / `embedding-job`** — KFP に置換済、`infra/modules/runtime/main.tf` から削除
- ❌ **PostgreSQL / Cloud SQL** — Phase 3 から継続して不採用 (全て BigQuery)
- ❌ **Redis / Memorystore** — `cachetools.TTLCache` の in-memory キャッシュ (Phase 3 から継続)

### 1. 目的

- この構成は実運用最適化より、学習理解と MLOps 技術名の可視化を優先する
- レイヤ責務を名前で即判別できることを最重要要件とする

### 2. 命名ルール（必須）

- ディレクトリ名は小文字スネークケースのみ
- Python ファイル名は小文字スネークケースのみ
- ジョブのエントリーポイントは `pipeline/*_job/main.py` に統一
- API エントリーポイントは `app/main.py` に統一
- 評価・学習・推論・登録など MLOps 固有語を省略しない

### 3. ルート直下の必須ディレクトリ

- `app`
- `ml`
- `pipeline`
- `infra`
- `docs`
- `scripts`
- `tests`

上記は必須。別名（例: `applications`, `core`, `batch`, `qa`）への置換は禁止。

### 4. `ml` 配下の必須責務ディレクトリ

- `data`
- `training`
- `evaluation`
- `registry`
- `serving`
- `common`

上記は必須。`trainer` と `training` の混在、`eval` と `evaluation` の混在は禁止。

### 5. `pipeline` 配下の必須ジョブ名

- `data_job`
- `training_job`
- `evaluation_job`
- `batch_serving_job`
- `workflow`

ジョブごとのエントリーポイントは `main.py` に固定し、`run.py` `entry.py` などの別名は使わない。

### 6. `tests` 配下の必須区分

- `unit`
- `integration`
- `e2e`

テストはこの3区分に配置し、`api/` `ml/` などの責務別トップレベルを増やさない。

### 7. 禁止事項

- 旧構成（例: `app/src`, `common/src`, `jobs/src`, `pipelines/src`）の再導入
- 同義語ディレクトリの追加（例: `train`, `eval`, `serve`, `ops`）
- レイヤ責務をまたぐ配置（例: 学習コードを `app/` に置く）

### 8. 例外運用

- 一時互換レイヤを置く場合でも、最終形は本仕様に収束させる
- 例外を許可する場合は `docs/decisions/` に理由と期限を記録する

### 9. レビュー観点（チェックリスト）

- ルート必須ディレクトリが揃っているか
- `ml` と `pipeline` の必須サブディレクトリが揃っているか
- `pipeline/*_job/main.py` 形式になっているか
- 命名が小文字スネークケースに統一されているか
- 旧構成パスが新規追加されていないか

## デザインパターンの必須仕様

### 1. 目的

- 学習理解を優先し、責務分離と変更容易性を確保する
- テスト容易性と耐障害性を高めるため、外部連携の直接依存を排除する
- 特に GCP リソース連携と W&B 連携は Port/Adapter を必須とする

### 2. 適用アーキテクチャ（必須）

- 基本は Ports and Adapters（Hexagonal Architecture）を採用する
- ドメイン/ユースケース層は外部 SDK やクラウド API を import しない
- 依存方向は内側（ドメイン）に向け、外側（インフラ）は内側に依存する

### 3. レイヤ責務（必須）

- `app`:
  - API 入出力とユースケース起動のみを担当する
  - 外部サービスの詳細実装を持たない
- `ml/*`:
  - 学習・評価・推論ロジックを担当する
  - 外部連携は Port 経由で呼び出す
- `infra` / adapter 実装:
  - GCP、W&B、DB、ストレージ、キューなどの具体実装を担当する
  - Port インターフェースを実装して差し替え可能にする

### 4. 外部連携の Port 定義（必須）

最低限、以下は Port を定義する。

- Model Artifact Port（モデル保存/取得）
- Experiment Tracking Port（W&B などの記録）
- Feature Store / Warehouse Port（BigQuery など）
- Message / Event Port（Pub/Sub など）
- Scheduler / Job Trigger Port（Cloud Scheduler/Eventarc など）

Port は抽象（interface/protocol/abc）として定義し、ユースケース層は Port のみ参照する。

### 5. Adapter 実装ルール（必須）

- GCP 連携（BigQuery, GCS, Pub/Sub, Vertex AI, Cloud Run, Scheduler, Eventarc）は GCP Adapter に隔離する
- W&B 連携は Tracking Adapter に隔離する
- Adapter 内でのみ SDK 初期化・認証・リトライ・例外変換を扱う
- Adapter は Port 契約を満たすことを契約テストで検証する

### 6. テスト戦略（必須）

- Unit Test:
  - ユースケース層は Port のモック/スタブで検証する
  - 外部 SDK を直接モックしない（Port をモックする）
- Integration Test:
  - Adapter 単体で外部サービス接続を検証する
- E2E Test:
  - 主要シナリオの疎通のみを最小本数で維持する

### 7. 禁止事項

- `app` や `ml` から GCP SDK / W&B SDK を直接呼び出す実装
- ユースケース層で環境変数を直接参照して分岐する実装
- 外部入出力とビジネスロジックを1つの関数/クラスに混在させる実装
- テストでクラウド実サービス前提の結合に依存する実装（常時必須化）

### 8. 例外運用

- 例外的に直接連携が必要な場合でも、暫定実装として期限付きで扱う
- 例外は `docs/decisions/` に理由・影響範囲・解消期限を記録する

### 9. レビュー観点（チェックリスト）

- ユースケース層が Port のみに依存しているか
- GCP/W&B 連携が Adapter 層へ隔離されているか
- Port ごとにモック可能なインターフェースが定義されているか
- Unit Test が外部 SDK ではなく Port を差し替えているか
- 例外実装に期限付き ADR が付与されているか
