# 01_仕様と設計

> 決定的仕様は `docs/02_移行ロードマップ.md` を正とする。本ファイルは Phase 4 の実装配置を説明する。

Phase 4 の主目的は、Phase 3 の不動産ハイブリッド検索を維持したまま、**GCP serverless 基盤へ移行する** こと。

## 0. 最重要ルール

- 不変なのは題材と検索中核
- 検索中核は `Meilisearch + BigQuery VECTOR_SEARCH + multilingual-e5 + RRF + LightGBM LambdaRank`
- 変えるのは実行基盤、認証、運用の責務分担

## 1. Phase 4 で追加するもの

| 技術 | 配置 |
|---|---|
| Cloud Run Service / Jobs | `app/`, `pipeline/`, `infra/` |
| BigQuery / Dataform | `definitions/`, `common/`, `infra/` |
| GCS | モデル成果物保管 |
| Pub/Sub | ranking log / feedback |
| Terraform | `infra/terraform/` |
| WIF | `.github/workflows/`, `infra/` |
| Secret Manager | `infra/` と `app` 設定 |

## 2. 中核アーキテクチャ

1. Dataform で `property_features_daily` を生成する
2. Cloud Run Job で埋め込み更新と学習を回す
3. `search-api` は GCP 上の検索 API として動く
4. ranking log / feedback は Pub/Sub 経由で BigQuery へ流す
5. Secret Manager から Meilisearch key を注入する

## 3. 主要コンポーネント

### 3.1 `search-api`

- `app/` が Cloud Run 上の API
- lexical retrieval は Meilisearch
- semantic retrieval は BigQuery `VECTOR_SEARCH`
- 再学習判定や feedback 連携を持つ

### 3.2 `pipeline/` と job

- 埋め込み更新 job
- 学習 job
- Vertex ではなく Cloud Run Jobs を使う

### 3.3 データと特徴量

- BigQuery が検索、特徴量、ログの中心
- Dataform で特徴量テーブルを管理する
- feature parity を維持する

### 3.4 認証とシークレット

- CI/CD 認証は WIF
- API / job の実行権限は SA 分離
- Meilisearch master key は Secret Manager から注入する
