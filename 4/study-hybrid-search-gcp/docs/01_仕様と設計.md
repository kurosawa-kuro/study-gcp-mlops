## 0. 位置付け (Phase 4: GCP MLOps の土台 = 独立コード)

Phase 4 は **コード化対象 Phase** で、**Phase 7 完成版からの引き算ではなく、Phase 3 (Local 不動産ハイブリッド検索) からの足し算で構築** する。Phase 5 / 6 / 7 で扱う上位レイヤー (Vertex AI / Composer / Feature Store / GKE / KServe) の **前提となるクラウド土台** を理解することが目的。

### 0.1 境界の原則

> **Phase 4 は「local 不動産検索を GCP に移す phase」で止める。Vertex AI 以降は Phase 5 (論理 Phase) / Phase 7 (canonical) で扱う。**

中核は **GCP マネージドサービスの基礎習得** (Cloud Run / Cloud Run Jobs / BigQuery / GCS / Secret Manager / Terraform / WIF) と、**BigQuery feature table / view + BigQuery `VECTOR_SEARCH`** によるサーバーレス検索構成、**軽量 retrain trigger** (Cloud Scheduler + Eventarc + Cloud Function (Gen2))。

### 0.2 Phase 5 / 6 / 7 との関係

- Phase 5 (論理 Phase): Phase 4 の GCP 構成をベースに、Vertex AI MLOps 化 (Pipelines / Feature Store / Vector Search / Model Registry / Monitoring) を Phase 7 完成版コードを参照しながら資料で説明する
- Phase 6 (論理 Phase): Phase 5 の Vertex AI 構成をベースに、PMLE 追加技術 + Composer 本線化を Phase 7 完成版コードを参照しながら資料で説明する
- Phase 7 (canonical = 教材コード完成版): Phase 5 / 6 の技術要素をすべて本 phase 配下に本実装し、serving 層は GKE + KServe に置換した到達ゴール

## 1. Phase 3 → Phase 4 の差分 (足し算)

| 観点 | Phase 3 (Local) | Phase 4 (GCP MLOps 土台、追加) |
|---|---|---|
| 実行基盤 | Docker Compose | **Cloud Run Service / Cloud Run Jobs** |
| データ層 | PostgreSQL | **BigQuery** (検索 + 特徴量 + ログの中心) |
| ベクトル検索 | pgvector / 簡易 ANN | **BigQuery `VECTOR_SEARCH`** (Phase 7 完成版で Vertex Vector Search に発展) |
| 特徴量管理 | local files | **BigQuery feature table / view** (Dataform) |
| モデル成果物 | local filesystem | **GCS** |
| ログ送出 | PostgreSQL 直接 | **Pub/Sub → BigQuery** (BQ Subscription) |
| ranking 再学習 trigger | 手動 | **Cloud Scheduler + Eventarc + Cloud Function (Gen2)** (軽量 orchestration、Composer 本線化は Phase 7 完成版コードに集約 — Phase 4 では扱わない) |
| シークレット | local `.env` | **Secret Manager → Cloud Run secret injection (必須習得)** |
| IaC | なし | **Terraform** |
| CI/CD 認証 | なし | **WIF (Workload Identity Federation、SA Key 禁止)** |
| 監視 | 手動 | **Cloud Logging + Cloud Monitoring** |
| デプロイ | docker compose up | **GitHub Actions + Artifact Registry + Cloud Build** |

中核 5 要素 (Meilisearch BM25 / multilingual-e5 / ベクトルストア / RRF / LightGBM LambdaRank) は Phase 3 から継承して不変。**変えるのは実行基盤・認証・運用の責務分担のみ**。

## 2. Phase 4 の中核アーキテクチャ

```
[Cloud Scheduler]
   │ daily
   ▼
[Eventarc] ─→ [Cloud Function (Gen2): pipeline-trigger]
                │
                ▼
[Cloud Run Job: embed_pipeline]
   │
   ▼
[BigQuery: feature_mart.* / property_embeddings (BQ VECTOR_SEARCH source)]
   ▲
   │
[Cloud Run Service: search-api]
   │  /search
   ├──→ Meilisearch (Cloud Run、BM25 + filter)
   ├──→ BigQuery VECTOR_SEARCH (semantic ANN)
   ├──→ RRF 融合
   ├──→ LightGBM LambdaRank (in-process or Cloud Run inference)
   └──→ Pub/Sub (ranking_log / search-feedback)
                │
                ▼
        [BigQuery: mlops.ranking_log / mlops.feedback_events]
                │
                ▼
        [Cloud Run Job: training_job → GCS モデル更新]
```

### 2.1 主要コンポーネント

- **search-api** (`app/`): Cloud Run 上の API。lexical = Meilisearch、semantic = BigQuery `VECTOR_SEARCH`、再学習判定 + feedback 連携
- **embed pipeline / training job** (`pipeline/`): Cloud Run Jobs (Vertex AI Pipelines は使わない、Phase 5 / 7 完成版で扱う)
- **データ・特徴量**: BigQuery が検索・特徴量・ログの中心。Dataform で feature table / view 管理
- **軽量 retrain trigger**: Cloud Scheduler + Eventarc + Cloud Function (Gen2) で Cloud Run Job を起動 (Composer 本線化は Phase 7 完成版コードに集約)
- **認証・シークレット**: CI/CD = WIF、ランタイム SA = 分離、Meilisearch master key = Secret Manager → Cloud Run `--set-secrets` 注入

## 3. Secret Manager (必須習得要素)

Phase 4 で初導入する必須 4 段:

1. Secret 作成 (`gcloud secrets create meili-master-key`)
2. SA IAM bind (`roles/secretmanager.secretAccessor`)
3. Cloud Run `--set-secrets MEILI_MASTER_KEY=meili-master-key:latest`
4. app 側 pydantic-settings で `MEILI_MASTER_KEY` を読み取り

**Phase 4 から Phase 7 まで一貫して必須**。Phase 7 では External Secrets Operator が Secret Manager → K8s Secret を自動同期するが、Secret Manager 自体は同じ。

## 4. Phase 4 で扱わないもの (Phase 5 以降の論理 Phase / Phase 7 完成版の責務)

> **Phase 4 の境界原則**: Vertex AI 以降は入れない。Phase 5 / 6 (論理 Phase) で資料化、Phase 7 (canonical) で本実装。

| 技術領域 | 導入 phase | 位置づけ |
|---|---|---|
| Vertex AI Pipelines (KFP v2) / Endpoint / Model Registry / Monitoring | **Phase 5 (論理 Phase)** | Phase 7 完成版コードを参照する資料化 |
| **Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store)** | **Phase 5 (論理 Phase)** | Phase 7 完成版コードを参照する資料化 |
| **Vertex Vector Search** (Phase 4 の BQ `VECTOR_SEARCH` を発展させる serving index) | **Phase 5 (論理 Phase)** | Phase 7 完成版コードを参照する資料化 |
| Cloud Composer / Managed Airflow Gen 3 (本線 orchestration) / BQML / Dataflow Flex Template / TreeSHAP Explainability / Monitoring SLO + burn-rate / Composer-managed BQ monitoring query | **Phase 6 (論理 Phase)** | Phase 7 完成版コードを参照する資料化 |
| GKE Autopilot / KServe / Gateway API / External Secrets Operator / Workload Identity / GMP / HPA / IAP / NetworkPolicy / Helm provider | **Phase 7 (canonical = 教材コード完成版)** | 本実装 (Phase 5 / 6 の技術要素も Phase 7 配下に集約) |
| 全 Phase 教材対象外 (Agent Builder / Vizier / Model Garden / Gemini RAG / W&B / Looker Studio / Doppler) | — | 親 [`README.md` §1](../../../README.md) |

## 5. 不変ルール (Phase 4)

- 題材は不動産ハイブリッド検索 (Phase 3 から継承)
- 中核 5 要素は Phase 3-7 で不変 (Phase 4 = BQ `VECTOR_SEARCH`、Phase 7 完成版 = Vertex Vector Search に発展)
- **Phase 4 はベクトル検索を BigQuery `VECTOR_SEARCH` に閉じる** (Vertex Vector Search は Phase 5 論理 Phase で説明、Phase 7 で本実装)
- **Phase 4 は軽量 retrain trigger (Cloud Scheduler + Eventarc + Cloud Function) で完結する** (Cloud Composer は Phase 6 論理 Phase で説明、Phase 7 で本実装)
- Secret Manager は必須習得
- Terraform を正とする
- **Vertex AI / Cloud Composer / GKE / KServe / PMLE 追加技術は持ち込まない** (Phase 5 / 6 / 7 の責務)
- 設計思想 (Port/Adapter / `core → ports ← adapters` / DI) は **Phase 2 から継承して不変**、adapter 実装だけ差し替え
