# 01_仕様と設計

> 決定的仕様は `docs/02_移行ロードマップ.md` を正とする。本ファイルはそれに従って、Phase 6 の実装配置を説明する。

Phase 6 の主目的は、Phase 5 の不動産ハイブリッド検索コードを維持しながら、採用継続する PMLE 追加技術を **実コードへ bolt-on で統合して学ぶ** こと。削除済みの旧追加機能は採用対象外である。

## 0. 最重要ルール

- 不変なのは題材と中核検索経路だけ
- 中核検索経路は `Meilisearch + BigQuery VECTOR_SEARCH + multilingual-e5 + RRF + LightGBM LambdaRank`
- `/search` の既定挙動は Phase 5 と一致させる
- 追加技術は新 endpoint、補助スコア、監視、別 pipeline、別テーブルとして積み増す

## 1. 試験範囲との対応

| PMLE 観点 | Phase 5 までで既習 | Phase 6 で追加 |
|---|---|---|
| Low-code / classical ML | LightGBM 学習と配備 | T1 BQML |
| Scaling / data processing | Vertex Pipelines | T2 Dataflow |
| Serving / inference | Cloud Run + Vertex Endpoint | T4 Explainable AI, T6 RAG |
| Monitoring / operations | Vertex Monitoring | T5 Monitoring SLO |
| Foundation model 活用 | なし | T8 Model Garden enrichment |

## 2. 統合トピック詳細

### T1. BQML popularity scorer

**目的**: `ML.PREDICT` を実コードで扱い、検索結果へ補助スコアを追加する。

| 要素 | 実装 |
|---|---|
| Port | `app/services/protocols/popularity_scorer.py` |
| Adapter | `app/services/adapters/bqml_popularity_scorer.py` |
| SQL | `scripts/bqml/train_popularity.sql` |
| composition root | `app/main.py::_build_popularity_scorer` |
| 既定値 | `BQML_POPULARITY_ENABLED=false` |

### T2. Dataflow streaming

**目的**: ranking log から時間窓 CTR を集計する並列経路を追加する。

| 要素 | 実装 |
|---|---|
| Pipeline | `ml/streaming/pipeline.py` |
| Flex Template | `ml/streaming/container/` |
| Terraform | `infra/terraform/modules/streaming/` |
| 出力先 | `mlops.ranking_log_hourly_ctr` |
| 既定値 | `enable_streaming=false`, `enable_streaming_job=false` |

### T4. Explainable AI

**目的**: reranker の説明可能性を opt-in で確認できるようにする。

| 要素 | 実装 |
|---|---|
| reranker server | `ml/serving/reranker.py` |
| adapter 拡張 | `app/services/adapters/vertex_prediction.py` |
| endpoint | `app/main.py` の `/search?explain=true` |
| レスポンス | `SearchResultItem.attributions` |
| 既定値 | `explain=false` |

### T5. Monitoring SLO

**目的**: Cloud Run `search-api` の可用性と遅延を SLO として管理する。

| 要素 | 実装 |
|---|---|
| Terraform | `infra/terraform/modules/slo/` |
| 監視対象 | availability / latency |
| 運用コマンド | `make ops-slo-status` |

### T6. RAG (Gemini)

**目的**: 既存検索結果を Gemini に要約させる新 endpoint を追加する。

| 要素 | 実装 |
|---|---|
| Port | `app/services/protocols/generator.py` |
| Adapter | `app/services/adapters/gemini_generator.py` |
| Service | `app/services/rag_summarizer.py` |
| endpoint | `app/main.py` の `/rag` |
| 既定値 | `ENABLE_RAG=false` |

### T8. Model Garden enrichment

**目的**: 物件説明文を構造化し、別テーブルへ補助情報として保持する。

| 要素 | 実装 |
|---|---|
| スクリプト | `scripts/local/enrichment/run_enrichment.py` |
| 出力先 | `feature_mart.properties_enriched` |
| 用途 | tags / target_audience / summary_ja の補助生成 |

## 3. Port / Adapter 境界

- Port は `app/services/protocols/` に置く
- 外部 SDK 依存は `app/services/adapters/` または `ml/streaming/` に閉じ込める
- service 層は adapter を直接 import しない
- Phase 6 追加コードも `make check` の対象に含める

## 4. 既定経路

Phase 6 の `search-api` は既定で次の構成を使う。

1. lexical retrieval: `Meilisearch`
2. semantic retrieval: `BigQuerySemanticSearch`
3. fusion: `RRF`
4. rerank: `LightGBM LambdaRank`

この経路は User 合意なしに差し替えない。
