## 5. 設計決定事項（提案）

### 5.1 K8s 実行基盤

- **GKE Autopilot を第一候補**。ノード管理を持たず、学習関心事（KServe / HPA / Workload Identity）に集中する。
- Standard クラスタは cost / GPU / ノードプール分離の要件が顕在化したときに再判断。

### 5.2 SA 戦略

- `sa-endpoint-encoder` / `sa-endpoint-reranker` / `sa-api` は **SA 名を維持**し、Workload Identity で GKE の KSA（Kubernetes Service Account）にバインド。
- Phase 5 の 9 SA 構成は GKE 側でもそのまま効かせる（新規 SA は原則追加しない）。

### 5.3 認可境界（IAM-gated 相当）

- Cloud Run の `--no-allow-unauthenticated` + IAM 呼び出し相当を、GKE 側では **Gateway API + AuthorizationPolicy（or NetworkPolicy）** で再現。
- KServe `InferenceService` は **クラスタ外部に公開しない**。search-api Pod のみが呼べるように絞る。

### 5.4 モデル配布

- 学習パイプラインは Phase 5 と同じく Vertex Model Registry に登録。
- GKE 側は **Model Registry から GCS artifact URI を引き、KServe の `storageUri` にそのまま渡す** 経路を第一候補。
- 代替案: Registry → GCS ミラー → KServe（Registry 依存を切りたい場合）。

### 5.5 モニタリング

- Vertex Model Monitoring v2 は Vertex Endpoint 前提のため、encoder / reranker は **KServe 側でリクエスト数 / レイテンシ / エラー率を Managed Prometheus + PodMonitoring で取得**。
- Training-Serving Skew 監視 (`monitoring/validate_feature_skew.sql`) は **Phase 5 と同一**（BigQuery 側検知なので GKE 化の影響を受けない）。
- **予測分布 drift 検知は Phase 6 では縮退**。Vertex Model Monitoring v2 の drift 検知は KServe 化で失い、Managed Prometheus の量 / レイテンシ / エラー率では代替にならない。Phase 6 スコープでは再構築せず、後続フェーズで復活方針を判断（§8 参照）。

## 11. 工数目安

前提スキル: **K8s / EKS 実務 1 年 + GKE 1 ヶ月 + KServe 1 ヶ月**。この背景では Gateway API / Workload Identity / Istio / HPA / NetworkPolicy / Managed Prometheus / KServe CRD (`InferenceService` / `ServingRuntime`) 全般を学習コストから除外する。

| 段階 | 作業内容 | 想定日数 |
|---|---|---|
| A | `modules/gke` + `modules/kserve` Terraform スケルトン | 1 日 |
| B | multilingual-e5 の KServe ランタイム選定・PoC（Huggingface runtime vs 自前 Python predictor）+ latency 実測 | 2〜3 日 |
| C | LightGBM lambdarank の KServe `InferenceService`（MLServer LightGBM runtime）と Phase 5 reranker 契約の整合確認 | 0.5〜1 日 |
| D | `KServeEncoder` / `KServeReranker` adapter 実装 + Vertex Endpoint 版との parity test | 1 日 |
| E | search-api の GKE Deployment + Gateway + IAP + HPA | 1 日 |
| F | Model Registry → `InferenceService.storageUri` の同期スクリプト (`scripts/local/deploy/kserve_models.py`) | 0.5 日 |
| G | E2E 疎通（クラスタ apply → KServe デプロイ → `/search` 実疎通） | 1 日 |
| H | docs 整備（`01_仕様と設計.md` / `03_実装カタログ.md` / `04_運用.md` / `CLAUDE.md`） | 1 日 |

**合計: 7〜9 人日 / 1〜1.5 週間**

### 単一リスク要因

段階 `B` の ME5 ランタイム選定のみ。predictor 起動時間 / 推論 latency が許容外（目安: P95 500ms 超）の場合に取りうる選択肢:

- GPU ノードプール化 — Autopilot では制約あり、Standard クラスタに切替判断
- Triton Inference Server + ONNX 変換 — 本フェーズのスコープ外（「GKE + KServe 学習」の軸から外れる）
- 撤退（Vertex Endpoint 維持、search-api だけ GKE 化に縮退）

`B` の PoC 結果で go/no-go を先に判定する設計にしておけば、他段階に波及しない。
