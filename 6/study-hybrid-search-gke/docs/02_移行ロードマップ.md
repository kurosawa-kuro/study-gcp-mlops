## 1. serving 疎通 — KServe encoder / reranker の protocol mismatch (CRITICAL)

Phase 6 の KServe 切り替えで **4 箇所の契約不一致** が積み重なっている。搭載してもまず `/search` が 4xx/5xx になる。

### 1.1 Encoder payload schema の不一致

- **client** (`app/services/adapters/kserve_prediction.py:60`): `{"instances": [{"text": f"{kind}: {text.strip()}"}]}` — `kind` を前置詞として text に埋め込み、`kind` フィールドは送らない
- **server** (`ml/serving/encoder.py:73-79`): `EncoderInstance` が `text: str` + `kind: Field(pattern="^(query|passage)$")` を **両方必須** として Pydantic validation
- ⇒ 422 Unprocessable Entity。server は `kind == "query"` で分岐して `encode_queries` / `encode_passages` を切替えるロジックなので、text 先頭の `query:` 接頭辞は無視される (= 無害ではなく、contract 違反で落ちる)
- 付随: `tests/unit/app/test_adapters.py:70-74` が **client の壊れた shape をそのまま期待している**。テストをフィクスすると client の修正も強制される

**過去実害**: Phase 5 Run 6 (2026-04-23) で *まったく同じバグ* に既に当たっている — `VertexEndpointEncoder.embed` が `{"text": "query: ..."}` 単一フィールドで送って `/search` が HTTP 500 (encoder 側 422 `Field required: kind`)。adapter を `{"text": text.strip(), "kind": kind}` に直して決着。**Phase 6 はその修正を undo してしまっている**。

**修正方針**:
1. `KServeEncoder.embed()` を `{"instances": [{"text": text.strip(), "kind": kind}]}` に戻す (Phase 5 `VertexEndpointEncoder` と同じ payload)
2. `tests/unit/app/test_adapters.py` の `sent_json` assert を対応する shape に修正
3. ME5 の `query: ` / `passage: ` 接頭辞は **server 側** (`E5Encoder.encode_queries` / `encode_passages`) が適用する契約を維持 (Phase 5 の `app/services/adapters/vertex_prediction.py` のコメントが正)

### 1.2 Encoder URL path の不一致

- **client** (`infra/manifests/search-api/deployment.yaml`): `KSERVE_ENCODER_URL = http://property-encoder.kserve-inference.svc.cluster.local/v1/models/property-encoder:predict`
- **server** (`ml/serving/encoder.py:121,126`): `@app.get(os.getenv("AIP_HEALTH_ROUTE", "/health"))` / `@app.post(os.getenv("AIP_PREDICT_ROUTE", "/predict"))`
- KServe custom-container (RawDeployment) は **pass-through** なので、Service DNS はそのまま container port 8080 に届くが、container は `/predict` しか listen しない
- ⇒ 404 Not Found

**修正方針**: どちらかに寄せる。学習容易性優先なら選択肢 A を推奨。

- **選択肢 A (推奨)**: deployment.yaml で `KSERVE_ENCODER_URL` を `.../predict` に変更。server は Vertex CPR 規約 (AIP_*) のまま
- **選択肢 B**: KServe 規約に寄せるため、encoder 側 Deployment (= `infra/manifests/kserve/encoder.yaml` の `env`) に `AIP_PREDICT_ROUTE=/v1/models/property-encoder:predict` と `AIP_HEALTH_ROUTE=/v1/models/property-encoder` を追加

reranker (MLServer LightGBM runtime) はフレームワーク側で `/v1/models/<name>:predict` を実装するので、reranker 側は URL 変更しない。つまり encoder 側のみ揃える。

### 1.3 Encoder STORAGE_URI の env var 名不一致

- **server** (`ml/serving/encoder.py:104`): `os.getenv("AIP_STORAGE_URI", "").strip()` を要求。empty の場合 `RuntimeError("AIP_STORAGE_URI is required")` で startup 失敗
- **manifest** (`infra/manifests/kserve/encoder.yaml:34`): `env: - name: STORAGE_URI`
- **patcher** (`scripts/local/deploy/kserve_models.py:107`): `{"name": "STORAGE_URI", "value": storage_uri}`
- ⇒ Pod CrashLoopBackOff

**修正方針**: encoder.yaml と `kserve_models.py::_patch_encoder_storage_uri` の env var 名を `STORAGE_URI` → `AIP_STORAGE_URI` に統一。server 側の env var 名は変更しない (Phase 5 encoder server を継承しているため)。

### 1.4 Encoder `_download_artifact_dir` の self-contradict バグ復活

Phase 5 Run 6 で修正済みのバグが Phase 6 で **そのまま復活している**。

- `infra/manifests/kserve/encoder.yaml:36`: `STORAGE_URI: "gs://mlops-dev-a-models/encoder/latest"` (trailing slash なし)
- `ml/serving/encoder.py:90-91` (Phase 6): `if prefix.prefix and not prefix.prefix.endswith("/"): raise RuntimeError("AIP_STORAGE_URI must point to a directory prefix")`
- ⇒ placeholder でも deploy_kserve_models 前でも Pod startup が failed。また `kserve_models.py` が Vertex Model Registry から取った `artifact_uri` も trailing slash 保証なし

**過去実害**: Phase 5 Run 6 で encoder container が起動直後 `RuntimeError: AIP_STORAGE_URI must point to a directory prefix` で exit、reranker endpoint PASS まで行っていたのに encoder だけ詰まった。真因は `GcsPrefix.parse()` が trailing slash を strip する仕様に反して「/ で終わる必要がある」チェックを掛けていた self-contradict (pre-existing バグ)。Phase 5 の修正コードは:

```python
prefix_with_slash = f"{prefix.prefix}/" if prefix.prefix else ""
for blob in client.list_blobs(prefix.bucket, prefix=prefix_with_slash):
    ...
    rel = blob.name[len(prefix_with_slash):] if prefix_with_slash else blob.name
```

これを「strip された prefix に `/` を補ってから list_blobs、blob name から prefix_with_slash を剥がす」で解決した。Phase 6 は `if not endswith("/"): raise` + `list_blobs(prefix=prefix.prefix)` の壊れた形に逆戻り。

**修正方針**:
1. `ml/serving/encoder.py` の `_download_artifact_dir` を **Phase 5 Run 6 後の実装**に戻す (trailing slash を補う、self-contradict チェックを削除)。当該部分の Phase 5 vs Phase 6 diff は `git log -p ml/serving/encoder.py` 相当で 1:1 復元可能
2. 予防として encoder.yaml の placeholder を `gs://mlops-dev-a-models/encoder/latest/` (trailing slash つき) に統一
3. `scripts/local/deploy/kserve_models.py::_patch_encoder_storage_uri` で `storage_uri.rstrip("/") + "/"` と正規化してから patch

## 3. Docker build context の巻き込み事故再発 (CRITICAL)

`.dockerignore` が Phase 5 の **minimal-whitelist 方針** から Phase 6 で **broad-exclude 方針** に退行している。User memory (`feedback_dockerignore_policy.md`) の事故多発ポイントに該当。

| Phase 5 (working) | Phase 6 (current, 危険) |
|---|---|
| Python cache / TF state / `.git` 系のみ除外 | `docs/`, `infra/`, `**/data/`, `**/tests/`, `**/*.md` 等を包括除外 |
| 過去事故のコメント (16行) で意図明示 | コメント削除 |

### 過去実害 (`/home/ubuntu/repos/study-gcp-mlops/docs/動作検証結果.md` より)

- **Phase 5 Run 3 (2026-04-23)**: `**/data/` パターンが `ml/data/` を巻き込み除外 → `infra/run/services/search_api/Dockerfile` 内の `COPY ml/ ml/` に `ml/data/` が来ず、Cloud Run revision が `ModuleNotFoundError: No module named 'ml.data'` で起動失敗。Run 2 で Dockerfile に `COPY` を追加しても直らなかった真因がこれ。対処は `.dockerignore` から `**/data/` を削除
- **Phase 5 Run 7 (2026-04-23)**: meili-sync 専用 image を作ろうとしたが、`.dockerignore` の `infra/` が `infra/run/jobs/meili_sync/sync_meili_job.py` ごと build context から落とし、`COPY` 失敗が連鎖。6 経路 (IAM grant / impersonation / cloud-run-proxy / shell escape 等) 試した末の正解は「専用 image + Cloud Run Job」だったが、`.dockerignore` の `infra/` 一行で半日溶かした
- 結論として **`.dockerignore` は事故要因ランキング最上位**と認定、「Python cache / tfstate / .git のみ」の最小構成で確定、memory に永続化

### Phase 6 の危険度 (確度高)

Phase 6 で追加されている `infra/` 除外は Phase 5 Run 7 の再発シナリオに完全合致。加えて `**/data/` は Run 3 の再発、`**/tests/` は `ml/embed/container/` / `ml/train/container/` 等の container build で tests fixture が COPY 対象になっている場合に連鎖する。

**修正方針**:
1. Phase 5 の `.dockerignore` をそのまま復元 (改変せずコピー)。コメントの `Phase 5 debug:` を `Phase 5/6 共通:` に書き換えるだけに留める
2. Phase 6 で top-level wildcard を足したい場合 (Cloud Build tarball が 1.1 GiB になる問題 = Phase 5 Run 9 教訓) は、**Dockerfile per-service** で `.dockerignore.service` を別ファイルとして持ち、`gcloud builds submit --ignore-file=` で切替える方針。ルート `.dockerignore` は絶対に触らない
3. 追加前には必ず全 Dockerfile の `COPY` 行を grep し、Python package 名 (`ml/data` / `ml/embed` / `ml/train` / `ml/serve` / `ml/sync` / `ml/trigger` / `ml/pipeline_utils`) と衝突しないかチェック

## 4. pipeline / image naming 回帰 (HIGH)

### 4.1 `train_reranker` の Output artifact 復活 + `register_reranker` の本実装復活 (2 連鎖)

Phase 6 は Phase 5 Run 3 と Run 4 で対処した 2 つの KFP executor 事故の修正を **両方 undo している**。そのまま走らせれば pipeline は Phase 5 Run 3/4 の失敗シーケンスを再現する。

**過去実害 (Phase 5 Run 3, 2026-04-23)**:
- `train_reranker` KFP component が `Path(model.path).write_text(...)` で単一ファイル扱いしていたため、下流の `Model.upload(artifact_uri=model.uri)` がディレクトリとして参照した時点で空に見え `NotFound 404 There are no files in directory "gs://.../model"` で落ちた
- 一次対処: `train_reranker.py` を修正、`model.path` をディレクトリとして確保し、その中に `model.json` を書き出す
- その後: `Input[dsl.Model]` → `Input[dsl.Artifact]` → `str` パラメータ (URI 直渡し) と段階退行させ、最終的に `def train(...) -> str:` で URI を string で返す workaround に落ち着いた (Phase 5 コメントに "KFP executor issue where Input artifacts caused worker to exit before emitting logs" として記録)

**過去実害 (Phase 5 Run 4, 2026-04-23)**:
- `register-reranker` が `@dsl.component(packages_to_install=["google-cloud-aiplatform>=1.71,<2"])` 付きで **pipeline の最後の step** として走ると `workerpool0-0` が 0 ログで exit 1 する事故が 5 連続発生 (164728 / 165819 / 170643 / 171318 / 171912)
- 同じ `base_image=python:3.12` + 同じ `packages_to_install` で走る `resolve_hyperparameters` (pipeline 冒頭) は SUCCEED するため、pip install 自体は正常。pipeline 末尾の pip install がタイミング的に pile up して quota/rate-limit/VM プロビジョニング遅延、という仮説
- 決着: `register-reranker` を MINIMAL stub (print + return のみ、`packages_to_install` 無し) に縮退させて pipeline は SUCCEED させ、**Model.upload はローカル `scripts/local/ops/register_model.py` から叩く** 構成に切り出して Phase 5 vertex-run-all 8/8 PASS を達成

**Phase 6 現状の問題**:
- `pipeline/training_job/components/train_reranker.py`: `def train(...) -> None:` + `model: dsl.Output[dsl.Model]` に戻った (Phase 5 Run 3 前の形)
- `pipeline/training_job/components/register_reranker.py`: `packages_to_install=["google-cloud-aiplatform>=1.71,<2"]` + `aiplatform.Model.upload(...)` の本実装復活 (Phase 5 Run 4 が捨てた形)
- `pipeline/training_job/main.py:93`: `model_artifact_uri=train_task.outputs["Output"]` → `model_artifact=train_task.outputs["model"]` に戻った

⇒ Pipeline submit すれば Run 3 で `model.path` 単一ファイル書き込み → 404、または Run 4 で register-reranker worker 0 ログ exit 1 のどちらか (or 両方) が再発する確率が高い。

**修正方針**:
- `train_reranker.py` → Phase 5 形式 (`-> str` で URI を返す / 内部で `model.path` をディレクトリとして確保し `model.json` 書き出し) に戻す
- `main.py` の pipeline definition → Phase 5 形式 (`model_artifact_uri=train_task.outputs["Output"]`) に戻す
- `register_reranker.py` → **MINIMAL stub** に縮退 (`packages_to_install` 外す / print + return のみ)。実 Model.upload は `scripts/local/ops/register_model.py` を継承 or 新設して pipeline の外で実行する。**Phase 6 では `scripts/local/deploy/kserve_models.py` が既に Model Registry を読む側なので、register 側 (書き込み) を pipeline 内から外に出すと、デプロイパイプラインとしても綺麗に切れる**
- `evaluate.py` / `vizier.py` / `load_features.py` の STEP 1/2 debug logging は Phase 5 に揃えて復活 (KFP component 障害時のトリアージに load-bearing)
