## 3. 不変ルール (論理 Phase)

- **Phase 6 は独立コードを保守しない**。Phase 7 完成版コードを基準にして PMLE 追加技術 + Composer 本線化の論理境界を理解する
- **題材は Phase 5 論理 Phase と同じ不動産ハイブリッド検索ドメインを維持** (PMLE 試験のための独立ドメインは作らない)
- **ハイブリッド検索中核コード (Meilisearch BM25 + Vertex Vector Search + multilingual-e5 + RRF + LightGBM LambdaRank) の挙動は絶対に変えない**
- **default feature flag では Phase 5 挙動を維持** (PMLE 4 技術は opt-in で有効化、Phase 7 配下で確認)
- **Composer 本線 + Vertex `PipelineJobSchedule` 撤去 + 軽量経路の格下げ** はカニバリ禁止 (上下関係厳守、Phase 7 docs/01 §3 が canonical)

### 3.1 Feature parity invariant (6 ファイル同 PR 原則、Phase 7 完成版コード上で適用)

特徴量を追加 / 変更するとき、以下 6 つを **必ず同じ PR で揃える** (Phase 7 で本実装):

1. `7/.../pipeline/data_job/dataform/features/property_features_daily.sqlx`
2. `7/.../ml/data/feature_engineering/ranker_features.py::build_ranker_features`
3. `7/.../ml/data/feature_engineering/schema.py::FEATURE_COLS_RANKER`
4. `7/.../infra/terraform/modules/data/main.tf` の `ranking_log.features` RECORD
5. `7/.../infra/sql/monitoring/validate_feature_skew.sql` の UNPIVOT
6. `7/.../infra/terraform/modules/vertex/main.tf::google_vertex_ai_feature_group_feature` × N
