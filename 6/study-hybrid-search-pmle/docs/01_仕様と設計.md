## 0. 位置付け (Phase 6: PMLE 追加技術 + Composer 本線化の論理境界 = 論理 Phase)

Phase 6 は **論理 Phase** (個別コード保守なし)。題材は Phase 5 から継承して **同じ不動産ハイブリッド検索ドメインを維持** し、**PMLE 試験範囲の追加技術 + Cloud Composer 本線化** を Phase 7 完成版コードを参照しながら資料で理解する。

### 0.1 論理 Phase であることの意味

```text
Phase 6 は独立コードを保守しない。
Phase 7 完成版コードを基準に、PMLE 追加技術 + Composer 本線化を理解するための論理 Phase とする。
```

**核心の 2 軸**:
1. **PMLE 4 技術 (T1 BQML / T2 Dataflow / T4 TreeSHAP Explainability / T5 Monitoring SLO + burn-rate)** が Phase 7 完成版コードに統合されている経路を理解する
2. **Cloud Composer / Managed Airflow Gen 3 を本線オーケストレーターとして使う** 上下関係 (Composer = 上位 orchestrator / Vertex Pipelines = 下位 ML executor) を理解する。Vertex `PipelineJobSchedule` 完全撤去、Cloud Scheduler / Eventarc / Cloud Function は smoke / 軽量代替に格下げ

### 0.2 参照する Phase 7 完成版コード

Phase 6 で説明する技術要素はすべて Phase 7 配下に本実装されている:

## 1. Phase 5 → Phase 6 の差分 (技術境界の理解)

> **論理 Phase の説明方針**: 以下の差分は **Phase 7 完成版コードに本実装されている** ものを Phase 6 の技術境界として整理した表。Phase 6 配下に独立コードはない。

### 1.1 Cloud Composer 本線化 (論理境界)

| 観点 | Phase 5 (論理 Phase = Phase 4 軽量経路を継続して理解) | Phase 6 で増える理解 (Phase 7 配下に本実装) |
|---|---|---|
| 本線 orchestration | Cloud Scheduler + Eventarc + Cloud Function | **Cloud Composer / Managed Airflow Gen 3** (3 本 DAG: `daily_feature_refresh` / `retrain_orchestration` / `monitoring_validation`) |
| 上下関係 | (なし、軽量経路のみ) | **Composer = 上位 orchestrator / Vertex Pipelines = 下位 ML executor** (同責務を両層に持たせない、カニバリ禁止) |
| Vertex `PipelineJobSchedule` | (Phase 4-5 で使用) | **完全撤去** (二重起動防止) |
| Cloud Scheduler / Eventarc / Cloud Function | 本線 trigger | **smoke / 軽量代替 / manual trigger 用途に格下げ** (リソース自体は残置) |
| `7/.../infra/terraform/modules/composer/` | なし | 本実装 (Composer Gen 3 SMALL、`mlops-dev-a` PDCA 都合で `make destroy-all` で破棄) |
| `7/.../pipeline/dags/` | なし | 本実装 (3 本 DAG: `daily_feature_refresh.py` / `retrain_orchestration.py` / `monitoring_validation.py`) |

詳細は [Phase 7 docs/01 §3 Cloud Composer の位置づけ](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md) (canonical)。

### 1.2 PMLE 4 技術の統合 (Phase 7 完成版コードに本実装)

| # | 技術 | Phase 7 配下の配置 | 役割 (Phase 5 中核を変えない opt-in) |
|---|---|---|---|
| **T1** | **BQML** | `7/.../app/services/adapters/bqml_popularity_scorer.py` + `scripts/bqml/train_popularity.sql` + `infra/terraform/modules/data/` | property-popularity model を BQML で学習し、reranker の副経路 (popularity boost) として bolt-on |
| **T2** | **Dataflow Flex Template** | `7/.../ml/streaming/` + `infra/terraform/modules/streaming/` | ranking_log → 特徴量 streaming ingest (low-latency ctr / fav_rate refresh)。Phase 5 論理 Phase で扱う Feature Store 入力源を強化 |
| **T4** | **TreeSHAP Explainability** | `7/.../ml/serving/reranker.py::/explain` ルート | `/search?explain=true` で TreeSHAP attributions 取得 (Vertex CPR 互換 server) |
| **T5** | **Monitoring SLO + burn-rate alert** | `7/.../infra/terraform/modules/slo/` | search-api availability / latency SLO + burn-rate alert |
| 補助 | **Composer-managed BQ monitoring query** | Composer DAG `monitoring_validation` 内の BigQueryInsertJobOperator | feature skew / model output drift を Composer DAG が定期実行 |

### 1.3 中核 5 要素 (不変)

中核 5 要素 (Meilisearch BM25 + **Vertex Vector Search** + multilingual-e5 + RRF + LightGBM LambdaRank) と `/search` デフォルト挙動は Phase 5 論理 Phase から継承して **絶対に変えない**。PMLE 4 技術は Phase 7 完成版コードで **default off の opt-in** として導入されている (新 feature flag は Phase 5 挙動 default を維持)。

## 2. Phase 6 で理解する中核アーキテクチャ (Phase 7 完成版コードを参照)

> 以下のアーキテクチャは Phase 7 配下に本実装されている (Phase 6 配下に独立コードなし)。Phase 5 論理 Phase で理解した Vertex AI 本番 MLOps 基盤に Composer 本線 + PMLE 4 技術が足された姿として理解する。


```
[Cloud Composer (Managed Airflow Gen 3)] ← 本線 orchestrator
   │
   ├─[DAG: daily_feature_refresh] (01:00 JST)
   │    ├─ Dataform run (feature_mart.* refresh)
   │    ├─ Feature View sync (Vertex AI Feature Online Store 更新)
   │    └─ (Phase 6B 増設) Composer-managed BQ monitoring query
   │
   ├─[DAG: retrain_orchestration] (04:00 JST、本線 retrain schedule)
   │    ├─ check_retrain (BQ feature freshness / model drift 判定)
   │    ├─ Vertex AI Pipelines submit (data_job → training_job → evaluation_job)
   │    ├─ wait SUCCEEDED
   │    ├─ (Phase 6B 増設) BQML training (T1 副経路)
   │    └─ Model Registry promote (NDCG 閾値 PASS 後、production alias 切替)
   │
   └─[DAG: monitoring_validation] (04:30 JST)
        ├─ run_feature_skew (BigQueryInsertJobOperator)
        ├─ run_model_output_drift (BigQueryInsertJobOperator)
        ├─ check_slo_burn_rate (T5)
        └─ (Phase 6B 増設) TreeSHAP Explainability eval

  ↓ submit
[Vertex AI Pipelines (KFP v2)] ← 下位 ML executor
   │
   ├─[Vertex Vector Search] (ME5 ベクトル serving index)
   ├─[Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store)]
   ├─[Vertex Model Registry] (production / staging alias)
   └─[Vertex Model Monitoring v2]
                                  ▲
                                  │
                          [Cloud Run: search-api]
                          ├─ Meilisearch (BM25)
                          ├─ multilingual-e5 (Vertex Endpoint)
                          ├─ Vertex Vector Search (ANN)
                          ├─ RRF 融合
                          ├─ Feature Online Store (Feature View 経由 fresh feature)
                          ├─ LightGBM LambdaRank (Vertex Endpoint)
                          ├─ + (T1 opt-in) BQML popularity bolt-on
                          ├─ + (T4 opt-in) /search?explain=true → TreeSHAP
                          └─ Pub/Sub (ranking_log / feedback)
                                  │
                                  ▼
                          + (T2 opt-in) Dataflow streaming
                          + (T5 opt-in) Monitoring SLO + burn-rate
```

### 2.1 主要コンポーネント (Phase 7 完成版コード上で理解)

- **Composer 環境**: Composer Gen 3 SMALL、`asia-northeast1`、`7/.../infra/terraform/modules/composer/`。`mlops-dev-a` PDCA 都合で `make destroy-all` で破棄、`make deploy-all` で再作成
- **3 本 DAG**: `7/.../pipeline/dags/{daily_feature_refresh,retrain_orchestration,monitoring_validation}.py`
- **Composer SA** (`sa-composer`): `roles/composer.worker` / `roles/aiplatform.user` / `roles/bigquery.{jobUser,dataViewer}` / `roles/run.invoker`
- **`make composer-deploy-dags`**: DAG file を Composer DAG GCS bucket に upload
- **PMLE 4 技術**: 上記 §1.2 の 4 ファイル群 (Phase 7 配下に本実装)

## 3. Cloud Composer canonical (上下関係 + カニバリ禁止)

詳細は [Phase 7 docs/01 §3](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md)。要点:

- **Composer (上位 orchestrator)**: Dataform run + BQ feature freshness check + Dataflow Flex Template launch + BQML train + Vertex AI Pipeline submit + monitoring query
- **Vertex Pipelines (下位 ML executor)**: train_pipeline / eval_pipeline / register / promote (Composer から submit される側)
- **NG (カニバリ禁止)**:
  - Composer DAG が `train/evaluate/register` を直接実装する (Vertex Pipelines にもある → 重複)
  - Composer DAG schedule + Vertex `PipelineJobSchedule` が同じ retrain を二重起動する
  - 軽量経路 (Cloud Scheduler / Eventarc / Cloud Function) が本線 retrain trigger に再昇格する

### 3.1 Phase 6 内部マイルストーン (6A / 6B、Phase 7 完成版コード上で確認)

- **6A**: Cloud Composer DAG 本線化 (`daily_feature_refresh` / `retrain_orchestration` / `monitoring_validation` の本実装、Vertex `PipelineJobSchedule` 撤去)
- **6B**: PMLE 4 技術を DAG に増設 (Composer-managed BQ monitoring query + BQML training + TreeSHAP Explainability eval)

## 5. 不変ルール (Phase 6 = 論理 Phase)

- **Phase 6 は独立コードを保守しない**。Phase 7 完成版コードを基準にして PMLE 追加技術 + Composer 本線化の論理境界を理解する
- **題材は Phase 5 と同じ不動産ハイブリッド検索ドメインを維持** (PMLE 試験のための独立ドメインは作らない)
- **ハイブリッド検索中核コード (Meilisearch BM25 + Vertex Vector Search + multilingual-e5 + RRF + LightGBM LambdaRank) の挙動は絶対に変えない** (Phase 5 論理 Phase から継承)
- **default feature flag では Phase 5 挙動を維持** (PMLE 4 技術は opt-in で有効化、Phase 7 配下で確認)
- **Composer 本線 + Vertex `PipelineJobSchedule` 撤去 + 軽量経路の格下げ** はカニバリ禁止 (上下関係厳守、Phase 7 docs/01 §3 が canonical)
- 中核を変える提案 (Meilisearch 置換 / LambdaRank 置換 / RRF 廃止 等) は事前合意必須

### 5.1 Feature parity invariant (6 ファイル同 PR 原則、Phase 7 完成版コード上で適用)

特徴量を追加 / 変更するとき、以下 6 つを **必ず同じ PR で揃える** (Phase 7 で本実装):

1. `7/.../pipeline/data_job/dataform/features/property_features_daily.sqlx`
2. `7/.../ml/data/feature_engineering/ranker_features.py::build_ranker_features`
3. `7/.../ml/data/feature_engineering/schema.py::FEATURE_COLS_RANKER`
4. `7/.../infra/terraform/modules/data/main.tf` の `ranking_log.features` RECORD
5. `7/.../infra/sql/monitoring/validate_feature_skew.sql` の UNPIVOT
6. `7/.../infra/terraform/modules/vertex/main.tf::google_vertex_ai_feature_group_feature` × N
