## 0. 位置付け (引き算戦略上の Phase 6)

Phase 6 は親 [`README.md` §4](../../../README.md) の引き算チェーンで **Phase 7 から GKE Autopilot / KServe / Gateway API / External Secrets Operator / Workload Identity / GMP / HPA / IAP / NetworkPolicy / Helm provider / KServe → Feature Online Store opt-in 参照経路 / TreeSHAP 用 explain 専用 Pod を引き算した結果**。

題材は Phase 5 から継承して **同じ不動産ハイブリッド検索ドメインを維持** し、PMLE 試験範囲の追加技術を **Phase 5 実コードに adapter / 副経路 / 追加エンドポイント / 追加 Terraform として統合** する phase。実装は Phase 5 の **Cloud Run + Vertex AI** スタックで動かし続ける (GKE/KServe 化は Phase 7)。

**核心の 2 軸**:
1. **PMLE 4 技術 (T1 BQML / T2 Dataflow / T4 TreeSHAP Explainability / T5 Monitoring SLO + burn-rate)** を Phase 5 実コードへ統合
2. **Cloud Composer / Managed Airflow Gen 3 を本線オーケストレーターとして導入** (= **Phase 5 → 6 引き算境界**、Vertex `PipelineJobSchedule` 完全撤去、Cloud Scheduler / Eventarc / Cloud Function は smoke / 軽量代替に格下げ)

## 1. Phase 5 → Phase 6 の差分

### 1.1 Cloud Composer 本線昇格 (Phase 5 → 6 引き算境界)

| 観点 | Phase 5 (継承元) | Phase 6 (Composer 本線昇格) |
|---|---|---|
| 本線 orchestration | Cloud Scheduler + Eventarc + Cloud Function | **Cloud Composer / Managed Airflow Gen 3** (3 本 DAG: `daily_feature_refresh` / `retrain_orchestration` / `monitoring_validation`) |
| 上下関係 | (なし、軽量経路のみ) | **Composer = 上位 orchestrator / Vertex Pipelines = 下位 ML executor** (同責務を両層に持たせない、カニバリ禁止) |
| Vertex `PipelineJobSchedule` | (Phase 4-5 で使用) | **完全撤去** (二重起動防止) |
| Cloud Scheduler / Eventarc / Cloud Function | 本線 trigger | **smoke / 軽量代替 / manual trigger 用途に格下げ** (リソース自体は残置) |
| `infra/terraform/modules/composer/` | なし | 新設 (Composer Gen 3 SMALL、`mlops-dev-a` PDCA 都合で `make destroy-all` で破棄) |
| `pipeline/dags/` | なし | 新設 (3 本 DAG: `daily_feature_refresh.py` / `retrain_orchestration.py` / `monitoring_validation.py`) |

詳細は [Phase 7 docs/01 §3 Cloud Composer の位置づけ](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md) (canonical)。

### 1.2 PMLE 4 技術の統合

| # | 技術 | 配置 | 役割 (Phase 5 中核を変えない opt-in) |
|---|---|---|---|
| **T1** | **BQML** | `app/services/adapters/bqml_popularity_scorer.py` + `scripts/bqml/train_popularity.sql` + `infra/terraform/modules/data/` | property-popularity model を BQML で学習し、reranker の副経路 (popularity boost) として bolt-on |
| **T2** | **Dataflow Flex Template** | `ml/streaming/` + `infra/terraform/modules/streaming/` | ranking_log → 特徴量 streaming ingest (low-latency ctr / fav_rate refresh)。Phase 5 の Feature Store 入力源を強化 |
| **T4** | **TreeSHAP Explainability** | `ml/serving/reranker.py::/explain` ルート | `/search?explain=true` で TreeSHAP attributions 取得 (Vertex CPR 互換 server) |
| **T5** | **Monitoring SLO + burn-rate alert** | `infra/terraform/modules/slo/` | search-api availability / latency SLO + burn-rate alert |
| 補助 | **Composer-managed BQ monitoring query** | Composer DAG `monitoring_validation` 内の BigQueryInsertJobOperator | feature skew / model output drift を Composer DAG が定期実行 (Phase 5 の Scheduled Query を DAG に統合) |

### 1.3 中核 5 要素 (不変)

中核 5 要素 (Meilisearch BM25 + **Vertex Vector Search** + multilingual-e5 + RRF + LightGBM LambdaRank) と `/search` デフォルト挙動は Phase 5 から継承して **絶対に変えない**。PMLE 4 技術は **default off の opt-in** で導入する (新 feature flag は Phase 5 挙動 default を維持)。

## 2. Phase 6 の中核アーキテクチャ

```
[Cloud Composer (Managed Airflow Gen 3)] ← 本線 orchestrator
   │
   ├─[DAG: daily_feature_refresh] (01:00 JST)
   │    ├─ Dataform run (feature_mart.* refresh)
   │    ├─ Feature View sync (Vertex AI Feature Online Store 更新)
   │    └─ (Phase 6B 増設) Composer-managed BQ monitoring query
   │
   ├─[DAG: retrain_orchestration] (04:00 JST、本線 retrain schedule)
   │    ├─ check_retrain (BQ feature freshness / model drift 判定)
   │    ├─ Vertex AI Pipelines submit (data_job → training_job → evaluation_job)
   │    ├─ wait SUCCEEDED
   │    ├─ (Phase 6B 増設) BQML training (T1 副経路)
   │    └─ Model Registry promote (NDCG 閾値 PASS 後、production alias 切替)
   │
   └─[DAG: monitoring_validation] (04:30 JST)
        ├─ run_feature_skew (BigQueryInsertJobOperator)
        ├─ run_model_output_drift (BigQueryInsertJobOperator)
        ├─ check_slo_burn_rate (T5)
        └─ (Phase 6B 増設) TreeSHAP Explainability eval

  ↓ submit
[Vertex AI Pipelines (KFP v2)] ← 下位 ML executor
   │
   ├─[Vertex Vector Search] (ME5 ベクトル serving index)
   ├─[Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store)]
   ├─[Vertex Model Registry] (production / staging alias)
   └─[Vertex Model Monitoring v2]
                                  ▲
                                  │
                          [Cloud Run: search-api]
                          ├─ Meilisearch (BM25)
                          ├─ multilingual-e5 (Vertex Endpoint)
                          ├─ Vertex Vector Search (ANN)
                          ├─ RRF 融合
                          ├─ Feature Online Store (Feature View 経由 fresh feature)
                          ├─ LightGBM LambdaRank (Vertex Endpoint)
                          ├─ + (T1 opt-in) BQML popularity bolt-on
                          ├─ + (T4 opt-in) /search?explain=true → TreeSHAP
                          └─ Pub/Sub (ranking_log / feedback)
                                  │
                                  ▼
                          + (T2 opt-in) Dataflow streaming
                          + (T5 opt-in) Monitoring SLO + burn-rate
```

### 2.1 主要コンポーネント

- **Composer 環境**: Composer Gen 3 SMALL、`asia-northeast1`、`infra/terraform/modules/composer/`。`mlops-dev-a` PDCA 都合で `make destroy-all` で破棄、`make deploy-all` で再作成
- **3 本 DAG**: `pipeline/dags/{daily_feature_refresh,retrain_orchestration,monitoring_validation}.py`
- **Composer SA** (`sa-composer`): `roles/composer.worker` / `roles/aiplatform.user` / `roles/bigquery.{jobUser,dataViewer}` / `roles/run.invoker`
- **`make composer-deploy-dags`**: DAG file を Composer DAG GCS bucket に upload
- **PMLE 4 技術**: 上記 §1.2 の 4 ファイル群

## 3. Cloud Composer canonical (上下関係 + カニバリ禁止)

詳細は [Phase 7 docs/01 §3](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md)。要点:

- **Composer (上位 orchestrator)**: Dataform run + BQ feature freshness check + Dataflow Flex Template launch + BQML train + Vertex AI Pipeline submit + monitoring query
- **Vertex Pipelines (下位 ML executor)**: train_pipeline / eval_pipeline / register / promote (Composer から submit される側)
- **NG (カニバリ禁止)**:
  - Composer DAG が `train/evaluate/register` を直接実装する (Vertex Pipelines にもある → 重複)
  - Composer DAG schedule + Vertex `PipelineJobSchedule` が同じ retrain を二重起動する
  - 軽量経路 (Cloud Scheduler / Eventarc / Cloud Function) が本線 retrain trigger に再昇格する

### 3.1 Phase 6 内部マイルストーン (6A / 6B)

- **6A**: Cloud Composer DAG 本線昇格 (`daily_feature_refresh` / `retrain_orchestration` / `monitoring_validation` の最小実装、Vertex `PipelineJobSchedule` 撤去)
- **6B**: PMLE 4 技術を DAG に増設 (Composer-managed BQ monitoring query + BQML training + TreeSHAP Explainability eval)

## 5. 不変ルール

- **題材は Phase 5 と同じ不動産ハイブリッド検索ドメインを維持** (PMLE 試験のための独立ドメインは作らない)
- **ハイブリッド検索中核コード (Meilisearch BM25 + Vertex Vector Search + multilingual-e5 + RRF + LightGBM LambdaRank) の挙動は絶対に変えない** (Phase 5 から継承)
- **default feature flag では Phase 5 挙動を維持** (PMLE 4 技術は opt-in で有効化)
- **中核以外の改変は PMLE 学習のため自由** (新 Port / Adapter / Service / Endpoint / KFP component / Terraform モジュール / parity 6 ファイル同 PR 更新を積極使用)
- **Composer 本線 + Vertex `PipelineJobSchedule` 撤去 + 軽量経路の格下げ** はカニバリ禁止 (上下関係厳守)
- 中核を変える提案 (Meilisearch 置換 / LambdaRank 置換 / RRF 廃止 等) は事前合意必須

### 5.1 Feature parity invariant (6 ファイル同 PR 原則)

特徴量を追加 / 変更するとき、以下 6 つを **必ず同じ PR で揃える** (Phase 6 起点、Phase 7 へ継承):

1. `pipeline/data_job/dataform/features/property_features_daily.sqlx`
2. `ml/data/feature_engineering/ranker_features.py::build_ranker_features`
3. `ml/data/feature_engineering/schema.py::FEATURE_COLS_RANKER`
4. `infra/terraform/modules/data/main.tf` の `ranking_log.features` RECORD
5. `infra/sql/monitoring/validate_feature_skew.sql` の UNPIVOT
6. `infra/terraform/modules/vertex/main.tf::google_vertex_ai_feature_group_feature` × N
