# 02_移行ロードマップ（Phase 6 の決定的仕様 + 未対応項目）

本ドキュメントは **Phase 6 (`study-gcp-ml-engineer-cert`) の決定的仕様** を載せる。
「何を学ぶ / 何を作る」「何を学ばない / 何を作らない」の最終決定と、未対応の学習項目のみを管理する。対応済みの履歴は記載しない（`docs/README.md §2` の権威順位に従う）。

Phase 6 は Phase 3-5 と違い **検索ドメイン実装を前進させるフェーズではない**。GCP Professional Machine Learning Engineer (以下 PMLE) 認定相当の理解を総仕上げするための、**ドキュメント主体 + 小規模ハンズオン** のフェーズ。ただし題材としては **Phase 5 と同じ不動産ハイブリッド検索ドメインを維持する**（§1.4 参照）。

## 0. 最重要ルール — Phase 5 コードはなるべく変えない (不要な改変はダメージ)

Phase 5 から継承した以下のディレクトリ / ファイルは Phase 6 の全作業を通じて **なるべく変えない** 方針で扱う。不要な改変はダメージに繋がる。

```
6/study-gcp-ml-engineer-cert/
├── app/         # なるべく変えない
├── ml/          # なるべく変えない
├── pipeline/    # なるべく変えない
├── infra/       # なるべく変えない
├── scripts/     # なるべく変えない
├── tests/       # なるべく変えない
├── monitoring/  # なるべく変えない
├── env/         # なるべく変えない
├── tools/       # なるべく変えない
├── pyproject.toml / uv.lock / Makefile / doppler.yaml / .github/  # なるべく変えない
└── docs/, labs/, README.md, CLAUDE.md  # Phase 6 で更新対象
```

### なぜ不要な改変がダメージなのか

1. **Phase 5 は完成系の教材である** — 231 tests / Port/Adapter / 9 SA 分離 / Vertex AI 全プリミティブ統合 / 6 ファイル parity invariant / `make check` PASS / `make deploy-all` ↔ `make destroy-all` 完走という **整合の取れた状態**。不用意に崩すと Phase 5 の教育価値そのものを毀損する
2. **Phase 6 の目的 (PMLE 総仕上げ) には改変がほぼ不要** — 試験範囲のトピックは Phase 5 実装を **比較対象として参照する** ことで概ねカバーできる（§2 マッピング参照）。新規ハンズオンは `labs/` 配下に隔離すれば足りるため、既存ディレクトリに手を入れる必要は通常発生しない
3. **改変は 6 ファイル parity invariant を簡単に壊す** — Phase 5 は Dataform SQL / Python feature_engineering / feature_schema / infra Terraform / monitoring SQL / Vertex Feature Group の 6 ファイルを同一 PR で揃える制約がある。Phase 6 の学習目的で片方だけ触ると、Phase 5 の parity test が FAIL し Phase 5 自体が壊れる
4. **`.dockerignore` / Terraform / KFP パイプラインは連鎖事故を起こしやすい領域** — Phase 5 CLAUDE.md の「紛らわしい点」「境界維持の長期方針」に記録された過去の事故が、軽率な改変で再発する
5. **PMLE 試験対策としての劣化** — Phase 5 の実装を基準に「別選択肢を評価する」のが試験対策の軸。基準が揺れると比較ができず、試験判断を学ぶ題材として機能しなくなる

### ダメージになりやすい改変の例 (避けられるなら避ける)

以下は Phase 5 の整合を壊しやすいので、**避けられる手段 (labs/ 配下での代替等) があれば優先する**:

- 既存ファイルの編集・リネーム・削除・移動
- 既存ディレクトリへの新規ファイル追加 (例: `app/services/new_handler.py` / `ml/train/src/train/new_feature.py`)
- `pyproject.toml` / `uv.lock` への依存追加 (Phase 5 の CI / parity test に影響する)
- `Makefile` / `.github/workflows/` の追記・修正
- `infra/` への Terraform リソース追加 (apply すれば GCP 側も Phase 5 状態から逸脱する)
- `tests/` への新規テスト追加 (Phase 5 の `make check` 集合を変える)

### やむを得ず変える場合

Phase 6 作業中に既存ディレクトリを触らざるを得ない判断が出たら、**作業開始前に** User に相談し、以下を示した上で合意を得る:
- 改変対象のファイル / ディレクトリ
- Phase 5 の parity invariant / make check / make deploy-all / make destroy-all への影響評価
- 回避策 (labs/ 配下での代替実装) を先に検討した上で、それでも既存改変が必要と判断した理由

**原則**: 「触らずに済ませる方法があるなら触らない」が正しい。改変が Phase 6 の学習に不可欠と判断された場合のみ、合意の上で最小範囲に限定して行う。

## 1. スコープ (採用 / 不採用)

### 1.1 採用 (Phase 6 内で扱う)

- ✅ **PMLE 試験範囲の整理** — 公式試験ガイドの 6 セクションを `docs/` 配下で章立てしてカバーする
- ✅ **Phase 5 実装との差分比較による設計判断の言語化** — 「Phase 5 は LambdaRank を Vertex Training で学習しているが、BQML ならどう書くか / 何が変わるか」のように、Phase 5 を基準に別選択肢を評価する
- ✅ **Responsible AI** — Fairness (住宅差別の観点を含む) / Explainable AI (既存 reranker への attribution 取得) / Model Cards / データガバナンス (IAM, VPC-SC, CMEK) を整理する
- ✅ **GenAI / RAG** — Vertex AI (Gemini API) / Model Garden / Agent Builder / Vector Search を、**Phase 5 のハイブリッド検索の自然な拡張**として整理する (lexical + semantic + rerank に Gemini を重ねれば RAG になる)
- ✅ **MLOps 運用判断** — モデル更新戦略 (Phase 5 の canary を基準に shadow / blue-green を比較) / 再学習トリガー / コスト最適化 を整理する
- ✅ **監視** — Phase 5 で attach した Vertex Model Monitoring v2 を基準に、Cloud Monitoring SLI/SLO の位置付けを整理する
- ✅ **小規模ハンズオン** — Phase 5 で未体験の領域 (BQML / Dataflow / Model Garden / Agent Builder / Vertex Vector Search / Explainable AI / Cloud Monitoring SLO) に限って、**`labs/<topic>/` 配下に隔離した単発スクリプト / 手順書** として行う

### 1.2 不採用 (Phase 6 で作らない / 触らない)

- ⚠️ **Phase 5 コード (`app/` / `ml/` / `pipeline/` / `infra/` / `scripts/` / `tests/` / `monitoring/` 等) の不要な改変** — §0 で規定。避けられる改変はしない。やむを得ない場合は User 合意を得て最小範囲で
- ⚠️ **Phase 5 状態の GCP リソース (既存 Endpoint / Pipeline / BQ テーブル / Artifact Registry) への書き込み** — 原則参照のみ。新規リソースは `labs/` 配下で別名を使って作り、手順書の末尾で必ず削除する
- ❌ **新しい不動産検索機能の実装** — Phase 5 までで完結。Phase 6 では検索ドメインに新規コードを追加しない
- ❌ **Terraform / CI/CD の本格整備** — Phase 4/5 で既習のため Phase 6 では行わない。ハンズオンは `gcloud` / コンソールで完結させる
- ❌ **本番相当の SLO / SRE 運用** — 監視は概念整理に留め、実アラートの配線は行わない
- ❌ **Port/Adapter レイヤリング** — Phase 2-5 の設計テーゼ。`labs/` 配下のハンズオンコードには強制しない (単発スクリプトで良い)

- ❌ **LightGBM + multilingual-e5 + Meilisearch の置換** — Phase 5 の非負制約を継承。比較検討 (BQML / AutoML / Model Garden) は OK だが、Phase 5 実装の置換はしない

### 1.4 題材: 不動産ハイブリッド検索ドメインを維持する

Phase 6 の題材は **Phase 5 と同じ不動産ハイブリッド検索ドメインを維持する**。理由:

- **Phase 5 実装を基準に「別選択肢を評価する」形式が PMLE 試験対策として最も有効** — 試験問題は「Company X は既に Y をしている。最適な選択は？」のケーススタディ形式。抽象トピックより具体ドメインで判断を積む方が試験対策として有利
- **Phase 3-5 の設計思想 (同じドメインで実行基盤だけ差し替える) の一貫性を維持する** — Phase 6 だけドメインを捨てると Phase 3-5 で積んだ軸が断絶する
- **Responsible AI / Fairness が実題材 (住宅差別の観点) で学べる** — 無機質な toy dataset より学習価値が高い
- **GenAI / RAG が Phase 5 ハイブリッド検索の自然な拡張になる** — lexical + semantic + rerank + Gemini 要約で RAG パターンが組める (§2 参照)
- **新ドメインを用意するコストを回避** — トピックごとに別データを用意する手間がゼロになる

**重要**: 題材を維持することと Phase 5 コードを改変することは別問題。ドメインは同じでも `app/` / `ml/` / `pipeline/` / `infra/` は **なるべく変えない** (§0)。新規ハンズオンは原則 `labs/<topic>/` に隔離する。

## 3. ハンズオンの方針

### 3.1 ハンズオンを置く条件

次のいずれかを満たすトピックのみ、`labs/<topic>/` としてハンズオンを作る:

- Phase 4/5 で **触れていない** GCP ML サービス (例: BQML, Dataflow, Model Garden, Agent Builder, Vertex Vector Search, Explainable AI)
- PMLE 試験で **コード / 設定レベルの理解が問われる** と判断したもの (例: BQML `CREATE MODEL` 構文、Vertex Explainable AI の `explanation_spec`)

逆に、Phase 4/5 で既に手を動かしたトピック (KFP Pipelines / Endpoints / Registry / Model Monitoring v2) は **再ハンズオンしない**。ドキュメント側で「Phase 5 の `xxx` を参照」とリンクする。

### 3.2 ハンズオンの規模制約 (不要なダメージを避けるため)

- **`labs/<topic>/` に隔離する** — Phase 5 コード (`app/` / `ml/` / `pipeline/` / `infra/`) を import / 継承することは原則避ける (読み取り参照は可)
- **1 トピック = 1 ディレクトリ = 1 README + 最小スクリプト**
- **既存 GCP リソースには原則書き込まない** — Phase 5 状態の Endpoint / Pipeline / Registry / BQ テーブルは参照のみで進める。新規リソースは `labs-<topic>-` プレフィックスなど別名で作る
- GCP 認証が必要な場合は `gcloud` / コンソール手順として書き、Terraform / CI は通さない
- `uv` / `pytest` / `mypy` などの品質チェックは **強制しない** (Phase 6 はドキュメント主体)
- 作ったリソースは手順書の最後で必ず削除する (`gcloud ... delete`)。課金リスク / Phase 5 状態への混入を残さない

### 3.3 既存 BQ テーブルの扱い

Phase 5 の BigQuery テーブル (`feature_mart.*` / `mlops.*` / `raw.properties`) はハンズオンで**読み取り参照してよい**。ただし以下は Phase 5 状態を壊すため原則避ける:

- 既存テーブルへの INSERT / UPDATE / DELETE / DDL (CREATE / DROP / ALTER)
- Scheduled Query の作成 / 変更
- Dataform 定義の変更

ハンズオンで書き込みが必要な場合は、`labs_<topic>` データセットを新規作成し、手順末尾で dataset ごと削除する。
