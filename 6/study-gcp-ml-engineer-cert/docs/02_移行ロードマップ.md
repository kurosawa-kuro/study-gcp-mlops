# 02_移行ロードマップ (Phase 6 の決定的仕様 + 未対応項目)

本ドキュメントは **Phase 6 (`study-gcp-ml-engineer-cert`) の決定的仕様** を載せる。「何を学ぶ / 何を作る」「何を学ばない / 何を作らない」の最終決定と、未対応の学習項目のみを管理する。対応済みの履歴は記載しない (`docs/README.md §2` の権威順位に従う)。

Phase 6 の主目的は **GCP Professional Machine Learning Engineer (PMLE) 試験範囲の技術を Phase 5 実コード (`app/` / `ml/` / `pipeline/` / `infra/`) に実際に組み込み、動くコードとして手を動かして学ぶ** こと。Phase 3-5 の「同じドメインで実行基盤を差し替える」思想の延長線上にあり、**Port/Adapter のまま adapter / 副経路 / 新規エンドポイント / 新規 Terraform を追加** する形で統合する。

---

## 0. 最重要ルール — ハイブリッド検索というテーマと中核コードは絶対に変えない

Phase 6 の全作業を通じて **以下だけは絶対に変えない**:

- **題材 / ドメイン**: 不動産ハイブリッド検索 (クエリ文 + フィルタ → 物件ランキング上位 20 件)
- **ハイブリッド検索の中核コードと挙動**:
  - (1a) Meilisearch BM25 (lexical)
  - (1b) BigQuery `VECTOR_SEARCH` + multilingual-e5 embedding (semantic)
  - (2) RRF 融合
  - (3) LightGBM LambdaRank 再ランク
  - `/search` エンドポイントの現行動作 (デフォルト構成での出力 = 既存テスト全通過)

この中核を **置換・削除・無効化する変更はしない**。親リポ `CLAUDE.md` の非負制約 (「ハイブリッド検索の基本構成は LightGBM + multilingual-e5 + Meilisearch を必須」) を Phase 6 でもそのまま継承する。

### 一方で変えてよいもの (Phase 6 の主目的)

- 新しい Port / Adapter の **追加** (alternative lexical / semantic / rerank backend、副経路)
- 既存エンドポイントへの **追加フィールド / 追加クエリパラメータ** (例: `/search?explain=true` で attribution を返す)
- **新規エンドポイント追加** (例: `/rag`)
- **新規 pipeline / KFP component 追加** (例: Dataflow streaming、Gemini enrichment)
- **新規 Terraform モジュール追加** (例: Monitoring SLO、Vertex Vector Search index、Dataflow Flex Template)
- **既存 feature engineering への列追加** (6 ファイル parity invariant を同 PR で揃える前提で)
- 既存 reranker endpoint への `explanation_spec` 追加

= **PMLE 技術の学習に必要な修正は Phase 5 実コードに対して自由に行う**。中核挙動を壊さないことだけ守る。

### NG (やり方として誤り)

- `labs/<topic>/` のように **既存コードから隔離された手順書 / GUI ノート** を書いて「学習した」ことにする — User の意図と真逆
- 設計判断を **問題集 / Q&A** としてドキュメントに書き連ねる — User の意図は「動くコードで学ぶ」
- 「Phase 5 コード全体をなるべく変えない」を非負制約に置く — 中核以外の改変は Phase 6 の主目的そのもの

### 3.1 共通原則

- **Port/Adapter 境界を維持** — `scripts/checks/layers.py::RULES` に違反しない。新 adapter は `adapters/__init__.py` で re-export し、services から Port 経由で使う
- **`make check` を通す** — Phase 6 追加コードも ruff / ruff format / mypy strict / pytest の CI 同等チェックに含める
- **6 ファイル parity invariant を壊さない** — 新 feature 列を追加する場合は `definitions/features/property_features_daily.sqlx` / `common.feature_engineering.build_ranker_features` / `FEATURE_COLS_RANKER` / `infra/.../ranking_log.features` / `monitoring/validate_feature_skew.sql` / Vertex Feature Group を同 PR で揃える
- **Terraform で宣言** — 手で `gcloud` / `bq` を叩かない。`make tf-plan` → `make tf-apply` 経由
- **非負制約 (`--no-allow-unauthenticated` / WIF / SA 分離)** は Phase 5 と同一。新 endpoint / Cloud Function も SA 分離を守る
- **default は Phase 5 挙動** — feature flag は env or `env/config/setting.yaml` に追加、default 値で既存テストが全通過する状態を保つ

### 3.2 学習リポとしての最小スペック選定 (コスト配慮)

- Vertex Endpoint / Vector Search endpoint / Dataflow job はすべて **min=1 / max=1 / 小型マシン** をデフォルトにする
- Agent Builder / Vector Search / Dataflow は常時課金されるサービスのため、**学習終わり次第 destroy する運用** を `make destroy-*` に追加
- コストが増える方向の変更 (`min-instances > 1` / large machine / GPU) は User 合意を明示的に取ってから
