## 3. 設計判断集 (想定 Q&A)

PMLE 試験で問われる「2 択 / 3 択の判断」を言語化する。Phase 4/5 で実際に下した判断と、Phase 6 で新規に整理する判断を混在させる。本節は随時追記する。

### 3.1 モデリング

- **BQML を使うか、Vertex Training を使うか**
  - BQML: SQL で完結、前処理済みデータが BQ にある、線形 / boosted tree / matrix factorization / ARIMA_PLUS 等で足りる場合
  - Vertex Training: カスタムモデル (LightGBM LambdaRank / PyTorch / TF)、分散学習、GPU/TPU が必要な場合
  - Phase 5 では LambdaRank のため Vertex Training を採用。BQML は Phase 6 で補完ハンズオン

- **AutoML を使うか、カスタムを使うか**
  - AutoML: ラベル済みデータが揃っており、ハイパラ調整のコストを回避したい場合
  - カスタム: ランキング学習 / 特殊なロス / 独自アーキテクチャが必要な場合

### 3.2 サービング

- **Online Endpoint か、Batch Prediction か**
  - Online: ユーザリクエスト同期、レイテンシ制約あり (Phase 5 の `search-api` はこれ)
  - Batch: 夜間バッチで大量スコアリング、レイテンシ不問、コスト優先
  - 判断軸: レイテンシ SLO / リクエスト頻度 / スコア対象の静的さ

- **Vertex Endpoint か、Cloud Run + CPR か、Cloud Run のみか**
  - Vertex Endpoint: Model Registry と連携、カナリア / トラフィック分割が組み込み、Monitoring v2 を attach できる (Phase 5 採用)
  - Cloud Run + CPR: Phase 5 で採用。Endpoint ランタイムとして Custom Prediction Routine を使う
  - Cloud Run のみ: モデル推論を自前管理。Registry 連携や canary は自前で組む

### 3.3 パイプライン

- **Vertex AI Pipelines か、Cloud Composer (Airflow) か、Workflows か**
  - Vertex AI Pipelines (KFP): ML Metadata 自動連携、lineage 追跡、Phase 5 採用
  - Cloud Composer: 汎用 DAG、ETL と ML が混在、演算子エコシステムが必要
  - Workflows: サーバレスのシンプルな状態遷移、ML Metadata 連携は不要

### 3.4 Responsible AI

- **Explainable AI をいつ使うか** — feature attribution が必要 (規制業界 / ステークホルダー説明 / デバッグ)
- **Model Card をいつ書くか** — モデル公開時・ステークホルダー説明時・監査対応時
- **Fairness / Bias チェックをいつ走らせるか** — 学習データ作成後・モデル評価時・本番リリース前

### 3.5 コスト

- **Preemptible / Spot VM をいつ使うか** — バッチ学習 / 中断耐性のある前処理。Online Endpoint では使わない
- **Region 選定** — データ所在 / レイテンシ / 料金。`asia-northeast1` は Phase 4/5 の固定値
- **Autoscaling の `min-instances`** — コールドスタート許容なら `0`、許容しないなら `1+` (Phase 5 は `min=1`)

本節は `02_移行ロードマップ.md §4.1` に従い最低 20 問まで拡張する。

## 4. Responsible AI

PMLE 試験では Responsible AI の設問が近年増えている。Phase 6 では以下を整理する:

- **Vertex Explainable AI** — feature attribution (Sampled Shapley / Integrated Gradients / XRAI)。AutoML / custom model それぞれでの取得方法
- **Model Cards** — Vertex AI Model Registry から Model Card を出力する手順
- **Fairness Indicators** — TFMA の FairnessIndicators を使ったオフライン評価
- **データガバナンス** — IAM (最小権限 / 9 SA 分離は Phase 5 既習) / VPC Service Controls / CMEK (Customer-Managed Encryption Keys) / Data Loss Prevention (DLP)
- **プライバシー** — 差分プライバシー (TensorFlow Privacy) / Federated Learning の位置付け (試験範囲としての存在を押さえる程度)

実ハンズオンは `labs/explainable-ai/` に絞る (`02_移行ロードマップ.md §4.2`)。

## 5. GenAI / RAG

PMLE 試験に GenAI が組み込まれた範囲をカバーする:

- **Vertex AI (Gemini API)** — `generate_content` の基本 / system instruction / function calling / JSON mode
- **Model Garden** — 公開モデル (Gemini / Claude / Llama / Gemma) の選び方、fine-tuning (Supervised Tuning / RLHF) の位置付け
- **Vertex AI Agent Builder** — Search App / Chat App / Agent の 3 種、use case 別の選び方
- **Vector Search (旧 Matching Engine)** — Vertex AI Vector Search の index 作成 / query 経路。BigQuery `VECTOR_SEARCH` (Phase 4/5 採用) との使い分け基準
- **RAG パターン** — retrieval (Vector Search / BigQuery) + augment (prompt) + generate (Gemini) の 3 段構成を、Phase 3-5 のハイブリッド検索と比較して位置付ける

小規模ハンズオンは `labs/model-garden/` / `labs/agent-builder/` / `labs/vector-search/` に分割する (`02_移行ロードマップ.md §4.2`)。

## 6. MLOps 運用判断

Phase 4/5 で実装した運用機能を、**試験で問われる判断軸** として言語化し直す:

- **モデル更新戦略** — canary (Phase 5 採用、traffic_split={new:10, current:90}) / shadow / blue-green の違いと使い分け
- **再学習トリガー** — 精度劣化 (NDCG_DEGRADATION=0.03, Phase 5 採用) / データドリフト (Model Monitoring v2) / 定期 (STALE_DAYS=7) の 3 経路
- **Training-Serving Skew 対策** — Feature Store / parity test / Dataform lockstep (Phase 5 で 6 ファイル同一 PR 原則として確立)
- **SLI/SLO 設計** — availability / latency (p95) / quality (NDCG) の 3 軸。Cloud Monitoring でのアラート構成
- **コスト最適化** — preemptible / region 選定 / autoscaling / Endpoint vs Batch / BQ slots vs on-demand
