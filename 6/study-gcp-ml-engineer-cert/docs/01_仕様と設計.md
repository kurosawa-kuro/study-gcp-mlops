## 1. 共通前提

- **目的**: PMLE 認定試験範囲の GCP ML 技術を、Phase 5 の不動産ハイブリッド検索コードに **adapter / 副経路 / 追加エンドポイント / 追加 pipeline として統合**し、動くコードで手を動かして学ぶ
- **題材**: Phase 5 と同じ不動産ハイブリッド検索 (不変)
- **前提知識**: Phase 4 (GCP マネージド構成) + Phase 5 (Vertex AI プリミティブ) を既習扱い
- **開発環境**: WSL + `uv` workspace + `make check` / `make tf-*` / Terraform 1.9+ (Phase 5 継承)
- **GCP プロジェクト**: `mlops-dev-a` / リージョン `asia-northeast1` (Phase 4/5 継承)
- **実行方式**: Phase 4/5 と同じ。Phase 6 で追加したコードも `make check` / Terraform / WIF 経由 CI でデプロイされる

## 2. PMLE 試験範囲と Phase 5 実装の対応

PMLE 試験ガイドの 6 セクションごとに、**Phase 5 で既に実装済みの内容** と **Phase 6 で統合する PMLE 技術** を並べる。Phase 6 では右列の統合を `app/` / `ml/` / `pipeline/` / `infra/` への改変として実装する:

| # | PMLE セクション | Phase 5 既習 (中核 = 不変) | Phase 6 統合 (実コードに新規追加する PMLE 技術) |
|---|---|---|---|
| 1 | Architecting low-code ML | LightGBM LambdaRank (Vertex Training + CPR Endpoint) | **BQML** (BOOSTED_TREE_REGRESSOR 補助スコア) / **Model Garden (Gemini)** による description enrichment / **Agent Builder** 副経路 |
| 2 | Collaborating within and across teams | Vertex AI Experiments / Feature Group / Model Registry | (追加統合なし — Phase 4/5 で網羅) |
| 3 | Scaling prototypes | Vertex AI Pipelines (KFP) での学習 / Python 前処理 | **Dataflow** streaming による ranking-log 前処理 (並列経路) |
| 4 | Serving and scaling | Vertex AI Endpoints (online) / Cloud Run Service | **Vertex AI Vector Search** を alternative semantic backend として adapter 追加 / **RAG (`/rag` エンドポイント)** 追加 |
| 5 | Automating and orchestrating | Vertex AI Pipelines / Cloud Scheduler / Eventarc / Cloud Functions | (統合実装の Terraform / Workflow はすべて既存 IaC に追加) |
| 6 | Monitoring | Vertex AI Model Monitoring v2 (Endpoint attach 済み) | **Cloud Monitoring SLI/SLO + AlertPolicy** を `search-api` に attach / **Explainable AI** を reranker endpoint に attach (`explanation_spec` + `/search?explain=true`) |

左列 (中核) は §0 により不変。右列を Phase 6 で実装する。

## 3. 統合トピック詳細設計 (ファイル配置 / Port / Adapter / feature flag)

各トピックで **どの Port を追加するか、どの adapter ファイルに書くか、どの flag で Phase 5 挙動を保つか** を具体化する。`docs/02_移行ロードマップ.md §3` を詳細化したもの。

### T1. BQML による補助スコア (ポピュラリティ / Matrix Factorization)

**目的**: BQML の `CREATE MODEL` を実際に叩き、Phase 5 の ranker に補助スコア列として取り込む。PMLE で頻出の BQML 構文 (`BOOSTED_TREE_REGRESSOR` / `MATRIX_FACTORIZATION` / `ML.PREDICT` / `ML.EVALUATE`) を実コードで触る。

**統合先**:

- **新 Port**: `app/src/app/ports/popularity_scorer.py::PopularityScorer` (Protocol: `score(property_ids) -> dict[str, float]`)
- **新 Adapter**: `app/src/app/adapters/bqml_popularity_scorer.py::BQMLPopularityScorer` (BQ `ML.PREDICT` を叩く実装)
- **BQML モデル定義**: `scripts/bqml/train_popularity.sql` (SQL で `CREATE OR REPLACE MODEL mlops.property_popularity` + `BOOSTED_TREE_REGRESSOR`)
- **BQML 再学習 make target**: `make bqml-train-popularity` (SQL を `bq query` で実行)
- **既存統合**: `app/src/app/services/ranking.py` が composition root で Port を受け取り、feature flag `BQML_POPULARITY_ENABLED=true` の時のみ rerank feature に加算 (`FEATURE_COLS_RANKER` への列追加 = 6 ファイル parity 全更新が必要)
- **Terraform**: BQML モデル所在 dataset は既存 `mlops` 維持、追加リソースなし

**中核への影響**: デフォルト flag=false → 既存 /search 挙動完全維持。flag=true で rerank feature が 10 → 11 列に増えるので parity 6 ファイル同時更新。

### T2. Dataflow による ranking-log streaming 前処理

**目的**: Apache Beam Python で Pub/Sub → windowed aggregation → BQ を書く経験。Phase 5 の BQ Subscription と並列に動かし、1 時間ごとの CTR を算出して BQ テーブルに書き込む。

**統合先**:

- **新ワークスペース**: `ml/streaming/` (`pyproject.toml` に apache-beam 依存追加、`src/streaming/pipeline.py` で Beam pipeline 定義)
- **BQ テーブル追加**: `mlops.ranking_log_hourly_ctr` (`infra/modules/data/main.tf` に追加、schema: `window_start TIMESTAMP / property_id STRING / impressions INT64 / clicks INT64 / ctr FLOAT64`)
- **Flex Template**: `ml/streaming/container/Dockerfile` + `ml/streaming/template_metadata.json` + `scripts/deploy/streaming_job_local.py` で Artifact Registry へ push & Dataflow job 起動
- **Terraform**: `infra/modules/streaming/` (Dataflow job 宣言 + Dataflow worker SA 新設 `sa-dataflow`)
- **既存統合**: BQ Subscription は維持、Dataflow は並列経路

**中核への影響**: なし (並列経路、/search 挙動に影響しない)。

### T3. Vertex AI Vector Search (alternative semantic backend)

**目的**: Matching Engine (ANN index) を実際に build / deploy / query し、Phase 5 の BQ `VECTOR_SEARCH` と切り替え可能にする。Port/Adapter 差し替えの学習素材そのもの。

**統合先**:

- **既存 Port**: `app/src/app/ports/candidate_retriever.py::SemanticRetriever` (既存) をそのまま使う
- **新 Adapter**: `app/src/app/adapters/vertex_vector_search_retriever.py::VertexVectorSearchRetriever`
- **Embedding export 新 component**: `ml/embed/src/embed/pipeline/components/export_to_matching_engine.py` (既存 `feature_mart.property_embeddings` → GCS JSONL)
- **Terraform**: `infra/modules/vector_search/` (index + endpoint + deployed_index、minimum replicas=1)
- **Feature flag**: `env/config/setting.yaml` + `app/src/app/config.py::Settings.semantic_backend = Literal["bq", "vertex"]`、default `"bq"`
- **Composition root**: `app/src/app/entrypoints/api.py::lifespan` で flag に応じて adapter を差す

**中核への影響**: なし (default bq)。flag=vertex にしてもレスポンススキーマ互換。

### T4. Explainable AI を reranker に attach

**目的**: Vertex Explainable AI の `explanation_spec` (Sampled Shapley / tabular) を既存 reranker Vertex Endpoint に実際に attach し、`/search?explain=true` で feature attribution を返す。

**統合先**:

- **既存ファイル変更**: `ml/serve/reranker_server.py` を Vertex CPR の `predict` + `explain` 両対応に拡張 (google-cloud-aiplatform の `Prediction.Prediction` + `Prediction.Explanation` に合わせる)
- **Terraform**: `infra/modules/vertex/reranker_endpoint.tf` の `google_vertex_ai_endpoint_deployed_model` に `explanation_spec = { parameters, metadata }` を追加
- **既存エンドポイント拡張**: `app/src/app/entrypoints/api.py::/search` に `explain: bool = False` クエリパラメータを追加。`True` の時だけ reranker `explain()` を叩き、`SearchResponse.items[*].attributions` に attribution を埋める
- **新 schema フィールド**: `app/src/app/schemas/search.py::SearchItem` に optional `attributions: dict[str, float] | None = None`
- **既存 adapter 変更**: `app/src/app/adapters/candidate_retriever.py` 系には触らない。reranker 呼び出し側 (`services/ranking.py`) のみ拡張

**中核への影響**: なし (default `explain=false` → 既存レスポンス同一)。

### T5. Cloud Monitoring SLO + AlertPolicy

**目的**: PMLE で頻出の SLI / SLO / Error Budget / Burn Rate を Terraform で実際に宣言する。Phase 5 で attach 済みの Vertex Model Monitoring v2 (モデル drift 軸) と併存させ、サービス品質軸の監視を追加。

**統合先**:

- **新 Terraform モジュール**: `infra/modules/slo/main.tf` — `google_monitoring_service` (Cloud Run ベース) + `google_monitoring_slo` × 2 (availability 99.0% / latency p95 < 500ms) + `google_monitoring_alert_policy` (burn rate fast/slow) + `google_monitoring_notification_channel` (email = `oncall_email` from `setting.yaml`)
- **既存 Terraform**: `infra/modules/compute/` (search-api Cloud Run) には変更なし、SLO モジュールから Service を参照
- **make target 追加**: `make ops-slo-status` (SLO burn rate を `gcloud monitoring` で確認)
- **app コード変更**: なし (監視の追加のみ)

**中核への影響**: なし (観測のみ)。最初に着手する候補 (parity 影響ゼロ / 依存浅い / Terraform のみ)。

### T6. RAG エンドポイント (Gemini)

**目的**: Phase 5 ハイブリッド検索 (retrieval) + Gemini (generate) で最小 RAG を組む。Phase 5 の `/search` は変更せず、新規 `/rag` エンドポイントを追加。PMLE で頻出の GenAI / function calling / grounding を実コードで触る。

**統合先**:

- **新 Port**: `app/src/app/ports/generator.py::Generator` (Protocol: `generate(prompt, context) -> str`)
- **新 Adapter**: `app/src/app/adapters/gemini_generator.py::GeminiGenerator` (`google-genai` SDK、vertexai=True)
- **新 Service**: `app/src/app/services/rag_summarizer.py::RagSummarizer` (既存 `RankingService` を内部で使い、top-N を Gemini に渡す)
- **新エンドポイント**: `app/src/app/entrypoints/api.py::/rag` (POST, query + filter → `{query, top_properties, summary}` を返す)
- **新 schema**: `app/src/app/schemas/rag.py` (RagRequest / RagResponse)
- **依存追加**: `pyproject.toml` に `google-genai`
- **Terraform**: なし (Gemini API は Vertex AI API の一部、既存 API enable で到達可能)
- **SA 権限追加**: `sa-api` に `roles/aiplatform.user` (既存で付与済みのはず、要確認)

**中核への影響**: なし (新規エンドポイント、`/search` は不変)。

### T7. Agent Builder alternative lexical backend (副経路)

**目的**: Vertex AI Agent Builder (Search App) を副経路 lexical retriever として adapter 追加。Phase 5 の Meilisearch BM25 が主経路のまま維持される (親リポ非負制約準拠)。

**統合先**:

- **既存 Port**: `app/src/app/ports/candidate_retriever.py::LexicalRetriever` (新設 — 現状は Meilisearch 直呼びなら抽象化を先にする)
- **新 Adapter**: `app/src/app/adapters/agent_builder_lexical.py::AgentBuilderLexicalRetriever` (Discovery Engine API 経由)
- **Meilisearch adapter も明示化**: `app/src/app/adapters/meilisearch_lexical.py::MeilisearchLexicalRetriever` (既存 呼び出しを adapter として抽出)
- **Feature flag**: `LEXICAL_BACKEND=meili|agent_builder`、default `meili` (非負制約準拠)
- **Terraform**: `infra/modules/agent_builder/` (Datastore + Search App + Engine、最小構成)
- **Datastore 初期化**: 既存 `feature_mart.properties_cleaned` を source にして BigQuery Datastore

**中核への影響**: default で Meilisearch のまま。adapter 抽出リファクタのみ既存コードに入るが挙動は同一 (parity test で検証)。

### T8. Model Garden (Gemini) による description enrichment

**目的**: Model Garden から Gemini を使って物件 description を構造化 (tag / 対象ユーザ層抽出)。KFP pipeline に optional component として組み込む。

**統合先**:

- **新 KFP component**: `ml/embed/src/embed/pipeline/components/enrich_with_gemini.py`
- **新 BQ テーブル**: `feature_mart.properties_enriched` (property_id + tags ARRAY<STRING> + target_audience STRING) を `infra/modules/data/` に追加
- **新 pipeline flag**: embed pipeline の arg に `--enable-enrichment` (default false)
- **pipeline 組み込み**: 既存 `embed_pipeline.py` で `enable_enrichment=True` の時のみ component を先頭に挿入、後続の embed は enriched description を入力にできる (optional)
- **Terraform**: BQ テーブルのみ追加、新規サービスなし

**中核への影響**: default false → 既存 embed pipeline 挙動維持。true の時は description に enrich されたタグ文が prepend されるため、embedding 生成に影響する。学習 / embed run で明示的に切り替える想定。

## 4. Port / Adapter 境界と make check の扱い

Phase 6 で追加する Port / adapter / service は Phase 5 と同じ境界ルールに従う:

- **Port**: `<workspace>/src/<pkg>/ports/<name>.py` の Protocol。`ports/__init__.py` で re-export
- **Adapter**: `<workspace>/src/<pkg>/adapters/<name>.py`。外部 SDK (`google.cloud.*` / `google.genai` / `apache_beam` 等) は adapter でのみ import
- **Service**: `<workspace>/src/<pkg>/services/<name>.py`。Port だけを import、adapter は import しない
- **境界検知**: `scripts/checks/layers.py::RULES` に新 Port / service を追加。`make check-layers` で CLI 実行、`tests/arch/test_import_boundaries.py` で CI 実行

Phase 6 追加コードも **`make check` (ruff / ruff format / mypy strict / pytest) を PASS** させる。parity 6 ファイル invariant に波及する場合は同 PR で揃える。

## 5. 実装順序 (依存順 / 影響度順)

`docs/02_移行ロードマップ.md §4.1` の順序に従う:

1. **T5 Monitoring SLO** — Terraform のみ、app コード変更なし、parity 影響ゼロ。最初に着手
2. **T4 Explainable AI** — reranker endpoint に `explanation_spec` 追加 + `/search?explain=true`。既存コード微修正 + 新フィールド
3. **T3 Vertex Vector Search** — alternative adapter 追加 + Terraform。`make check` 影響軽微
4. **T6 RAG (Gemini)** — 新エンドポイント `/rag`、既存 /search への影響ゼロ
5. **T1 BQML** — 補助スコア flag 有効化時に parity 6 ファイル全更新が必要、影響が大きいので T3/T4 で Port/Adapter 追加に慣れてから
6. **T8 Model Garden enrichment** — KFP pipeline に新 component、embed 挙動に影響し得る
7. **T2 Dataflow** — 新ワークスペース + 新 SA + Flex Template、規模最大
8. **T7 Agent Builder** — Meilisearch adapter 抽出リファクタが前提、順序最後

ただし学習優先順は User が決める。PMLE 試験で優先したい技術があれば順番を入れ替える。
