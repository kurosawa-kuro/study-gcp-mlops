## 3. 設計判断集 (Phase 5 を基準にした Q&A)

PMLE 試験で問われる「2 択 / 3 択の判断」を、**Phase 5 が既に下した判断を基準に別選択肢と比較する形式**で言語化する。本節は随時追記する。

### 3.1 モデリング

- **Phase 5 は LightGBM LambdaRank を Vertex Training で学習している。BQML にしたら何が変わるか**
  - 書ける: SQL 1 本で `CREATE MODEL ... OPTIONS(model_type='BOOSTED_TREE_REGRESSOR')` / Matrix Factorization
  - 書けない: `lambdarank` objective / `group=request_id` でのランキング学習 / Vizier ハイパラ探索
  - 判断: **LambdaRank が必要な限り Vertex Training を維持する**。BQML は候補抽出側 (特徴量モデル) への補助として使う余地はある

- **Phase 5 の reranker を AutoML Tables にしたら何が変わるか**
  - AutoML: ラベルと特徴量を渡せばハイパラ / アーキテクチャを自動選択、`ctr` / `fav_rate` / `inquiry_rate` などの行動統計で回帰モデルが作れる
  - 失う: ランキング学習としての最適化 (NDCG 直接最適化)、group 単位の学習
  - 判断: **ランキング指標を直接最適化する必要があるため AutoML は不採用**。AutoML 採用が合理になるのは「分類 / 回帰で済むドメイン」

### 3.2 サービング

- **Phase 5 は online Endpoint。Batch Prediction にしたら何が変わるか**
  - Batch: 夜間バッチで全物件 × 全クエリをスコアリングしてキャッシュ、レイテンシ不問、コスト削減
  - 失う: リアルタイム性 (新着物件 / 直近行動の反映)、クエリ多様性対応
  - 判断: **自由文クエリのため online を維持**。物件側の定期スコアリング (人気度など) は Batch 化する余地あり

- **Phase 5 は Cloud Run + Vertex Endpoint 2 本。1 本の Vertex Endpoint (Model Registry 直結) にしたら何が変わるか**
  - 統合: Model Registry と 1:1、canary / traffic split が Endpoint 機能だけで完結
  - 失う: Cloud Run 層でやっている RRF fusion / build_ranker_features / feature parity の injection ポイント
  - 判断: **検索ロジックと推論を分離する責務分離のため Cloud Run + Endpoint を維持**

### 3.3 パイプライン

- **Phase 5 は Vertex AI Pipelines (KFP)。Cloud Composer (Airflow) / Workflows ならどう書くか**
  - KFP: ML Metadata / Experiments と自動連携、lineage 追跡、dsl.Condition での評価ゲート
  - Composer: 汎用 DAG、ETL 混在に強い、Airflow 演算子エコシステム、ML Metadata は自前
  - Workflows: サーバレス、シンプルな状態遷移、ML 連携は弱い
  - 判断: **ML Metadata + lineage が欲しい Phase 5 では KFP が最適**。ETL 主体なら Composer、単純な job chaining なら Workflows

### 3.4 データ基盤

- **Phase 5 は Dataform で feature_mart を作っている。Dataflow にしたら何が変わるか**
  - Dataform: SQL + JavaScript、SCD / assertions が書ける、BQ ネイティブ、スケジュール実行内蔵
  - Dataflow: Apache Beam、ストリーミングと大規模シャッフルに強い、SDK が Python/Java
  - 判断: **BQ 内完結のバッチ ETL は Dataform が軽い**。ストリーミング (Pub/Sub → BQ リアルタイム加工) は Dataflow が合理

- **Phase 5 は BigQuery `VECTOR_SEARCH`。Vertex AI Vector Search にしたら何が変わるか**
  - BQ VECTOR_SEARCH: IVF index、SQL で書ける、属性フィルタと JOIN で組める、p50 < 200ms
  - Vertex Vector Search: Matching Engine、ANN index (Tree-AH / ScaNN)、大規模 / 低レイテンシ、別 service
  - 判断: **200ms 実績が取れている Phase 5 では BQ を維持**。ドキュメント件数 / QPS が 10-100x になるなら Vertex Vector Search が合理

### 3.5 Responsible AI

- **Phase 5 reranker に Explainable AI を足すべきか** — Phase 5 への attach は §0 の「なるべく変えない」原則に反するため通常は避ける。Phase 6 では `labs/explainable-ai/` で独立モデルに試すのが基本方針
- **Model Card をいつ書くか** — モデル公開時・ステークホルダー説明時・監査対応時
- **Fairness / Bias チェックをいつ走らせるか** — 学習データ作成後・モデル評価時・本番リリース前。不動産ドメインでは「特定地域 / 属性への推薦偏り」が典型
- **住宅差別の観点** — `city` / 最寄駅 / 家賃帯ごとの推薦分布を分析、Fairness Indicators で disparate impact を評価

### 3.6 コスト

- **Phase 5 の `min-instances 1` は正しいか** — コールドスタート回避で必要 (ME5 ロードに数秒〜数十秒)。`0` にできるのは「コールドスタート許容 SLA」の場合のみ
- **Preemptible / Spot VM をいつ使うか** — バッチ学習 / 中断耐性のある前処理。Online Endpoint では使わない
- **Region 選定** — データ所在 / レイテンシ / 料金。`asia-northeast1` は Phase 4/5 の固定値 (変更しない)

本節は `02_移行ロードマップ.md §4.1` に従い最低 20 問まで拡張する。

## 4. Responsible AI

PMLE 試験では Responsible AI の設問が近年増えている。Phase 6 では以下を整理する:

- **Vertex Explainable AI** — feature attribution (Sampled Shapley / Integrated Gradients / XRAI)。AutoML / custom model それぞれでの取得方法。**Phase 5 reranker への attach は §0 の原則により原則避け**、`labs/explainable-ai/` で独立モデルに対して行う
- **Model Cards** — Vertex AI Model Registry から Model Card を出力する手順
- **Fairness Indicators** — TFMA の FairnessIndicators を使ったオフライン評価。**不動産ドメインでは「住宅差別」(特定地域 / 属性への推薦偏り) が実題材**
- **データガバナンス** — IAM (最小権限 / 9 SA 分離は Phase 5 既習) / VPC Service Controls / CMEK (Customer-Managed Encryption Keys) / Data Loss Prevention (DLP)
- **プライバシー** — 差分プライバシー (TensorFlow Privacy) / Federated Learning の位置付け (試験範囲としての存在を押さえる程度)

実ハンズオンは `labs/explainable-ai/` に絞る (`02_移行ロードマップ.md §4.2`)。

## 5. GenAI / RAG (Phase 5 ハイブリッド検索の自然拡張)

PMLE 試験に GenAI が組み込まれた範囲をカバーする。**Phase 5 の lexical + semantic + rerank は RAG の retrieval 段そのもの**であり、これに Gemini の generate を乗せれば RAG パターンが完成する。概念整理は既存資産に紐付けて行い、実装は `labs/` 配下に隔離して Phase 5 側はなるべく触らない (§0)。

- **Vertex AI (Gemini API)** — `generate_content` の基本 / system instruction / function calling / JSON mode
- **Model Garden** — 公開モデル (Gemini / Claude / Llama / Gemma) の選び方、fine-tuning (Supervised Tuning / RLHF) の位置付け
- **Vertex AI Agent Builder** — Search App / Chat App / Agent の 3 種。**Phase 5 Meilisearch BM25 との比較**を `labs/agent-builder/` に書く
- **Vertex AI Vector Search (Matching Engine)** — index 作成 / query 経路。**Phase 5 BigQuery `VECTOR_SEARCH` との使い分け基準**を `labs/vector-search/` に書く
- **RAG パターン** — retrieval (Phase 5 のハイブリッド検索に相当) + augment (prompt に top-N 物件を埋め込む) + generate (Gemini で要約 / 質問応答)。`labs/rag/` で最小構成を試す
- **Phase 5 ハイブリッド検索との構造比較**:
  - Phase 5 は「top-N 物件を rerank で並び替えて返す」
  - RAG は「top-N 物件を Gemini に渡して自然文の回答を生成する」
  - 同じ retrieval 基盤の上に serving 段を 1 段足すだけ、というのが設計視点での見どころ

## 6. MLOps 運用判断 (Phase 5 実装の言語化)

Phase 5 で実装した運用機能を、**試験で問われる判断軸** として言語化し直す (Phase 5 コードはなるべく変えず、設計意図を Q&A として書き起こす):

- **モデル更新戦略** — Phase 5 は canary (`traffic_split={new:10, current:90}`)。shadow / blue-green との違いと使い分け
- **再学習トリガー** — Phase 5 は 3 経路 (精度劣化 `NDCG_DEGRADATION=0.03` / データドリフト via Monitoring v2 / 定期 `STALE_DAYS=7`)。試験での選択肢パターン
- **Training-Serving Skew 対策** — Phase 5 は 6 ファイル同一 PR 原則 (Dataform SQL / Python feature_engineering / feature_schema / infra Terraform / monitoring SQL / Vertex Feature Group)。**この 6 ファイルは parity が崩れやすいので Phase 6 では特に触らない**
- **SLI/SLO 設計** — availability / latency (p95) / quality (NDCG) の 3 軸。Cloud Monitoring でのアラート構成は `labs/monitoring-slo/` で最小本数を試す
- **コスト最適化** — preemptible / region 選定 / autoscaling / Endpoint vs Batch / BQ slots vs on-demand
