## 0. 位置付け (Phase 3: 不動産ハイブリッド検索の Local 完成形)

Phase 3 は **コード化対象 Phase**。題材を **不動産ハイブリッド検索** に切り替えた最初の phase で、**ローカル完結で 3 段検索 (lexical + semantic + rerank) を一巡** させる Local 完成形。Phase 4 (GCP MLOps 土台) は Phase 3 からの足し算で構築する起点でもある。

Phase 2 → Phase 3 の飛躍 (詳細は親 [`README.md` §「Phase 2 → 3 の接続」](../../../README.md)):

- ドメイン: California Housing 回帰 → **不動産検索 (lexical + semantic + rerank の多段構成)**
- ML タスク: 回帰 → **ランキング学習 (LambdaRank / NDCG)**
- Adapter 数: 増 (Meilisearch / multilingual-e5 / Redis)

## 1. Phase 3 の中核 (不変)

中核 5 要素は Phase 3 / 4 / 7 のコード化対象で共通 + Phase 5 / 6 の論理 Phase でも参照される (canonical は Phase 7 docs/01 §2.3)。Phase 3 では Local 実装で構成する:

| 段 | Phase 3 (Local) 実装 |
|---|---|
| (1a) Lexical | **Meilisearch** (Docker) |
| Lexical 補助 | **Redis 同義語辞書** (query expansion、設計認識として明記、実装は学習用 substitute = Redis cache) |
| (1b) Semantic encoder | **multilingual-e5** (local) |
| (1b) Semantic vector store | **pgvector / 簡易 ANN** (Phase 4 で BigQuery `VECTOR_SEARCH` に足し算、Phase 7 完成版で Vertex Vector Search に発展) |
| (2) 融合 | **RRF** (Reciprocal Rank Fusion) |
| (3) Rerank | **LightGBM LambdaRank** (in-process) |

中核 5 要素 (Meilisearch BM25 / multilingual-e5 / ベクトルストア / RRF / LightGBM LambdaRank) の挙動・データフロー・デフォルト `/search` 応答は **Phase 3-7 共通で不変**。詳細は [Phase 7 docs/01 §2.3](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md)。

## 2. Phase 3 の主要コンポーネント

### 2.1 `app/`

- `/search` — 自由文クエリ + 構造化フィルタ (`city` / `price_lte` / `walk_min` 等) → 物件ランキング上位 N 件

- `/health` — liveness

### 2.2 `ml/`

- `data/` — embedding / 特徴量補助
- `training/` — LambdaRank 学習
- `evaluation/` — 評価 (NDCG@K)
- `serving/` — 検索 / 再ランキング補助

### 2.3 `pipeline/`

- バッチ処理 / 学習補助 / 評価補助

### 2.4 `definitions/`

- PostgreSQL migration

## 3. データフロー (Phase 3 = Local)

1. **Meilisearch** で BM25 + filter で候補を取る (lexical)
2. **multilingual-e5** で意味類似度を補う (semantic encoder + 簡易 ANN)
3. **RRF** で異種 score の順位を統合し top-K に絞る
4. **LightGBM LambdaRank** で最終順位を決める
5. **Redis** で短時間キャッシュ
6. **PostgreSQL** にログ・特徴量を保持 (Phase 4 で BigQuery に移行)

オンライン経路の詳細シーケンス図は [Phase 7 docs/01 §2.2.2](../../../7/study-hybrid-search-gke/docs/architecture/01_仕様と設計.md) を参照。

## 5. 不変ルール

- 題材は不動産ハイブリッド検索 (Phase 3 / 4 / 7 のコード化対象 + Phase 5 / 6 論理 Phase で共通)
- 中核 5 要素 (Meilisearch BM25 / multilingual-e5 / ベクトルストア / RRF / LightGBM LambdaRank) は Phase 3-7 で不変 (Phase 4 = BQ `VECTOR_SEARCH` / Phase 7 = Vertex Vector Search)
- Meilisearch は学習用 substitute (実案件想定は Elasticsearch + Redis 同義語辞書、詳細は Phase 5 docs/01 §「実案件想定の reference architecture」)
- Cloud / Vertex / GKE 系の責務は持ち込まない (Phase 4 で GCP 土台を足し算、Vertex AI / Composer / GKE / KServe は Phase 7 完成版コードに集約)
- 設計思想 (Port/Adapter / `core → ports ← adapters` / DI) は **Phase 2 から継承して不変**
