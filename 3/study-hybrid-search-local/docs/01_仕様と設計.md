# 01_仕様と設計
> Phase 2 — ローカル ハイブリッド検索 | 非 Elasticsearch 不動産検索ランキング基盤

> 対象：個人開発 / PoC / 小中規模サービス / 学習用ポートフォリオ  
> 前提：Elasticsearch を使わず、軽量・低コスト・高速開発を重視

---

## 1. 本資料の目的

不動産検索ドメインに対し、以下を満たす **非 Elasticsearch 構成の検索ランキング基盤** を定義する。

- 高速な物件検索
- 条件絞り込み（家賃・駅近・築年数など）
- 意味検索（ME5）
- LightGBM によるランキング最適化
- CTR / CVR / 問い合わせ率改善
- Docker / GCP / AWS 展開可能
- 後日 EC検索へ横展開可能

---

```text
FastAPI
````

担当：

* 特徴量保存

役割：

---

## 5.3 ME5

担当：

* 曖昧検索対応
* 類似物件検索
* 説明文意味理解

例：

```text
静かな住宅街
広めの部屋
駅近
住みやすい
```

役割：

> キーワードで拾えない意味検索補完

---

## 5.4 LightGBM Ranker

担当：

候補100件を最終順位化

目的関数：

* `objective = lambdarank`（ペアワイズランキング損失、NDCG を直接最大化）
* `metric = ndcg` / `ndcg_eval_at = [10]`
* 学習データは `group_sizes` でクエリ単位にグループ化（同一クエリ内でのみ順序を比較）

重視：

* 類似度
* 人気度
* 問い合わせ率
* 家賃妥当性
* 写真品質
* 駅人気

役割：

> CV最大化順位制御（ランキング学習は回帰/分類とは別タスク。クエリ内の順序を学ぶ）

---

## 5.5 Redis

担当：

* `/search` の結果キャッシュ
* 同一条件検索の応答高速化
* API / Meilisearch 負荷の平準化

設定：

* `REDIS_HOST`（既定: `redis`）
* `REDIS_PORT`（既定: `6379`）
* `SEARCH_CACHE_TTL_SECONDS`（既定: `120`）

役割：

> 検索レスポンスの短期キャッシュ層

### 1. 目的

- この構成は実運用最適化より、学習理解と MLOps 技術名の可視化を優先する
- レイヤ責務を名前で即判別できることを最重要要件とする

### 2. 命名ルール（必須）

- ディレクトリ名は小文字スネークケースのみ
- Python ファイル名は小文字スネークケースのみ
- ジョブのエントリーポイントは `pipeline/*_job/main.py` に統一
- API エントリーポイントは `app/main.py` に統一
- 評価・学習・推論・登録など MLOps 固有語を省略しない

### 3. ルート直下の必須ディレクトリ

- `app`
- `ml`
- `pipeline`
- `infra`
- `docs`
- `scripts`
- `tests`

上記は必須。別名（例: `applications`, `core`, `batch`, `qa`）への置換は禁止。

### 4. `ml` 配下の必須責務ディレクトリ

- `data`
- `training`
- `evaluation`
- `registry`
- `serving`
- `common`

上記は必須。`trainer` と `training` の混在、`eval` と `evaluation` の混在は禁止。

### 5. `pipeline` 配下の必須ジョブ名

- `data_job`
- `training_job`
- `evaluation_job`
- `batch_serving_job`
- `workflow`

ジョブごとのエントリーポイントは `main.py` に固定し、`run.py` `entry.py` などの別名は使わない。

### 6. `tests` 配下の必須区分

- `unit`
- `integration`
- `e2e`

テストはこの3区分に配置し、`api/` `ml/` などの責務別トップレベルを増やさない。

### 7. 禁止事項

- 旧構成（例: `common/src`, `pipeline/batch`, `pipeline`）の再導入
- 同義語ディレクトリの追加（例: `train`, `eval`, `serve`, `ops`）
- レイヤ責務をまたぐ配置（例: 学習コードを `app/` に置く）

### 8. 例外運用

- 一時互換レイヤを置く場合でも、最終形は本仕様に収束させる
- 例外を許可する場合は `docs/02_移行ロードマップ.md` に理由と期限を記録する

### 9. レビュー観点（チェックリスト）

- ルート必須ディレクトリが揃っているか
- `ml` と `pipeline` の必須サブディレクトリが揃っているか
- `pipeline/*_job/main.py` 形式になっているか
- 命名が小文字スネークケースに統一されているか
- 旧構成パスが新規追加されていないか

## デザインパターンの必須仕様

### 1. 目的

- 学習理解を優先し、責務分離と変更容易性を確保する
- テスト容易性と耐障害性を高めるため、外部連携の直接依存を排除する
- 特に GCP リソース連携は Port/Adapter を必須とする

### 2. 適用アーキテクチャ（必須）

- 基本は Ports and Adapters（Hexagonal Architecture）を採用する
- ドメイン/ユースケース層は外部 SDK やクラウド API を import しない
- 依存方向は内側（ドメイン）に向け、外側（インフラ）は内側に依存する

### 3. レイヤ責務（必須）

- `app`:
  - API 入出力とユースケース起動のみを担当する
  - 外部サービスの詳細実装を持たない
- `ml/*`:
  - 学習・評価・推論ロジックを担当する
  - 外部連携は Port 経由で呼び出す
- `infra` / adapter 実装:
  - GCP、DB、ストレージ、キューなどの具体実装を担当する
  - Port インターフェースを実装して差し替え可能にする

### 4. 外部連携の Port 定義（必須）

最低限、以下は Port を定義する。

- Model Artifact Port（モデル保存/取得）
- Experiment Tracking Port（ローカル runs/ への記録）
- Feature Store / Warehouse Port（BigQuery など）
- Message / Event Port（Pub/Sub など）
- Scheduler / Job Trigger Port（Cloud Scheduler/Eventarc など）

Port は抽象（interface/protocol/abc）として定義し、ユースケース層は Port のみ参照する。

### 5. Adapter 実装ルール（必須）

- GCP 連携（BigQuery, GCS, Pub/Sub, Vertex AI, Cloud Run, Scheduler, Eventarc）は GCP Adapter に隔離する
- Experiment Tracking は Tracking Adapter に隔離する
- Adapter 内でのみ SDK 初期化・認証・リトライ・例外変換を扱う
- Adapter は Port 契約を満たすことを契約テストで検証する

### 6. テスト戦略（必須）

- Unit Test:
  - ユースケース層は Port のモック/スタブで検証する
  - 外部 SDK を直接モックしない（Port をモックする）
- Integration Test:
  - Adapter 単体で外部サービス接続を検証する
- E2E Test:
  - 主要シナリオの疎通のみを最小本数で維持する

### 7. 禁止事項

- `app` や `ml` から GCP SDK を直接呼び出す実装
- ユースケース層で環境変数を直接参照して分岐する実装
- 外部入出力とビジネスロジックを1つの関数/クラスに混在させる実装
- テストでクラウド実サービス前提の結合に依存する実装（常時必須化）

### 8. 例外運用

- 例外的に直接連携が必要な場合でも、暫定実装として期限付きで扱う
- 例外は `docs/02_移行ロードマップ.md` に理由・影響範囲・解消期限を記録する

### 9. レビュー観点（チェックリスト）

- ユースケース層が Port のみに依存しているか
- GCP 連携が Adapter 層へ隔離されているか
- Port ごとにモック可能なインターフェースが定義されているか
- Unit Test が外部 SDK ではなく Port を差し替えているか
- 例外実装に期限付き ADR が付与されているか
