# ハイブリッド検索入門

不動産検索を題材に  
**lexical + semantic + rerank** を学ぶ

---

## この Phase の主眼

- 検索を 1 つのモデルではなく、複数段の処理で考える
- `Meilisearch` / `multilingual-e5` / `LightGBM LambdaRank` の役割分担を理解する
- `Redis` を使ったキャッシュの位置づけをつかむ

---

## なぜハイブリッド検索か

| 手法 | 強み | 弱み |
|---|---|---|
| lexical search | 速い、厳密条件に強い | 言い換えに弱い |
| semantic search | 意味の近さに強い | 構造化条件が苦手 |
| rerank | 複数特徴をまとめて最終調整できる | 単独では候補を集めにくい |

---

## 3 段構成

```text
1. Meilisearch で候補取得
2. multilingual-e5 で意味類似度を計算
3. LightGBM LambdaRank で最終順位を決める
```

> この Phase の中核は `LightGBM + multilingual-e5 + Meilisearch`。

---

## Meilisearch を使う理由

- `Elasticsearch` より導入しやすい
- 学習用ローカル構成でも扱いやすい
- 厳密条件の絞り込みを高速に試せる
- BM25 ベースの lexical search を素直に体験しやすい

---

## semantic search の役割

- クエリと物件説明文を embedding に変換する
- 完全一致しない表現も「意味の近さ」で拾う

例:

- `静かな住宅街`
- `閑静なエリア`

> 表記一致ではなく意味近傍で候補を補う。

---

## rerank の役割

`LightGBM LambdaRank` は次のような特徴をまとめて見る。

- `me5_score`
- lexical の順位
- semantic の順位
- 物件属性
- 行動系特徴

候補取得の後に、**より良い順へ並べ替える** のが仕事。

---

## Redis の位置づけ

- 同一条件の検索結果を短時間キャッシュする
- 毎回すべてを再計算しない
- ただし検索品質の本体ではなく、応答性の補助

---

## 処理の流れ

```text
/search
  ↓
条件付き lexical 候補取得
  ↓
query embedding 作成
  ↓
semantic score 計算
  ↓
特徴量作成
  ↓
LambdaRank rerank
```

---

## ディレクトリの見方

| パス | 役割 |
|---|---|
| `app/` | `/search` / `/feedback` API |
| `ml/data/` | embedding 補助、特徴量補助 |
| `ml/training/` | LambdaRank 学習 |
| `pipeline/` | 定常バッチ、学習、評価 |
| `definitions/` | PostgreSQL migration |

---

## この Phase で押さえること

4. `LightGBM LambdaRank` は最終順位の主役
