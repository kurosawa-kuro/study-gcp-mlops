## 1. 問題認識 (なぜ「大失敗」か)

### 1.1 ハイブリッド検索の定義と実装の乖離

「ハイブリッド検索」は **dense (semantic / ベクトル) + sparse (lexical / キーワード BM25)** の 2 系統を融合する方式。参考実装では:

| 系統 | 参考実装のコンポーネント | 役割 |
|---|---|---|
| Sparse / lexical | **Meilisearch** (`src/clients/meilisearch_client.py`) | BM25 + typo tolerance + filterable attributes。`/search` の一次候補抽出 |
| Dense / semantic | **ME5 embeddings** + PostgreSQL `property_embeddings` + `similarity_service.py` | クエリ埋め込み × 物件埋め込みのコサイン類似度 |
| Fusion | `lgbm_trainer.py` + `/search` rerank | 9 次元特徴量 (lexical_rank / me5_score 含む) で LambdaRank |

本リポジトリの `BigQueryCandidateRetriever` (`app/src/app/adapters/candidate_retriever.py`) は **VECTOR_SEARCH 一本 (= dense のみ)** であり、`lexical_rank` という変数名は使っているが**実態は cosine 距離の順位**。9 次元特徴量のうち lexical_rank と me5_score は**同じ信号の別名**になっており、特徴量として独立していない。

### 1.2 キャッシュ層の消失

参考実装の `redis_client.py` は:

- SHA256(normalized params) をキーに `/search` 結果を TTL 120s でキャッシュ
- Redis 接続失敗を warn ログで飲み込み、呼び出し側は常に動く (graceful degradation)
- 負荷試験時の BigQuery スロット消費を抑える load-bearing な層

本リポジトリには**キャッシュ層が一切ない**。毎リクエスト VECTOR_SEARCH が走る。

### 1.3 ドキュメントと実装の整合も破綻

- `docs/01_仕様と設計.md:147-148` と `CLAUDE.md:12` で「**Meilisearch / Redis 不採用**」と宣言
- しかし `README.md:3` と `CLAUDE.md` 先頭は「**不動産ハイブリッド検索**」と呼称
- → lexical 系統を消した結果、名前と実装が矛盾。`01_仕様と設計.md:73-77` の「候補抽出層」の記述も semantic 単系統を「ハイブリッド」と呼んでおり誤り

### 1.4 失敗の根本原因

移植時に「**Meilisearch を BQ VECTOR_SEARCH で差し替え**」と短絡した判断 (旧 `02_移行ロードマップ.md` の差し替え表) が根本原因。正しくは:

- Meilisearch (lexical, BM25) → **BQ `SEARCH()` + `SEARCH INDEX`** など lexical の代替を別途用意
- ME5 vector search → **BQ `VECTOR_SEARCH` + `VECTOR INDEX`** で置換

二つの異なる役割を一つのコンポーネントに潰してしまった。

## 2. 是正の基本方針

### 2.1 ハイブリッド検索の 3-stage を正しく立て直す

```
Stage 1a (lexical):   Meilisearch で title/description/layout の BM25 → top 100 (lex)
Stage 1b (semantic):  BQ VECTOR_SEARCH で ME5 embedding cosine → top 100 (sem)
Stage 2  (fuse):      RRF (Reciprocal Rank Fusion) で 1a + 1b を融合 → 候補 100 件
Stage 3  (rerank):    LightGBM LambdaRank (Phase 6+) で 10 次元特徴量を使った再ランク → top 20
```

- 1a は Meilisearch REST API 経由 (property_id のみ返す)、1b は BQ VECTOR_SEARCH、融合 (2) は `app.services.ranking` 内で Python 実装 (property_id の rank リスト 2 本を RRF で合算し、上位 100 を BQ に戻して features / properties_cleaned と JOIN)
- `Candidate` dataclass に **`lexical_rank`** (= 1a の順位) と **`semantic_rank`** (= 1b の順位) を両方持たせる。現状の `lexical_rank` の意味が変わるので call site を全て直す
- `me5_score` は cosine 類似度 (浮動小数) をそのまま保持 (特徴量として使用)

### 2.2 キャッシュ層を戻す (ただし GCP-native に)

Redis 相当のキャッシュ層を復活させる。ただし **Memorystore for Redis を採用するか、別方式にするか** は選択肢 (§3.2) に分けて User 判断を仰ぐ。

### 2.3 ドキュメント側の決定を巻き戻す

`docs/01_仕様と設計.md` と `CLAUDE.md` と `README.md` の「Meilisearch 不採用 / Redis 不採用」宣言は、**正確には「Meilisearch コンテナと Redis コンテナの持ち込み不採用」であり、lexical 検索とキャッシュ層そのものは採用する**、と表現し直す。連動ドキュメントは同一 PR で直す (`docs/README.md §3`)。

## 3. 選択肢と User 判断待ち事項 (決定ポイント)

実装に入る前に User が決めるべき点を明示する。**これらが決まるまで実装フェーズに進まない**。

### 3.1 Lexical 検索の実装方式

| 選択肢 | 方式 | メリット | デメリット |
|---|---|---|---|
| **A** | BQ `SEARCH()` + `CREATE SEARCH INDEX` on `properties_cleaned(description, title, layout)` | GCP-native 維持 / 追加インフラなし / SQL 一本で VECTOR_SEARCH と同居可能 / IAM も同じ | BM25 ランキングが Meilisearch ほど洗練されていない / typo tolerance は弱い |
| **B** | Meilisearch を Cloud Run sidecar (同一サービス内 multi-container) として復活 | 参考実装と 1:1 / BM25 + typo tolerance が強力 | Cloud Run multi-container は Preview / 永続ボリューム設計必要 / index 同期ジョブが追加で必要 / `Meilisearch 不採用` の決定を巻き戻す |
| **C** | **A + B ハイブリッド**: Phase 5 までは A、本番精度が不足したら B に昇格可能な Port 分離 | 段階的 / 今は軽量 / 後戻り可能 | Port/Adapter 設計コストがわずかに増える |

**決定 (確定版): B** (2026-04-20 User 合意、当初 C の暫定決定から差し替え)。理由: Phase 2 (Local) と Phase 3 (GCP) で**検索スタックを揃える**ことで、親 `README.md` の「Phase 2 → 3 の学習ポイント (Local → Cloud 移植で何が変わるか)」が本来の切り分け (環境の違い / IaC・CI/CD・監視・コスト管理) に集中できる。BQ `SEARCH()` と Meilisearch BM25 のランキング差分を Phase 3 で別途検証する負荷を回避する。

- **Adapter 実装**: `MeilisearchLexical` (B)。Phase R3 で実装
- **非負制約の改訂**: 「BQ + Cloud Run のみ」→ 「BQ + Cloud Run + Meilisearch (on Cloud Run 2nd-gen multi-container、または別 Cloud Run Service)」に緩和。`CLAUDE.md` の非負制約表を Phase R1 で書き換える
- **Port 分離は維持**: `LexicalSearchPort` は残す。将来 BQ `SEARCH()` に差し替える選択肢は Port レベルで開けておく (A は future option)

### 3.2 キャッシュ層の実装方式

| 選択肢 | 方式 | メリット | デメリット |
|---|---|---|---|
| **a** | **Memorystore for Redis (Standard, 1GB)** | 参考実装とコード互換 / 永続 / Cloud Run から VPC connector 経由で接続 | 月 ~$35 / VPC connector も必要 ($10) / Redis 不採用の決定を巻き戻す |
| **b** | **Cloud Run Service の in-memory TTL キャッシュ (`cachetools.TTLCache`)** | 追加インフラゼロ / `min-instances=1` なら warm / 「同一クエリが短時間に繰り返される」パターンで実用ヒット率 | instance 間でキャッシュ共有されない / 再デプロイで消える |
| **c** | **BigQuery の query cache に任せる** (=何もしない) | ゼロ / BQ は同一クエリ 24h キャッシュする | クエリ埋め込み (FLOAT64 ARRAY 768d) が毎回微妙に違うと効かない / 実質的に効かない |

**決定: b** (2026-04-20 User 合意)。素の `functools.lru_cache` は TTL 非対応のため **`cachetools.TTLCache`** を採用 (`pyproject.toml` の `app` workspace に追加)。検索クエリは「同一クエリが短時間に繰り返される」パターンで効き、`min-instances=1` + 単一インスタンス運用下で実用ヒット率が出る前提。`CacheStore` Port で抽象化するため、将来トラフィックに応じて a (Memorystore) へ昇格可能。

### 3.3 RRF (Reciprocal Rank Fusion) の k パラメータと融合位置

- `k = 60` が文献デフォルト (Cormack et al.)。本リポジトリは `k = 60` を開始点とし、`app/src/app/services/ranking.py` に定数化
- 融合は **Cloud Run Service 内の Python で実行**。理由: lexical (Meilisearch) と semantic (BQ) が別システムになったため、どちらかに寄せると残りをそのシステムに引き込む必要があり現実的でない。融合対象は `property_id` の rank リスト 2 本だけ (各 top 200 = 計 400 行) なので Python 処理で十分軽い
- 融合後の top 100 を BQ に IN 句で戻して `properties_cleaned` / `property_features_daily` を JOIN 取得し、`Candidate` dataclass を組み立てる

### 3.4 特徴量の再定義 (9 次元 → 10 次元)

現状 9 次元 (`FEATURE_COLS_RANKER`) のうち `lexical_rank` は semantic rank の別名になっていた。正しい 10 次元に拡張する:

| # | カラム | 出所 | 訓練 / 推論 |
|---|---|---|---|
| 1 | `rent` | properties_cleaned | 両方 |
| 2 | `walk_min` | properties_cleaned | 両方 |
| 3 | `age_years` | properties_cleaned | 両方 |
| 4 | `area_m2` | properties_cleaned | 両方 |
| 5 | `ctr` | property_features_daily | 両方 |
| 6 | `fav_rate` | property_features_daily | 両方 |
| 7 | `inquiry_rate` | property_features_daily | 両方 |
| 8 | `me5_score` | クエリ時 (VECTOR_SEARCH cosine) | 推論 / 訓練はログ |
| 9 | `lexical_rank` | **クエリ時 (SEARCH() の BM25 順位)** — 再定義 | 推論 / 訓練はログ |
| 10 | `semantic_rank` | クエリ時 (VECTOR_SEARCH の順位) | 推論 / 訓練はログ |

訓練データは `mlops.ranking_log` に `lexical_rank` / `semantic_rank` / `me5_score` を全部残すため、log record のスキーマ拡張が必須 (§4.2 で詳述)。`lexical_rank` の「出所」が Meilisearch BM25 スコアの順位であることを Phase R3 commit で明記する。

### 3.5 Meilisearch / Redis 「不採用」宣言の撤回

現行ドキュメント (`01_仕様と設計.md:147-148` / `CLAUDE.md` / `README.md`) の「Meilisearch 不採用」「Redis 不採用」は **設計ミス**。過去の意図を汲んで言い換えるようなものではなく、誤った決定として撤回する。

- 誤り 1: lexical 検索を実装しないまま「ハイブリッド検索」と銘打っていた
- 誤り 2: キャッシュ層を最初から放棄し、BQ スロット消費を無対策にしていた
- 撤回理由: ハイブリッド検索の定義 (§1.1) と参考実装の設計意図 (§7) に照らして、lexical stage とキャッシュ層は両方とも load-bearing

修正後の宣言:

- ✅ **採用**: lexical 検索 (**Meilisearch v1.7+ を Cloud Run に乗せる**、Port 経由で将来 BQ `SEARCH()` に差し替え可) / オンラインキャッシュ (`cachetools.TTLCache`)
- ❌ **不採用**: Redis サーバ (コンテナ or Memorystore、将来必要なら §3.2 a に昇格) / PostgreSQL / Vertex AI / BQML
- ⚠️ **条件付き採用**: Meilisearch は「サーバコンテナとして GCP に載せる」。参考実装の `getmeili/meilisearch:v1.7` image を Artifact Registry に mirror し、Cloud Run Service としてデプロイする (sidecar vs 独立 Service の選択は Phase R3 実装時に §3.6 で決定)

本節は「表現の修正」ではなく「決定の巻き戻し」。Phase R1 のドキュメント整合 PR で `01 / CLAUDE / README` から「Meilisearch 不採用 / Redis 不採用」の単独列挙を削除し、本書 §3.5 の決定文で置き換える。

### 3.6 Meilisearch の GCP 実装形態

**決定: α (独立 Cloud Run Service + GCS FUSE mount)** (2026-04-20 User 合意、確定)。

判断根拠: 本リポジトリは**学習検証用であり実務運用ではない** (親 `README.md` Phase 3 は「不動産ハイブリッド検索(実務)」と題しているが、実態は Cloud Native 化の学習題材)。したがって α のデメリット (cold start で GCS FUSE から index ロード / マルチインスタンス時の一貫性課題) は許容範囲。β (sidecar) / γ (GCE VM) の複雑さは学習目的に対して過剰。

比較表 (参考):

| 方式 | 内容 | メリット | デメリット |
|---|---|---|---|
| **α (採用)** | **独立 Cloud Run Service** (`meili-search`)。search-api から IAM 認証付き HTTP で呼ぶ | シンプル / IaC 差分が小さい / Cloud Run 料金のみ / Meilisearch が落ちても search-api は動く (noop fallback) | 永続ストレージは **GCS FUSE mount** (index を `gs://mlops-dev-a-meili-data/` に置く) / cold start で index ロードが走る / 複数インスタンスでの GCS 書込競合は `meili-sync-job` 1 本に書込を限定して回避 |
| β | search-api と同一 Cloud Run Service の sidecar container | ネットワーク遅延ゼロ | sidecar の ライフサイクル管理 / search-api との blast radius 結合 |
| γ | GCE VM + Meilisearch + 永続ディスク | 永続化が自明 | 「Cloud Run only」から逸脱 / VM パッチ管理 |

**具体構成** (Phase R3 実装時の前提):

- Cloud Run Service 名: `meili-search`
- イメージ: Artifact Registry に mirror した `getmeili/meilisearch:v1.7`
- Cloud Run 設定: `--cpu 1 --memory 2Gi --min-instances 1 --max-instances 2 --no-allow-unauthenticated --execution-environment gen2`
- GCS FUSE mount: `gs://mlops-dev-a-meili-data` → `/meili_data` (環境変数 `MEILI_DB_PATH=/meili_data`)
- SA: `sa-meili` (新設、`storage.objectAdmin` on `mlops-dev-a-meili-data` のみ)
- search-api → meili-search の認証: `sa-api` に `roles/run.invoker` on `meili-search` を付与、search-api 側は `google.auth.transport.requests` で ID token を取得して Authorization header に載せる

Meilisearch index 同期は **Cloud Run Jobs `meili-sync-job`** を新設 (参考実装の `src/jobs/sync_properties.py` 相当を GCP 版に移植)。Dataform 完走後の Cloud Scheduler trigger で日次走らせる (04:30 JST 想定)。

   ```python
       lex_results = self._lexical.search(query=query_text, filters=filters, top_k=200)
       # Stage 1b: BQ VECTOR_SEARCH (filter 込み top 200)
       sem_results = self._vector_search(query_vector, filters, top_k=200)
       # Stage 2: RRF fusion
       fused_ids = rrf_fuse(lex_results, sem_results, k=60, top_n=100)
       # Stage 2.5: BQ に戻して features / properties 属性 JOIN 取得
       return self._enrich_from_bq(fused_ids, lex_results, sem_results)
   ```
5. **Retriever の DI 変更**: `BigQueryCandidateRetriever.__init__` に `lexical: LexicalSearchPort` を追加。composition root (`app/src/app/entrypoints/api.py` の lifespan) で `MeilisearchLexical` を注入
6. **推論側 Python** (`common/src/common/feature_engineering.py::build_ranker_features`) に `semantic_rank` を追加し、9 列 → 10 列で特徴量ベクトルを組む
7. **Pub/Sub payload** (`app/src/app/adapters/candidate_retriever.py:176-186`) の `features` RECORD に `semantic_rank` を追加
8. **Infra schema** (`infra/terraform/modules/data/main.tf` の `ranking_log.features` RECORD) に `semantic_rank FLOAT64 NULLABLE` を追加

**受け入れ基準**: `make check` PASS + `tests/parity/test_feature_parity_ranking.py` が 10 列で PASS + `make ops-search` で実環境に `meili-search` 経由の BM25 + BQ VECTOR_SEARCH の RRF 融合結果が返る + `meili-sync-job` の 1 回実行後に Meilisearch index に properties_cleaned 行数と同じドキュメント数が入る。

### Phase R4 — キャッシュ層の配線 (§3.2 で b を選んだ場合)

| Meilisearch の cold start で GCS FUSE からの index ロードが遅い | `min-instances=1` を初期設定。index サイズが MB オーダなら許容範囲。問題が出たら §3.6 β (sidecar) に切り替え |
