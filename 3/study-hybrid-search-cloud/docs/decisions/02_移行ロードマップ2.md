# フルリファクタリング実施計画 (Phase 3 / `study-hybrid-search-cloud`)

`jobs/` に詰め込まれている training / embedding / meili_sync を、Cloud Run Job 単位で独立した `ml/` 配下 workspace に分離する計画。本ロードマップの位置付けは `02_移行ロードマップ.md` の決定的仕様 (bq-first / Meilisearch + BigQuery `VECTOR_SEARCH` + RRF + LambdaRank) に **従属** する整理であり、スコープを変えるものではない。

---

## 0. 背景 (なぜ動かすか)

現状 `jobs/` workspace は 3 つの別コンポーネントを同居させている:

| コンポーネント | 実体 | デプロイ先 |
|---|---|---|
| training (rank_cli / rank_trainer / ranking_metrics / ports / adapters) | LightGBM LambdaRank 学習 | Cloud Run Job `training-job` |
| embedding (embed_cli / embedding_runner / embedding_writer) | multilingual-e5 バッチ推論 → `feature_mart.property_embeddings` 書き込み (`common.embeddings.E5Encoder` は app と共有するため撤去しない) | Cloud Run Job `embedding-job` |
| meili_sync (sync_cli) | `feature_mart.properties_cleaned` → Meilisearch index `properties` upsert | (未デプロイ、CLI 手動実行) |

このまま肥大化すると以下が崩れる:

1. **CI path filter が曖昧**: `deploy-training-job.yml` は `jobs/**` 全部発火、`deploy-embedding-job.yml` は `jobs/src/training/entrypoints/embed_cli.py` など個別列挙 — embed と train の依存境界が CI 側にしか表現されていない
2. **Docker image が 1 つ**: `jobs/Dockerfile` 1 本から training-job / embedding-job の両方を焼いている (sentence-transformers ~1.1GB + LightGBM + pandas を 1 image に同居)。embed-only 変更で training image が、train-only 変更で embedding image が意味なく再ビルドされる
3. **`common/embeddings/` は現状維持**: `common/src/common/embeddings/e5_encoder.py` は `training.services.embedding_runner` と **`app.entrypoints.api.create_app.lifespan`** (`app/src/app/entrypoints/api.py:188` で `settings.enable_search=True` のとき query encoding 用に load) の双方から使われている。app と ml/embed で共有される正真正銘の共有コードなので `common/` に残す (当初ロードマップ案で撤去予定としていたが、app 側利用の見落としが判明したため方針変更)
4. **AST 境界テスト (`scripts/checks/layers.py::RULES`) が 1 つの `training.*` 名前空間に依存**: 将来 embed と train の port / adapter を片方だけ差し替えたいときに、レイヤ毎の ban を書き分けられない

**やらないこと** (非スコープ):

- bq-first の設計テーゼそのもの (`02_移行ロードマップ.md §0`) は変えない — BigQuery `VECTOR_SEARCH` / RRF / LambdaRank / モデル GCS 配置 / `mlops.training_runs` 系譜 / 5 SA 分離 / WIF は維持
- Vertex AI / Vertex AI Pipelines / Vertex Endpoints は **引き続き採用しない** (Phase 4 への移行は別リポジトリ `4/study-hybrid-search-vertex/` 側で行う)
- Cloud Run Service `search-api` の責務 (lifespan で Booster ロード、Phase 6 以降) は今回触らない
- feature parity invariant (5 ファイル) を崩さない — `common/feature_engineering.py::build_ranker_features` は移動しない

### 3.4 `ml/*/container/Dockerfile` 設計

現行 `jobs/Dockerfile` の multi-stage (uv builder → python slim runtime + libgomp1) をそのまま踏襲。2 点だけ変える:

1. `uv sync --frozen --no-dev --package jobs --no-install-project` → `uv sync --frozen --no-dev --package ml-embed --no-install-project` (train 側は `--package ml-train`)
2. `COPY jobs/ jobs/` → `COPY ml/embed/ ml/embed/` (train 側は `COPY ml/train/ ml/train/`)
3. `ENTRYPOINT ["train"]` → embed 側は `ENTRYPOINT ["embed"]`、train 側は `ENTRYPOINT ["rank-train"]`
4. `common/` の COPY は両 Dockerfile に残す (両 workspace とも `common` に依存)

これにより:

- embed image は `common` 経由で sentence-transformers + torch (~1.5GB) を含むが LightGBM / wandb は含まない
- train image は `common` 経由で LightGBM / wandb は入るが (後述)、sentence-transformers / torch は依存グラフ的に `common.embeddings.e5_encoder` 経由でしか要求されないため、train 側で明示的に `lightgbm` `wandb` `pandas` を `ml/train/pyproject.toml::dependencies` に含める一方、`sentence-transformers` を transitive に引かないような切り分けは常に成立するわけではない — `common` が `sentence-transformers` を declared dep にしている現状では train image にも引き込まれる。image サイズ差を最大化したい場合は PR #6 以降で `common` を `common-core` + `common-embeddings` の 2 workspace に分割する選択肢があるが、本リファクタの射程外
- embed-only 変更で train image が、train-only 変更で embed image が再ビルドされなくなる (CI path filter の効果)
