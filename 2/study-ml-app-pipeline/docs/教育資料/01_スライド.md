# App + Pipeline 入門

同じ California Housing を題材に  
**API と設計パターン** を学ぶ

---

## この Phase の主眼

- `FastAPI` で推論 API を持つ
- job を分離して pipeline の入口を整理する
- `core -> ports <- adapters` の依存方向を理解する
- `lifespan` と DI で初期化を管理する

---

## Phase 1 との違い

| Phase 1 | Phase 2 |
|---|---|
| ML コア中心 | App / Pipeline / DI 追加 |
| 学習と評価が主役 | 推論 API も主役 |
| 直接呼び出しが多い | Port/Adapter で境界を切る |
| 成果物を作る | 成果物を使って予測する |

---

## 全体像

```text
HTTP リクエスト / job 起動
        ↓
Inbound Adapter
        ↓
ml/core
        ↓
ports
        ↓
adapters
        ↓
PostgreSQL / filesystem
```

---

## Port/Adapter の考え方

### Port
- 「何ができればよいか」という抽象

### Adapter
- その抽象を PostgreSQL や filesystem で実装した具体

### ねらい
- core が外部実装に引っ張られない
- Phase 3 以降で adapter を差し替えやすくする

---

## 代表的な Port

| Port | 役割 |
|---|---|
| `DatasetReader` | 学習データを読む |
| `ModelStore` | モデルを保存・読込する |
| `Predictor` | 推論を実行する |

見るべき点:

- core は Port しか見ない
- adapter が PostgreSQL や filesystem に触る

---

## FastAPI の位置づけ

- `/predict` で推論を受ける
- Web UI と REST API の入口になる
- `lifespan` で model / adapter / service を初期化する

```python
app.state.container = build_container()
```

> グローバル状態ではなく、起動時に依存を組み立てる。

---

## job 分離

| job | 役割 |
|---|---|
| `data_job` | データ投入 |
| `training_job` | 学習 |
| `batch_serving_job` | バッチ推論 |

1 つの巨大スクリプトにせず、  
**目的ごとに入口を分ける** のがこの Phase の重要点。

---

## ディレクトリの見方

| パス | 役割 |
|---|---|
| `app/` | FastAPI と API 入力 |
| `ml/core/` | 純粋ロジック |
| `ml/ports/` | 抽象定義 |
| `ml/adapters/` | 外部 I/O 実装 |
| `pipeline/` | job の入口 |

---

## 学習者が見るべきポイント

1. ML の式そのものより、責務の分かれ方
2. API と pipeline がどちらも inbound adapter であること
3. DB や model 保存先を変えても core を壊しにくいこと

---

## この Phase で押さえること

- `FastAPI` は推論の入口
- `lifespan` は初期化の入口
- `Port/Adapter` は依存方向を守るための骨格
- 次の Phase では adapter が `Meilisearch` / `Redis` / `multilingual-e5` に広がる
