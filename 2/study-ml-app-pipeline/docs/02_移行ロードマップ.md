# 02_移行ロードマップ

本ドキュメントは Phase 2 (`study-ml-app-pipeline`) の決定的仕様を管理する。Phase 2 は **Phase 3-7 の完成形から検索ドメイン、クラウド基盤、Vertex 標準 MLOps、PMLE 追加技術、GKE serving 差分を引いた、App / Pipeline / Port-Adapter 導入フェーズ** とする。

目的は、Phase 1 の California Housing 回帰を土台に、**FastAPI / job 分離 / DI / Port-Adapter** を導入し、実装の依存方向を整理すること。

## 0. 不変ルール

- 題材は California Housing 回帰のまま維持する
- ML コアは LightGBM 回帰を維持する
- 変えるのは実行入口、依存方向、DI の責務分担
- 検索ドメインやクラウド責務は持ち込まない

## 1. 学習主眼

Phase 2 で新しく吸収させるのは次のアプリケーション構成要素。

| 技術 | 学ぶ観点 |
|---|---|
| FastAPI | API 入口 |
| lifespan DI | 依存配線 |
| Port-Adapter | 依存方向の制御 |
| job 分離 | API と学習処理の分離 |
| container | adapter 配線 |

## 2. 対象外

Phase 2 では以下を扱わない。

- Meilisearch
- multilingual-e5
- Redis
- LightGBM LambdaRank
- 不動産検索ドメイン
- Cloud Run / BigQuery / GCS / Pub/Sub
- Terraform / WIF / Secret Manager
- Vertex AI Pipelines / Endpoints / Model Registry / Monitoring
- BQML / Dataflow / RAG / Explainable AI / SLO / GKE / KServe

## 3. 実装方針

- Phase 1 の ML コアは維持し、呼び出し方だけ整理する
- API と training job を分ける
- Port と adapter を切り、外部依存を閉じ込める
- DI は container に集約する

## 4. 修正方針

### 4.1 コード

- California Housing を前提とした説明に揃える
- 検索ドメインやクラウド前提の責務は入れない

### 4.2 文書

- README / CLAUDE / `docs/01_仕様と設計.md` は App / Pipeline / Port-Adapter を主題に統一する
- Phase 3 以降の技術は「次 Phase で追加するもの」としてのみ触れる
- 教育資料は FastAPI / job 分離 / Port-Adapter / DI を中心に整理する

### 4.3 検証方針

- 移行中は差分を意識しつつ、コード修正の速度を優先する
- E2E や CI/CD 上の総合検証は後段へ回す
- ローカルで回せる単体テスト、限定的な lint、局所的な確認を優先する
