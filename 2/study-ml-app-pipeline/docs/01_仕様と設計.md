## 0. 位置付け (Phase 2: App / Pipeline / Port-Adapter 導入)

Phase 2 は **コード化対象 Phase**。題材は Phase 1 から継承して California Housing (回帰) を維持しつつ、**App / Pipeline / Port-Adapter / DI** の設計パターンを学ぶフェーズ。検索ドメインや GCP 系の責務は持ち込まない (Phase 3 以降の責務)。

---

## 1. Phase 1 → Phase 2 の差分

| 観点 | Phase 1 | Phase 2 (追加) |
|---|---|---|
| 題材 | California Housing 回帰 | 同一 (不変) |
| 学習器 | LightGBM 回帰 | 同一 (不変) |
| 評価 | RMSE / R² | 同一 (不変) |
| API | なし | **`/predict` / `/health` (FastAPI)** |
| DI | なし | **lifespan DI** (`app/`, `ml/container.py`) |
| 設計パターン | flat module | **Port / Adapter** (`ml/ports/`, `ml/adapters/`) |
| job 構造 | data_job / training_job / evaluation_job | **+ batch_serving_job** (推論系) |
| データ層 | PostgreSQL 直接 | Filesystem / PostgreSQL adapter (Port 経由) |

**変えるのは**: API 入口・job 入口・依存方向・DI。
**変えないのは**: 題材と ML コア (LightGBM 回帰)。

---

## 2. 中核アーキテクチャ (`core → ports ← adapters`)

```
Inbound Adapter (FastAPI / job runner)
       ↓
   ml/core (純粋ロジック)
       ↓
   ml/ports (抽象 Protocol)
       ↑
   ml/adapters (PostgreSQL / Filesystem 実装)
       ↑
   ml/container.py (依存配線)
```

依存方向は **常に internal → external** (core は ports のみ知り、adapter は知らない)。この依存方向は **Phase 3 以降も不変**で、ハイブリッド検索ドメインに展開しても adapter だけ差し替えていく。

## 3. 主要コンポーネント

### 3.1 `app/` (FastAPI)

- `/predict` — 予測エンドポイント (Predictor 経由)
- `/health` — liveness
- `lifespan` — 起動時に Container を構築、DI 注入

### 3.2 `ml/core` (純粋ロジック)

- preprocess / feature engineering / trainer / evaluation

### 3.3 `ml/ports` / `ml/adapters` (抽象 ↔ 実装)

| Port | Adapter (production) |
|---|---|
| `DatasetReader` | `PostgresDatasetReader` |
| `ModelStore` | `FilesystemModelStore` |
| `Predictor` | `LightGBMPredictor` |

### 3.4 `pipeline/` (job 分離)

- `data_job` — sklearn → PostgreSQL 投入
- `training_job` — LightGBM 学習
- `batch_serving_job` — バッチ推論

### 3.5 `ml/container.py` (DI)

- 起動時に Adapter / Service を組み立て、`Container` (immutable) として返す
- handler は `Depends` で受け取る (`request.app.state.xxx` の `getattr` は使わない)

## 4. Phase 2 で扱わないもの (Phase 3 以降 / Phase 7 完成版の責務)

- ハイブリッド検索 (Meilisearch / multilingual-e5 / RRF / LightGBM LambdaRank / Redis) → **Phase 3** で導入 (コード化)
- 不動産検索ドメイン → **Phase 3** で導入 (コード化)
- GCP マネージドサービス (Cloud Run / Cloud Run Jobs / BigQuery / GCS / Secret Manager / Terraform / WIF / 軽量 retrain trigger) → **Phase 4** で導入 (コード化、Phase 3 からの足し算)
- Vertex AI 本番 MLOps 基盤 (Pipelines / Endpoint / Model Registry / Monitoring / **Feature Store** / **Vector Search**) → **Phase 5 (論理 Phase)** で資料化 (実装は Phase 7 完成版コード)
- Cloud Composer / PMLE 追加技術 (BQML / Dataflow / TreeSHAP / SLO + burn-rate) → **Phase 6 (論理 Phase)** で資料化 (実装は Phase 7 完成版コード)
- GKE / KServe → **Phase 7 (canonical = 教材コード完成版)** で本実装
- 全 Phase 教材対象外 (Agent Builder / Vizier / Model Garden / Gemini RAG / W&B / Looker Studio / Doppler) — 親 [`README.md` §1](../../../README.md)

---

## 5. 不変ルール

- 題材は California Housing 回帰 (Phase 1 から継承)
- ML コアは LightGBM 回帰
- Phase 2 は App / Pipeline / Port-Adapter を学ぶフェーズ — 検索ドメインや GCP 責務は持ち込まない
- 設計思想 (Port/Adapter / `core → ports ← adapters` / DI) は **Phase 3 以降も不変**で、本 phase が起点
