# 02_移行ロードマップ切り替え基盤

## 目的

Phase 7 では `scripts/ops/*` が

- local での検証
- GCP 上の実環境検証
- 明示 URL を使った一時的な検証

を安全に切り替えられる必要がある。

過去は各 script が個別に

- `cloud_run_url()`
- `identity_token()`
- `LOCAL_API_URL`

を直接扱っており、Phase 7 の GKE/Gateway 化後も Cloud Run `search-api` 前提が残り、`ops-livez` などが stale になった。

このズレを減らすため、**API 切替の最小責務を `scripts/_common.py` に集約する**。

---

## 基盤の責務

切替基盤の責務は次の 3 つに限定する。

1. どの API URL を使うか決める
2. Bearer token が必要か決める
3. script 側へ「解決済みの結果」だけ返す

逆に、次は責務に含めない。

- Gateway の readiness 判定
- Cloud Run / GKE の自動判定
- fallback 的な複雑な分岐
- endpoint ごとの payload 差分

切替基盤は **環境を推測しない**。必要な入力は env で明示する。

## canonical 実装

実装箇所:

- [scripts/_common.py](/home/ubuntu/repos/study-gcp-mlops/7/study-hybrid-search-gke/scripts/_common.py)

公開 API:

- `ResolvedApiTarget`
- `resolve_api_target()`

### `ResolvedApiTarget`

`ops` script が使う解決結果。

```python
@dataclass(frozen=True)
class ResolvedApiTarget:
    url: str
    token: str | None
    mode: str
```

意味:

- `url`
  - 実際に叩く base URL
- `token`
  - `Authorization: Bearer ...` に入れる token
  - 不要な場合は `None`
- `mode`
  - `explicit` / `local` / `gcp`

script 側は **この結果だけ**を使う。

---

## 解決順序

`resolve_api_target()` の優先順位は固定する。

1. `API_URL` があれば最優先
2. それ以外は `TARGET`
3. `TARGET` 未指定時は `gcp`

### 1. `API_URL` 優先

`API_URL` が与えられた場合は、

- `TARGET` を無視する
- `API_URL` をそのまま使う
- `API_REQUIRE_TOKEN=true` のときだけ token を発行する

用途:

- port-forward
- 一時 Gateway URL
- 手元の mock / proxy
- stale script 修正後の局所検証

### 2. `TARGET=local`

`LOCAL_API_URL` を使い、token は発行しない。

既定値:

```text
http://127.0.0.1:8080
```

### 3. `TARGET=gcp`

`cloud_run_url()` で URL を解決し、`identity_token()` を発行する。

これは **既存互換のための既定値** であり、Phase 7 では今後

- Gateway URL resolver
- `API_URL` 明示

へ段階的に寄せる。

## 利用する環境変数

### `TARGET`

許可値:

- `local`
- `gcp`

未指定時は `gcp`。

### `LOCAL_API_URL`

`TARGET=local` のときに使う URL。

例:

```bash
TARGET=local LOCAL_API_URL=http://127.0.0.1:18080 make ops-search
```

### `API_URL`

最優先の明示 URL。

例:

```bash
API_URL=https://search.example.com API_REQUIRE_TOKEN=true make ops-livez
```

### `API_REQUIRE_TOKEN`

`API_URL` 使用時に Bearer token を付けるかどうか。

truthy:

- `1`
- `true`
- `yes`
- `on`

## 設計上の注意

### 1. 切替基盤は小さく保つ

ここに

- Gateway 自動探索
- GKE / Cloud Run 自動判定
- retry
- healthcheck

まで入れると責務が膨らむ。

切替基盤は「URL と token を返す」だけに留める。

### 2. `gcp` 既定は将来の見直し候補

現在の `gcp` は `cloud_run_url()` を使う。
これは既存 script 互換を優先した暫定 canonical であり、Phase 7 の実態である GKE Gateway とは一致しない。

したがって、Phase 7 完了条件としては次のどちらかが必要。

1. `cloud_run_url()` 相当の GKE/Gateway resolver を作る
2. ops 実行時は `API_URL` 明示を canonical にする

### 3. script 側で再分岐しない

悪い例:

```python
if target == "local":
    ...
elif target == "gcp":
    ...
```

良い例:

```python
resolved = resolve_api_target()
status, body = http_json("GET", f"{resolved.url}/livez", token=resolved.token)
```

切替ロジックを複数箇所へ戻さない。

---

## 今回の移行理由

Phase 7 実検証で次が発覚した。

- `ops-livez` が Cloud Run `search-api` を見に行って失敗
- port-forward では `/livez` `/search` `/feedback` が通る
- つまり API 本体より、ops script 側の入口解決が stale

このため、まずは

- script ごとの入口解決の重複を除去
- local / explicit URL での再利用を可能にする

ことを優先した。
