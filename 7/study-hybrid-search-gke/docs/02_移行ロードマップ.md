# Phase 7 リファクタリング候補リスト

ドキュメントを精読した結果、**フォルダ構成・ファイル配置・Port/Adapter 設計を維持したまま**改善できる箇所を抽出します。

---

## 1. 命名・配置の一貫性

### 1.1 `protocols/` vs `ports/` の混在
- **現状**: `app/services/protocols/` (Port 群) と `ml/training/ports/` / `ml/registry/ports/` / `ml/serving/ports/` / `ml/streaming/ports/` / `pipeline/training_job/ports/` が併存
- **問題**: 同じ概念 (Port = Protocol) に `protocols/` と `ports/` の 2 名称
- **対応**: `docs/フォルダ-ファイル.md` 準拠の `ports/` に統一するか、`docs/01_仕様と設計.md §4` で「`app/` 層は `protocols/`、`ml/`/`pipeline/` 層は `ports/`」と明示的に意図を分離 (現状はどちらか不明瞭)

### 1.2 Fakes の置き場所が二分
- **現状**: `tests/fakes/` (test 用) と `app/services/fakes/` (production noop)
- **問題**: 「production noop」と「test fake」が同じ語彙で説明されており混乱を招く
- **対応**: `app/services/fakes/` → `app/services/null_adapters/` または `app/services/noop_adapters/` にリネーム (production の null object pattern であることを名称で示す)

### 1.3 `_kserve_common.py` / `_pubsub_diagnostics.py` の prefix
- **現状**: `app/services/adapters/` 内に `_` prefix の helper module が同居
- **問題**: 「1 Adapter 1 file」原則に対する例外が adapter dir 内に混入
- **対応**: `app/services/adapters/_helpers/` または `app/services/adapters/internal/` サブディレクトリに分離

## 2. 設定の二重管理

### 2.1 `vertex_*_endpoint_display_name` の死蔵
- **現状**: `setting.yaml` に `encoder_endpoint_display_name` / `reranker_endpoint_display_name` が「Phase 7 では実質未使用、将来復活用に残す」と明記
- **問題**: 死んだ設定値が source of truth に存在 (新規参加者の混乱要因)
- **対応**: コメントで `# Vertex Endpoint shell 用 (Phase 7 inactive)` と明示するか、`setting.yaml` から削除して `docs/decisions/` に移送

### 2.2 `training_job: training-job` の互換残置
- **現状**: 旧 Cloud Run Job 名を「互換目的で残置」
- **対応**: 同上 — どこから参照されているか grep で確認、参照ゼロなら削除

### 2.3 `ApiSettings` の肥大化
- **現状**: `enable_search` / `enable_rerank` / 4 種の topic / KServe 3 URL / BQML / RAG / Gemini 設定が単一クラスに同居
- **対応**: `ApiSettings` を `FeatureFlags` / `KServeSettings` / `MessagingSettings` / `RagSettings` の nested model に分割 (pydantic-settings の `model_config` で env prefix を維持)

## 3. Terraform モジュール

### 3.1 `vertex/` モジュールの Endpoint shell
- **現状**: `google_vertex_ai_endpoint.encoder/reranker` が「shell のみ残置」、serving は KServe
- **問題**: 死リソースが state を汚す
- **対応**: `enable_vertex_endpoint_shell` フラグで `count = var.enable_vertex_endpoint_shell ? 1 : 0` 制御し既定 `false`、復活時のみ `true`

### 3.2 `runtime/` モジュールの責務縮小
- **現状**: Cloud Run Service が `meili-search` 1 本に縮小されたが、`meili-search` は別途 `meilisearch/` モジュールに存在 — `runtime/` は実質 Pub/Sub + Scheduler のみ
- **対応**: `runtime/` → `messaging/` にリネーム (Pub/Sub topic + BQ Subscription + Cloud Scheduler の責務を名称で示す)

### 3.3 `monitoring/` の `time_sleep.wait_for_log_metric_indexing`
- **現状**: 90s の `time_sleep` リソースが apply のたびに走る
- **対応**: `triggers = { metrics = sha256(...) }` で metric 定義変更時のみ再 sleep するよう絞る (apply 時間短縮)

## 4. 命名統一・ドキュメント整合

### 4.1 `app/api/routes/` の placeholder
- **現状**: `__init__.py` のみの空ディレクトリが「将来の prefix 別 router 用」として残置
- **対応**: 削除 (必要時に追加すれば良い、空 dir は誤解の元)

### 4.2 `model_monitoring_alerts` テーブルの inactive 状態
- **現状**: BQ テーブルが残っているが Phase 7 では書き込み元がない
- **対応**: `deletion_protection=true` のまま据え置きは可だが、`docs/03_実装カタログ.md` の `Phase 7 では Vertex Endpoint 撤去で attach 先が無いため、実質 inactive` という記述を、テーブル定義側の `description` field にも書く (BQ コンソールから即座に判別可能に)

### 4.3 `docs/02_移行ロードマップ.md` が空
- **現状**: 本体が空でサブドキュメント 2 本が「暫定の決定仕様」
- **対応**: `02_移行ロードマップ.md` を index 化してサブドキュメントへの導線を明示、または 1 本に統合

## 5. テスト構成

### 5.1 `tests/integration/parity/` の lockstep test 群
- **現状**: 6 ファイル lockstep + Dataform drift で計 7 ファイル
- **対応**: `parity_invariant.py` のような共通基盤を抽出 — 「Python 側の正本 (`FEATURE_COLS_RANKER`) を読み込み、対象ファイルを与えると検証する」共通関数化

### 5.2 `tests/e2e/` が `__init__.py` のみ
- **対応**: `tests/e2e/README.md` に「現状 spec のみ、`make run-all-core` がカバー」と明記、もしくは `make run-all-core` の subset を pytest に取り込み

## 6. Scripts の冪等性・観測性

### 6.1 `scripts/_common.py` の hand-parser
- **現状**: PyYAML 不要のため stdlib hand-parser で `setting.yaml` を読む
- **問題**: nested 値や list に拡張する際の負債 (現状 flat key:value 制約)
- **対応**: 制約を `setting.yaml` 冒頭にコメントで明記 (`# Flat key:value only — see scripts/_common.py::_load_settings()`)

### 6.2 `scripts/setup/deploy_all.py` の責務集中
- **現状**: tf-bootstrap → tf-init → WIF recovery → sync-dataform → tf-plan → apply → `api_gke` を 1 ファイルで実行
- **対応**: 各 step を関数として分離し、`--from-step` / `--to-step` オプションで部分実行可能に (現状でも長期運用で部分再実行需要は確実に出る)

---

## 7. Phase 7 固有の追加点

### 7.1 `infra/manifests/` の `kubectl apply -k` が手動
- **現状**: deploy_all.py は manifest を kubectl で apply しない (試行錯誤を妨げないため)
- **対応**: `make apply-manifests` target を新設して手順を明示化 (手動運用を残しつつ「正規手順」を Makefile に置く)

### 7.2 `kserve/networkpolicy.yaml` と `search-api/networkpolicy.yaml` の対称性
- **現状**: 両 ns で別ファイル管理
- **対応**: `infra/manifests/policies/` 等にまとめて見通し向上 (kustomization.yaml の `resources:` 列挙のみ修正)
