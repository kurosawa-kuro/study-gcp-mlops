## B-3. `_optional_adapter.py` の prefix と配置

Run 3 で `app/container/_optional_adapter.py` が新設されたが、`_` prefix の helper が `container/` 直下に置かれている。`adapters/internal/` と同じ問題が `container/` でも発生する芽。

**対応**: 件数が増える前に `container/_helpers/optional_adapter.py` のような sub-dir を切るか、あるいは `composition_root.py` の private helper として吸収するか方針決定。現状 1 ファイルなので即時対応不要だが、ガイドラインを `docs/01_仕様と設計.md §4` に追記しておくと将来の拡散を防げる。

## B-4. テスト dir の `fakes/` 二重存在

ドキュメント上 `tests/fakes/` と `tests/unit/` が並列に置かれているが、`tests/fakes/` は `unit/` から import される共通モジュールなので階層が flat すぎる。

**対応**: `tests/_fakes/` (test 用 helper であることを `_` で示す) または `tests/unit/_fakes/` に移送。`tests/integration/` から `fakes/` を import するケースは現状なさそう (整理が必要)。

## B-5. `monitoring/` ルート dir の役割不明瞭

ルート直下に `monitoring/` (feature skew SQL) が単独で存在。一方で:
- `infra/terraform/modules/monitoring/` (Terraform 監視)
- `pipeline/data_job/dataform/monitoring/` (Dataform 監視 view)
- `scripts/sql/skew_latest.sql` (ops 系 SQL)

**問題**: 「monitoring」という名前のものが 4 箇所に散在。

**対応**: ルートの `monitoring/` を `infra/sql/monitoring/` または `pipeline/data_job/dataform/monitoring/` に統合 (Scheduled Query 用 SQL なら Dataform 側、純 SQL なら infra 側)。`docs/04_運用.md` で「監視は何処を見るか」の地図を追加。

## B-7. `scripts/ops/vertex_*.py` の散在

6 本の `vertex_*.py` が `scripts/ops/` 直下に flat に並ぶ。`make ops-vertex-all` で一括実行する関係性は dir に出ていない。

**対応**: `scripts/ops/vertex/` sub-dir に集約 (`scripts/ops/vertex/{models_list,pipeline_status,feature_group,monitoring,rag,explain}.py`)。Make target は `make ops-vertex-models-list` のような既存命名を維持。
