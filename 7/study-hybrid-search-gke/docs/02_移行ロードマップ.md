### C-5. §2.7 messaging モジュールの記載不一致

§2.7 のモジュール表で `messaging/` 行の説明文が:

> Phase 6 は `search-api` + `meili-search` の Cloud Run Service 2 本を管理していたが、Phase 7 では `search-api` を GKE に移譲し、本モジュールは Pub/Sub topic 3 + BQ Subscription 2 + Cloud Scheduler `check-retrain-daily` のみを担う

> Pub/Sub topic 3 (`ranking-log` / `search-feedback` / `retrain-trigger`) + BQ Subscription 2 + Cloud Scheduler `check-retrain-daily` を集約する messaging 専用モジュール (Phase 6 の `runtime/` を Run 3 で rename + 役割縮小、Cloud Run Service 管理は `meilisearch/` + `gke/` + manifests に分離済)

### C-6. §1 冒頭「Phase 7 では Vertex Endpoint 撤去で attach 先が無いため、実質 inactive」と §3.2 の不整合

§3.2 の `mlops.model_monitoring_alerts` の説明が更新され、self-managed Scheduled Query が attach されるようになった (= 実質 active になった)。一方、§1 冒頭の 7 件継承ギャップにはこの変化が反映されておらず、§6 GCP リソース表でも `model-monitoring-alerts` topic は依然「(Phase 7 では inactive)」のまま。

- §6 Pub/Sub Topic `model-monitoring-alerts`: 「Phase 7 では inactive」→ 「self-managed drift Scheduled Query → BQ Subscription、互換のため Vertex Monitoring 経路も残置」に書き換え
