## B. 新たに浮上した論点

### B-1. `ml/training/experiments/` の階層深度

`pipeline/data_job/dataform/`、`ml/streaming/container/` のように「機能 dir 配下に sub-dir」が散在。`ml/training/experiments/` だけが Port/Adapter 構造を持たない単独 sub-dir として残る。

**対応**: `ml/training/experiments/` も `ports/experiment_tracker.py` + `adapters/null_tracker.py` + `adapters/vertex_experiments_tracker.py` の Port/Adapter 構造に揃える。`null_tracker.py` は現状すでに「`ExperimentTracker` Protocol の noop 実装」なので、語彙上 `adapters/` に置く方が一貫する。

### B-2. `app/services/protocols/_types.py` の置き場所

`LexicalResult` / `SemanticResult` が `protocols/_types.py` にあるが、これは **domain object** であって Protocol ではない。

**対応**: `app/domain/search.py` または新設 `app/domain/retrieval.py` に移送。Protocol dir に値オブジェクトが混入すると「protocols/ にあるものは Port」という mental model が崩れる。

### B-3. `_optional_adapter.py` の prefix と配置

Run 3 で `container/_optional_adapter.py` が新設されたが、`_` prefix の helper が `container/` 直下に置かれている。`adapters/internal/` と同じ問題が `container/` でも発生する芽。

**対応**: 件数が増える前に `container/_helpers/optional_adapter.py` のような sub-dir を切るか、あるいは `composition_root.py` の private helper として吸収するか方針決定。現状 1 ファイルなので即時対応不要だが、ガイドラインを `docs/01_仕様と設計.md §4` に追記しておくと将来の拡散を防げる。

### B-4. テスト dir の `fakes/` 二重存在

ドキュメント上 `tests/fakes/` と `tests/unit/` が並列に置かれているが、`tests/fakes/` は `unit/` から import される共通モジュールなので階層が flat すぎる。

**対応**: `tests/_fakes/` (test 用 helper であることを `_` で示す) または `tests/unit/_fakes/` に移送。`tests/integration/` から `fakes/` を import するケースは現状なさそう (整理が必要)。

### B-5. `monitoring/` ルート dir の役割不明瞭

ルート直下に `monitoring/` (feature skew SQL) が単独で存在。一方で:
- `infra/terraform/modules/monitoring/` (Terraform 監視)
- `pipeline/data_job/dataform/monitoring/` (Dataform 監視 view)
- `scripts/sql/skew_latest.sql` (ops 系 SQL)

**問題**: 「monitoring」という名前のものが 4 箇所に散在。

**対応**: ルートの `monitoring/` を `infra/sql/monitoring/` または `pipeline/data_job/dataform/monitoring/` に統合 (Scheduled Query 用 SQL なら Dataform 側、純 SQL なら infra 側)。`docs/04_運用.md` で「監視は何処を見るか」の地図を追加。

### B-6. `infra/manifests/search-api/configmap.example.yaml` の運用

`example.yaml` を「コピーして apply」運用にしているが、これは手作業で drift する典型パターン。

**対応**: 2 案。
- **案 1 (軽量)**: `kustomize` の `configMapGenerator` で `env/config/setting.yaml` から自動生成 (`scripts/ci/sync_configmap.py` 新設、`make sync-configmap` で再生成)。drift 検知 test も追加。
- **案 2 (堅牢)**: ConfigMap も ESO の `ConfigMapTemplate` で Secret Manager 値とともに同期。

軽量案 (1) の方が `sync_dataform.py` と統一感がある。

### B-7. `scripts/ops/vertex_*.py` の散在

6 本の `vertex_*.py` が `scripts/ops/` 直下に flat に並ぶ。`make ops-vertex-all` で一括実行する関係性は dir に出ていない。

**対応**: `scripts/ops/vertex/` sub-dir に集約 (`scripts/ops/vertex/{models_list,pipeline_status,feature_group,monitoring,rag,explain}.py`)。Make target は `make ops-vertex-models-list` のような既存命名を維持。

### B-8. `app/main.py` の路由列挙とハンドラ命名

ドキュメントの router 配線図を見ると `health_router` / `search_router` / `rag_router` / `feedback_router` / `retrain_router` / `model_router` / `ui_router` の 7 本。ファイルは `*_handler.py` の命名だが、export 名は `*_router`。

**対応**: 命名を片方に統一。
- ファイル `*_handler.py` ↔ export `*_router` の不一致は地味に混乱要因。
- FastAPI 慣習に従って `app/api/routers/{health,search,...}.py` + export `router` (シンプル名) が一般的。
- 移行コストが高ければ「ファイル = `*_handler.py`、export = `router`」に統一するだけでも改善。

### B-9. `Container.observability` の test 観測性

新設 `Observability` frozen dataclass で `service_name` / `logger_factory` / `expose_prometheus()` を束ねたのは良いが、unit test での差し替えポイントが明示されていない。

**対応**:
- `Observability.null()` または `Observability.for_test()` factory を生やす (logger は `logging.getLogger("test")`、expose は no-op)。
- `tests/_fakes/observability.py` (上記 B-4 と整合) に置く。

### B-10. `tests/integration/parity/` の lockstep 拡張余地

7 ファイル lockstep が定着したので、parity の **追加対象**を明示しておくと将来の負債を防げる:
- `app/services/protocols/popularity_scorer.py` (BQML feature names) ↔ `bqml/train_popularity.sql` の input columns
- `app/schemas/search.py::SearchFilters` ↔ `app/domain/search.py::SearchFilters` ↔ `mlops.search_logs.filters` RECORD schema

`tests/integration/parity/README.md` に「parity invariant の登録基準」を文書化 (どういう変更で何が連動して壊れるか)。
