# 02_移行ロードマップ (Phase 7 の決定的仕様 + 完成までの残タスク)

本ドキュメントは **Phase 7 (`study-hybrid-search-gke`) の決定的仕様** を載せる。「何を学ぶ / 何を作る」「何を学ばない / 何を作らない」の最終決定と、**Phase 7 完成までの残タスク** のみを管理する。対応済みの履歴は記載しない (`docs/README.md §2` の権威順位に従う)。

Phase 7 の主目的は **Phase 6 (`study-hybrid-search-pmle`) の PMLE 8 技術統合を丸ごと継承した上で、serving 層のみを Vertex AI Endpoint → GKE + KServe InferenceService に差し替え、動くコードで GKE / Kubernetes manifest / Workload Identity / NetworkPolicy / HPA / KServe の一連を手を動かして学ぶ** こと。Phase 3–6 の「同じドメインで実行基盤を差し替える」思想の延長線上にあり、**Port/Adapter のまま adapter 実装を KServe 版に差し替え + 新 Terraform モジュール (`gke` / `kserve`) + K8s manifest (`infra/manifests/`) を追加** する形で移行する。

## 0. 最重要ルール — ハイブリッド検索というテーマと中核コードは絶対に変えない

Phase 7 の全作業を通じて **以下だけは絶対に変えない**:

- **題材 / ドメイン**: 不動産ハイブリッド検索 (クエリ文 + フィルタ → 物件ランキング上位 20 件)
- **ハイブリッド検索の中核コードと挙動**:
  - (1a) Meilisearch BM25 (lexical)
  - (1b) BigQuery `VECTOR_SEARCH` + multilingual-e5 embedding (semantic)
  - (2) RRF 融合
  - (3) LightGBM LambdaRank 再ランク
  - `/search` エンドポイントの現行動作 (デフォルト構成での出力 = 既存テスト全通過)
- **Phase 6 PMLE 8 技術の統合結果**: T1 BQML / T2 Dataflow / T3 Vertex Vector Search / T4 Explainable AI / T5 Monitoring SLO / T6 RAG (Gemini) / T7 Agent Builder / T8 Model Garden — ファイル配置 / Port / adapter / feature flag / Terraform モジュールはすべて Phase 6 のまま。Phase 7 で手を入れるのは serving 層差し替えに付随する最小限 (§3) だけ

この中核と Phase 6 統合を **置換・削除・無効化する変更はしない**。親リポ `CLAUDE.md` の非負制約 (「ハイブリッド検索の基本構成は LightGBM + multilingual-e5 + Meilisearch を必須」) を Phase 7 でもそのまま継承する。

### 一方で変えてよいもの (Phase 7 の主目的)

- **Serving 層の adapter 差し替え**: `EncoderClient` / `RerankerClient` Port の Vertex Endpoint 実装 → KServe 実装 (新 Port 追加は原則しない、既存 Port を KServe adapter が satisfy する形に揃える)
- **新 Terraform モジュール追加**: `infra/terraform/modules/gke/` (GKE Autopilot cluster) + `infra/terraform/modules/kserve/` (KServe Helm chart)
- **K8s manifests の追加 / 調整**: `infra/manifests/{search-api,kserve}/` 配下の Deployment / Service / Gateway / HPA / NetworkPolicy / PodMonitoring / InferenceService
- **新 deploy スクリプト**: `scripts/deploy/api_gke.py` (Cloud Build + kubectl set image) / `scripts/deploy/kserve_models.py` (Model Registry → kubectl patch storageUri) / `scripts/deploy/monitor.py` (リアルタイム進捗監視)
- **既存 Phase 6 Terraform への属性追加** (GKE 対応のため): `iam` モジュールに Workload Identity binding 追加 / `slo` モジュールの `service_type` を `cloud_run` → `k8s_service`
- **新 Make target 追加**: `make deploy-api` / `make deploy-kserve-models` / `make kube-creds` / `make ops-reload-api` / `make ops-deploy-monitor` / `make ops-run-all-monitor`

= **GKE / KServe を動くコードで学ぶのに必要な修正は、中核コードと Phase 6 統合を崩さない範囲で自由に行う**。

## 3. 差分実装 (Phase 6 実コードへの serving 層差し替え)

Phase 6 実コードのうち serving 層のみを差し替える。**中核挙動 (§0) はデフォルト維持**、Phase 6 PMLE 8 技術の adapter / 副経路 / 新 endpoint はそのまま継承。

### 3.1 serving 層差し替え一覧

| # | 差し替え対象 | Phase 6 (継承元) | Phase 7 (差し替え) | 中核挙動への影響 |
|---|---|---|---|---|
| S1 | `search-api` ランタイム | Cloud Run Service (`--memory 2Gi --min-instances 1`) | **GKE Deployment** (namespace `search`、`requests=500m/1Gi` `limits=2/2Gi`、`minReplicas=1 maxReplicas=10`、HPA CPU70%/Mem80%) | なし (/search レスポンス互換) |
| S2 | encoder 推論 | `VertexEndpointEncoder` (Vertex AI Endpoint `property-encoder-endpoint`) | **`KServeEncoder`** (`app/services/adapters/kserve_prediction.py`、cluster-local HTTP V1 `:predict`) | なし (Port `EncoderClient` 互換) |
| S3 | reranker 推論 | `VertexEndpointReranker` (Vertex AI Endpoint `property-reranker-endpoint`) | **`KServeReranker`** (cluster-local HTTP V2 `:infer`、MLServer LightGBM runtime) | なし (Port `RerankerClient` 互換) |
| S4 | 認可境界 | Cloud Run `--no-allow-unauthenticated` + IAM invoker | **Gateway API + IAP** (外部) + **NetworkPolicy** (内部、search → kserve-inference のみ許可) | なし |
| S5 | SA 紐付け | Cloud Run service account | **Workload Identity** (GKE KSA `search/search-api` / `kserve-inference/encoder` / `kserve-inference/reranker` に Phase 6 の `sa-api` / `sa-endpoint-encoder` / `sa-endpoint-reranker` を bind) | なし |
| S6 | モデル artifact | Model Registry → Endpoint の DeployedModel | Model Registry → KServe `storageUri` (`scripts/deploy/kserve_models.py` が `production` alias を引いて `kubectl patch`) | なし |
| S7 | デプロイ経路 | `gcloud run deploy search-api --image=<tag>` | Cloud Build → Artifact Registry → **`kubectl set image deployment/search-api`** (`scripts/deploy/api_gke.py`) | なし |
| S8 | 監視 | Cloud Run built-in metrics + Cloud Run SLO | **PodMonitoring** (GMP) + SLO の `service_type` を `k8s_service` に書き換え | なし (監視の土台だけ差し替え) |

serving 層差し替えに伴う Phase 6 T4 / T5 の最小改修のみ (他の T1 / T2 / T3 / T6 / T7 / T8 は **完全不変**):

| # | 対象 | 改修内容 |
|---|---|---|

### 3.4 共通原則

- **Port/Adapter 境界を維持** — `scripts/ci/layers.py::RULES` に違反しない。KServe adapter は `app/services/adapters/kserve_prediction.py` に置き、services から Port `EncoderClient` / `RerankerClient` 経由で使う
- **`make check` を通す** — Phase 7 追加コードも ruff / ruff format / mypy strict / pytest の CI 同等チェックに含める
- **`make tf-validate` を通す** — 新 provider (`kubernetes` / `helm`) も含めて offline validate が成功すること
- **`kustomize build infra/manifests/` を通す** — manifest の構文破綻を CI で検知
- **6 ファイル parity invariant を壊さない** — Phase 7 では基本触らないが、incidentally に特徴量追加が必要になったら Phase 6 と同じ 6 ファイル (`pipeline/data_job/dataform/features/property_features_daily.sqlx` / `ml/data/feature_engineering/ranker_features.py` / `ml/data/feature_engineering/schema.py` / `infra/terraform/modules/data/main.tf::ranking_log.features` / `monitoring/validate_feature_skew.sql` / `infra/terraform/modules/vertex/main.tf::google_vertex_ai_feature_group_feature` × N) を同 PR で揃える
- **Terraform で宣言** — 手で `gcloud container clusters create` / `helm install` / `kubectl create deployment` を叩かない。cluster と KServe は Terraform、Pod 系 manifest は `kubectl apply -k infra/manifests/`
- **immutable tag deploy** — Cloud Build で `<sha>-<epoch>` tag を毎回発行し、build 完了後に同一 tag を `kubectl set image`。tag 削除先行はしない (ImagePullBackOff レース回避)
- **default は Phase 6 挙動** — Phase 6 feature flag のデフォルト値は Phase 7 でもそのまま維持。既存テストが全通過する状態を保つ

### 3.5 学習リポとしての最小スペック選定 (コスト配慮)

- GKE Autopilot (cluster 自体は idle で ~¥2.4k/月の control plane 料金)
- search-api Deployment: `requests=500m/1Gi`、`limits=2/2Gi`、`minReplicas=1 maxReplicas=10` — 通常は 1 Pod
- KServe predictor: `minReplicas=1 maxReplicas=1` 固定 (コールドスタート回避、学習終了後は cluster 丸ごと destroy)
- Phase 6 から引き継いだ常時課金リソース (Vector Search / Dataflow job / Agent Builder Engine) は Phase 7 でも default で無効 (`enable_*=false`)
- **学習終わり次第 `make destroy-all` で全破棄する運用** を厳守。GKE Autopilot cluster も destroy 対象
- コストが増える方向の変更 (`maxReplicas > 1` 常駐、large machine、GPU) は User 合意を明示的に取ってから

### 4.3 ブロッカーになりうる技術的リスク

Phase 7 完成までに踏みそうな技術的ハマり。踏んだら `docs/04_運用.md §1.6` か §4 のインシデント対応に追記する:

- **GKE Autopilot + Gateway API の provisioning 時間**: 初回 cluster 作成 + Gateway の external IP 割当てまで 5–10 分かかる。`make deploy-all` のタイムアウト調整が必要になるかも
- **KServe Helm chart + cert-manager の依存**: kserve/kserve chart は cert-manager を前提にする。`modules/kserve/` でまとめて helm_release するか、cert-manager は別モジュールに切り出すか判断
- **Workload Identity bind の propagation**: KSA ↔ GCP SA の bind 反映に数秒〜数十秒遅延がある。deploy-all 直後に Pod が起動した場合 `iam.gke.io/gcp-service-account` annotation が効かず 403 を返すことがある
- **KServe storageUri 更新の反映時間**: `kubectl patch` 後に InferenceService が新 URI で predictor を pull → 起動するまで数分。`make run-all-core` の ops-train-now → 即 ops-search だと前世代の model を叩く可能性
- **IAP の OAuth brand 作成**: GCP コンソールで OAuth consent screen を手動で作成する必要がある (Terraform で完全自動化できない公式制約)
