### 0.3 なぜローカルで完結できないのか (= V1-V5 すべて live verify 必須の理由)

**ローカルで弾けるはずの失敗はローカルで弾く** が原則だが、以下は **GCP 実機でしか観測不可**:

- `alreadyExists` 系 error (= GCP 残置 resource との衝突): 実 GCP project state に依存
- VVS deployed index attach (~26 min): GCP managed operation
- Composer 環境作成 (~20 min) / Composer DAG task の Pod 起動: GCP managed operation
- KServe `production` alias 切替: cluster-side
- 3 種 (lexical/semantic/rerank) all non-zero: cluster-side serving

→ V1-V6 はすべて live verify 必須。**今 sprint の進捗状況** ([TASKS.md](../tasks/TASKS.md)) と **過去 incident** ([TASKS_ROADMAP.md §4.X](../tasks/TASKS_ROADMAP.md)) を別ファイルで参照。

## 1. 検証の考え方

PDCA を回す前 / 回しきった後に「何を見れば 'OK' とみなしてよいか」の checklist。**ローカルで早く落とせる失敗はローカルで落とし、cluster 依存は GCP 側で確認する**。加えて、**教育コードとしての完成条件は canonical 1 経路への収束**なので、「互換レイヤが残っていないこと」自体も検証対象に含める。

階層別の責務:

| 層 | 何が確認できる | 所要時間 | 認証 |
|---|---|---|---|
| ローカル (§2) | import / boot / 静的検証 / image 構成 / obvious legacy residue | 1-5 分 | ADC のみ (任意) |
| GCP 検証ゲート (§3) | canonical 経路の 3 種が all non-zero、NDCG@10、Gateway TLS、KServe Workload Identity、互換レイヤ不要の確認 | 3-10 分 (deploy 後) | フル GCP |
| 退化検知 (§5) | 性能ベースライン (build 時間、image size 等) との乖離 | 数分 | ローカル + GCP |

ローカルで弾けるはずの失敗が GCP に持ち込まれると、deploy-all を 1 回ぶん (~5 分) と Cloud Build コストが無駄になる → **§2 を先に通す**。

L5 が「ローカルだけで完結しない」のは設計上の制約。**lexical 単体の Meilisearch はローカルで局所確認できる** 一方、Phase 7 の完成条件は Meilisearch だけではなく、Vertex Vector Search / Feature View / KServe reranker を含む canonical 1 経路での成立にある。そもそも Meilisearch を採用している理由の 1 つが、**lexical 検索だけはローカルでも触って学べるようにすること**にある。Phase 6 (Cloud Run 単体) は in-process reranker でローカル統合できたが、Phase 7 は serving 層を KServe に移譲した結果、reranker が cluster-side HTTP 呼び出しに変わった。意図的なトレードオフ:

- **ローカルで早く回せるもの**: import / boot / 静的検証、Meilisearch の lexical 局所確認
- **実機でしか回せないもの**: 3 種が all non-zero、Vertex Vector Search 実検索、Feature View 経由 fetch、互換レイヤ不要の確認、NDCG@10、Gateway TLS、KServe Workload Identity

## 4. 検証中に出やすい一過性エラーと recovery

PDCA で実測したエラー (恒久対処済ではなく、**遭遇したらリトライ手順を踏む**もの)。コード修正済の永続的ハマりは `05_運用.md §1.6` を参照。

| 症状 | 発生フェーズ | 一次原因 | recovery |
|---|---|---|---|
| `Error waiting to create Service: ... container failed to start and listen on PORT=8080` (Meilisearch Cloud Run) | `terraform apply` 中 | 既存 service が部分作成状態 (cold-start race / image pull 失敗) | `gcloud run services delete meili-search --region=asia-northeast1 --quiet` で消してから `make deploy-all` 再実行 (Terraform が冪等で再作成) |
| `Error trying to delete bucket mlops-dev-a-pipeline-root containing objects without 'force_destroy' set to true` | `make destroy-all` 終盤 | run-all 中の Vertex Pipeline が `gs://...-pipeline-root/runs/<job-id>/` に artifact を書き込んだ後、destroy-all stage [3/6] の bucket wipe 後に新規 artifact が来た | `gcloud storage rm -r gs://mlops-dev-a-pipeline-root/runs/` → `make destroy-all` 再実行 |
| `cannot destroy table ... property_features_online_latest without setting deletion_protection=false` | `make destroy-all` 本体 destroy | online serving view を追加したのに `scripts/setup/destroy_all.py::PROTECTED_TARGETS` へ未反映 | `PROTECTED_TARGETS` に `module.data.google_bigquery_table.property_features_online_latest` を追加し、`tests/integration/infra/test_destroy_all_table_parity.py` baseline を更新 |
| `URLError: Connection reset by peer` on `make ops-livez` | `make run-all-core` 開始直後 | Gateway PROGRAMMED 完了前 / managed cert 発行中 | 60 秒待機 → `kubectl get gateway -n search` で `PROGRAMMED: True` を確認してから再実行 |
| `TimeoutError: The read operation timed out` on `make ops-search` | `make run-all-core` 中盤 (rolling restart 直後 / KServe cold start 中) | search-api Pod もしくは KServe predictor の cold-start で `/search` 応答が transient に遅延 | `scripts/ops/search.py` は `SEARCH_RETRIES` (default 3) / `SEARCH_RETRY_SLEEP` (default 2.0s) で透過リトライするため通常は再 invocation 不要。それでも継続 fail なら `kubectl get pods -n search,kserve-inference` と `kubectl logs deployment/search-api -n search` を確認 |
| Pod が `CrashLoopBackOff: ModuleNotFoundError: No module named 'scripts'` | rollout 直後 | search-api Dockerfile が `COPY scripts/` を欠いている (Phase 7 Run 5 で実害発生) | `infra/run/services/search_api/Dockerfile` の `COPY scripts/ scripts/` と runtime stage の `COPY --from=builder /src/scripts /home/app/scripts` が両方あるか確認 (現在の HEAD は対処済) |

**運用方針**: いずれも「同じコマンドをもう一度叩けば通る」もの。手動修復ループに入ったら `destroy-all` でクリーンスレートに戻すのが PDCA の trade-off。

| GCP `run-all-core` 完走 | `make run-all-core` | 成功 (exit 0)。`ops-livez → ops-search → ops-search-components → ops-vertex-vector-search-smoke → ops-vertex-feature-group → ops-feedback → ops-ranking → ops-train-now+wait → ops-label-seed → ops-daily → ops-accuracy-report` を全 PASS で通過。`ndcg_at_10=1.0 hit_rate_at_10=1.0 mrr_at_10=1.0` |

今回の live 修正で踏んだ根因:

- `deploy-all` が lexical lane (`sync-meili`) と semantic lane (`backfill-vvs`) を内包しておらず、完走後も `lexical=0` / `0 neighbors` になり得た
- `seed_minimal.py` が `feature_timestamp` を入れておらず、Feature View sync 前提を壊していた
- Feature View fetch 側が canonical resource name を使っていなかった
- Feature View source が online serving に不向きな schema を直接見ていた

恒久対処:

- `deploy-all` に `sync-meili` と `backfill-vvs` を追加し、`seed-test -> sync-meili -> backfill-vvs -> trigger-fv-sync` を workflow contract として固定
- seed に `feature_timestamp = CURRENT_TIMESTAMP()` を追加
- fetch 側で Admin API から canonical Feature View 名を解決
- online serving 用に `property_features_online_latest` view を追加し、Feature View source をそこへ固定

**ベースライン退化を疑うべき編集**:
- `pyproject.toml` の `dependencies` を増やした (特に `torch` / `tensorflow` / `kfp` 系)
- `Dockerfile` のレイヤ順を変えた (`COPY app/` を上に動かすと cache hit 率が崩れる)
- `cloudbuild.yaml` から kaniko を docker builder に戻した

## 6. 検証依頼プロンプト例 (Claude Code 向け)

「どの粒度で検証してほしいか」を明示すると、過不足のない作業範囲で動かせる。コピペで使える例:

### 6.1 ローカルだけ (1-5 分、GCP 認証不要)

```
ローカル検証だけ依頼。L1+L2 (make check + image build)
```
→ `make check-layers` → `make check` → `docker build` を回し、image size と test 結果を報告。

```
ローカル起動チェック依頼 (L4 まで)
```
→ 上に加えて `docker run` で `/livez` 200 確認まで。

### 6.2 単発 GCP 検証 (cluster は既に稼働中)

```
deploy 済みの cluster で 3 種 (lexical/semantic/rerank) チェック依頼。NDCG も
```
→ `make ops-search-components` + `make ops-vertex-vector-search-smoke` + `python -m scripts.ops.vertex.feature_group` + `make ops-accuracy-report` を叩いて結果報告。失敗時は §3 G3-G5 の分岐表に従って原因切り分け。

```
ops-search-components が rerank=0 になる原因切り分け依頼
```
→ KServe `production` alias / `make deploy-kserve-models` 状況 / pod log の順で診断。

### 6.3 フル PDCA (~30 分、cluster 全消し→再構築→検証→消す)

```
destroy-all, deploy-all, run-all, destroy-all の PDCA サイクル検証依頼
```
→ 4 step を順に実行、各 phase の elapsed と canonical ゲート (G3-G5) 結果を報告。

```
deploy-all + run-all-core だけ依頼 (destroy はしない、cluster 残す)
```
→ コストが残るため明示要求時のみ。run-all-core 完了後に「destroy しますか?」と確認。

### 6.4 変更後の影響確認 (退化検知)

```
Dockerfile の COPY 順を変えた。L1+L2+G3-G5 で退化検知依頼。
image size と build 時間が §5 のベースラインと乖離していないか確認
```
→ ローカル build → ベースライン照合 (705MB / kaniko cache hit ~3 分等) → 必要なら deploy して G3-G5 まで。

```
pyproject.toml の dependencies を増やした。重い依存が search-api image に
混入していないか確認依頼
```
→ ローカル build → `docker run --entrypoint=python ... -c "import torch, lightgbm, kfp"` が全て ImportError であることを検証 (§5 の退化検知項目)。

### 6.5 一過性エラー recovery

```
make deploy-all が Meilisearch Cloud Run health check で失敗。recovery 依頼
```
→ §4 の一次手順 (`gcloud run services delete meili-search` → `make deploy-all` 再実行)。

```
make destroy-all が pipeline-root bucket force_destroy エラー。クリーンアップ依頼
```
→ `gcloud storage rm -r gs://mlops-dev-a-pipeline-root/runs/` → `make destroy-all` 再実行。

### 6.6 計測 / ベースライン更新

```
今回の最適化で deploy-all 時間がどれだけ縮んだか計測依頼。
§5 のベースラインを実測値で更新
```
→ destroy → deploy → 各 phase elapsed 抽出 → 表を更新する PR を起こす。
