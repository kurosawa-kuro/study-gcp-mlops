## 1. PDCA メインフロー (`make deploy-all` → `make run-all` → `make destroy-all`)

### 1.0 設計意図

**学習用 dev project (`mlops-dev-a`) で PDCA を最速回転させる**。本リポは試行錯誤の場であり、確認プロンプト / 古いイメージの混入 / 半端な手順は全部排除している:

- **`make deploy-all`** で **provisioning ゼロから search-api の GKE rollout 完了まで一発**。途中で確認は出ない。encoder / reranker の KServe InferenceService は Model Registry に `production` alias が付いてから `make deploy-kserve-models` で storageUri 更新 (deploy-all からは外してある)。
- **`make run-all`** (= `make run-all-core`) で **疎通・検索・学習トリガ・日次観測までの検証を一連実行**。検証漏れを減らす。
- **`make destroy-all`** は **プロンプトなし** で全 Terraform リソースを破壊する (BQ table 10 件の `deletion_protection` も自動で剥がす)。
- イメージは **immutable tag** (`<sha>-<epoch>`) で毎回新規 push し、その tag を `kubectl set image deployment/search-api` で deploy する。tag 削除先行は行わない (`ImagePullBackOff` レース回避)。
- tfstate バケットと API 有効化は teardown 後も残す → `make deploy-all` を再実行すれば数分で戻る。

```
       ┌────────────────────────┐
   ┌──>│   make deploy-all      │  (約 12-15 分: GKE 作成 + Helm apply + Cloud Build + kubectl set image)
   │   └───────────┬────────────┘
   │               ↓
   │       実 GCP で実験 / 観測 / コード修正
   │               ↓
   │   ┌────────────────────────┐
   │   │   make ops-* で疎通    │  (livez / search / ranking / feedback / label-seed)
   │   └───────────┬────────────┘
   │               ↓
   │     コード変更 → 再度 make deploy-all (image は immutable tag で再 rollout)
   │               │
   │               ↓
   │   ┌────────────────────────┐
   └───┤   make destroy-all     │  (約 3-5 分: protect 解除 + terraform destroy)
       └────────────────────────┘
```

### 1.1 1 度だけ済ませる前提 (PDCA 周回前の準備)

ここは PDCA ループの**外側**。マシン交換時 / 新規メンバー初回のみ実行する。

```bash
# (a) 前提ツール — uv / terraform / gcloud / make / jq (詳細は §2 STEP 1)
make doctor                                        # 全行 OK で次へ

# (b) Python 依存
make sync

# (c) GCP 認証 + project 固定
gcloud auth application-default login
gcloud config set project mlops-dev-a              # = env/config/setting.yaml::project_id

# (d) env/config/setting.yaml の確認 (要なら編集)
cat env/config/setting.yaml
# project_id / region / github_repo / oncall_email 等が自分の環境で正しいか
```

`make doctor` は `WARN VIRTUAL_ENV=...` を出すことがある (外側で別 venv を `source` 済の場合)。害はないが、混乱回避のため `deactivate` してから make を叩くのを推奨。

### 1.2 PDCA 1 サイクル (典型操作)

```bash
# (1) 立てる
make deploy-all
#   → tf-bootstrap → tf-init → _recover_wif_state → sync-dataform
#   → tf-plan → terraform apply (GKE Autopilot + KServe Helm + Meilisearch on Cloud Run ほか)
#   → scripts.deploy.api_gke (Cloud Build → Artifact Registry → `kubectl set image deployment/search-api`)
#   約 12-15 分。途中の対話プロンプトなし。
#   ※ encoder/reranker の KServe InferenceService storageUri は別途 `make deploy-kserve-models`
#     (Vertex Model Registry に `production` alias が付いた後)。

# (2) 検証を一括実行
make run-all           # = run-all-core + リアルタイム監視 (ops-run-all-monitor)
#   → check-layers → seed-test → ops-train-now → ops-livez/search/search-components/feedback/ranking
#   → ops-label-seed → ops-daily → ops-accuracy-report
#   疎通 + 主要運用チェックをまとめて確認

# (3) 試したい変更を加える (コード / Terraform / Dataform / manifests)
$EDITOR ...

# (4) 再 rollout + 再検証
make deploy-all
make run-all
#   tfstate / バケット / API 有効化はそのまま、差分だけ apply。image は毎回 immutable tag を作って
#   `kubectl set image` で rollout。

# (5) PDCA セッション終了 → 全部消してコスト止める
make destroy-all
#   プロンプトなし。約 3-5 分で terraform destroy 完了。
#   tfstate バケットは残る → 翌日 (1) から再開可能。
```

### 1.3 `make deploy-all` の内訳 (7 step)

`scripts/setup/deploy_all.py` が以下を順に Python module 直 import で実行する。**個別 step を単独 make target で叩いても等価**:

| # | 中身 | 個別 target | 所要時間 (目安) |
|---|---|---|---|
| 1 | API 有効化 + tfstate バケット作成 (冪等) | `make tf-bootstrap` | ~30s (初回)、再実行 ~5s |
| 2 | tfstate バケット preflight + `terraform init` | `make tf-init` | ~10s |
| 3 | WIF pool / provider の soft-delete 復元 (直前に destroy した場合のみ必要、冪等) | (組込、Python) | <10s |
| 4 | `env/config/setting.yaml` → `pipeline/data_job/dataform/workflow_settings.yaml` 生成 | `make sync-dataform-config` | <1s |
| 5 | `terraform plan -out=tfplan` (vars は setting.yaml から) | `make tf-plan` | ~20s |
| 6 | `terraform apply -auto-approve tfplan` (GKE Autopilot + KServe Helm + Meilisearch 用 Cloud Run + BQ / Pub/Sub / Vertex / 監視ほか。`time_sleep` 含む) | (組込) | ~8-12 分 (初回) / ~2 分 (再 apply) |
| 7 | search-api: Cloud Build → Artifact Registry → `kubectl set image deployment/search-api` (immutable tag) | `make deploy-api` | ~5-7 分 |

**deploy-all に含まれない** (手動で叩く):

| target | タイミング | 用途 |
|---|---|---|
| `make deploy-kserve-models` | Vertex Model Registry に `production` alias 付与後 | `property-encoder` / `property-reranker` InferenceService の `storageUri` を `kubectl patch` |
| `make ops-train-now` | run-all 内で自動。手動で再学習したいとき | Vertex AI Pipelines (`property-search-train`) を submit |
| `make ops-pipeline-run TARGET=embed` | raw.properties が差分更新された時 | `property-search-embed` pipeline を submit → `feature_mart.property_embeddings` |

詰まった step は §2 の STEP に対応 (1↔STEP 8、2↔STEP 9 head、5-6↔STEP 9 後半、7↔ `scripts.deploy.api_gke`)。

### 1.4 `make destroy-all` の内訳

`scripts/setup/destroy_all.py`:

1. `terraform apply -auto-approve -var=enable_deletion_protection=false -target=<9 BQ tables>` — 計 9 table (`mlops.training_runs` / `mlops.search_logs` / `mlops.ranking_log` / `mlops.feedback_events` / `mlops.validation_results` / `feature_mart.property_features_daily` / `feature_mart.property_embeddings` / `mlops.model_monitoring_alerts` / `mlops.ranking_log_hourly_ctr`) の `deletion_protection` 属性だけを state-flip。`-target` が load-bearing — 半 destroy 状態で SA recreate が fail するのを回避するため、apply 範囲を tables に限定する。新しい protected table を `infra/terraform/modules/data/main.tf` に追加したら、`scripts/setup/destroy_all.py::PROTECTED_TABLE_TARGETS` にも同じアドレスを追記する。
2. Vertex AI Endpoint (`property-encoder-endpoint` / `property-reranker-endpoint`) の DeployedModel を undeploy (Phase 7 では KServe に serving を移譲しており、Vertex Endpoint は shell リソースのみ。残留 DeployedModel があると destroy が詰まるため事前に `scripts/setup/destroy_all.py::VERTEX_ENDPOINTS` 経由で消す)。
3. `force_destroy=false` な GCS バケット (`-models` / `-meili-data` 等) を `gcloud storage rm --recursive` で空に。
4. `terraform -chdir=infra/terraform/environments/dev destroy -auto-approve -var=enable_deletion_protection=false -var=github_repo=... -var=oncall_email=...` (vars は env > setting.yaml)。

**所要時間**: ~3-5 分 (state-flip apply ~30s + endpoint undeploy + bucket wipe + destroy ~2-4 分)。

**残るもの** (再 deploy のため意図的に保持):
- tfstate バケット `gs://mlops-dev-a-tfstate`
- API enablement (リソース不在時はゼロ課金)

完全クリーンスレートにしたい場合のみ、destroy 後に `gcloud storage rm --recursive gs://mlops-dev-a-tfstate` を手動実行。

### 1.5 PDCA を成立させている技術的保証

PDCA を回す上で「触るたびに違う結果」になると致命的。本リポは以下で再現性を担保:

| 機構 | 場所 | 効果 |
|---|---|---|
| `setting.yaml` を single source of truth に集約 | `env/config/setting.yaml` + `tests/integration/parity/test_dataform_workflow_settings.py` | drift を CI が検知。Make / Python / Dataform yaml が同じ値を見る |
| **immutable tag deploy** (`<sha>-<epoch>` を build→deploy で固定) | `scripts/deploy/api_gke.py` | tag 削除先行による `ImagePullBackOff` を回避しつつ、どの build を deploy したかを一意に追跡 (Cloud Build → Artifact Registry → `kubectl set image`) |
| **no-prompt** (確認プロンプトなし) | `scripts/setup/destroy_all.py` ほか | PDCA を `make` 1 発で完走、自動化を妨げない |
| placeholder image + `lifecycle.ignore_changes` | `infra/manifests/search-api/deployment.yaml` の初期 image + Deployment 側で image 差し替えを許容 | 初回 apply 時に image が無くても Deployment 作成成功、後続 `deploy-api` で real image に差し替え可 |
| WIF pool/provider の soft-delete 復元 | `scripts/setup/deploy_all.py::_recover_wif_state` | GCP は WIF pool を delete 後 30 日保持する仕様。直前に `destroy-all` した直後の `deploy-all` で 409 にならないよう undelete + state import を自動実行 |
| tfstate / API 有効化を destroy で消さない | `scripts/setup/destroy_all.py` | 翌サイクルの `deploy-all` が即座に開始可能 (Phase 0 を毎回やらない) |
| feature parity invariant (6 ファイル lockstep) | `tests/integration/parity/test_feature_parity_*.py` | 試行錯誤中に schema が崩れて scheduled query / Vertex Feature Group が FAIL するのを CI で防ぐ |

### 1.6 PDCA で踏みやすいハマり (修正済)

`make deploy-all` を初回叩いたとき、過去に踏んで **infra/scripts 側で恒久対処済** のハマりを置いておく (再発時にここを起点に追える):

| 症状 | 原因 | 対処箇所 |
|---|---|---|
| `Cannot find metric(s) that match type = "logging.googleapis.com/user/search_api_5xx"` | log-based metric 直後 ~60s は Monitoring API の index が追いつかず alert policy が参照不可 | `infra/terraform/modules/monitoring/main.tf` の `time_sleep.wait_for_log_metric_indexing` (90s) + 両 alert policy の `depends_on` |
| `Cannot create a transfer in JURISDICTION_US when destination dataset is located in REGION_ASIA_NORTHEAST_1` | `google_bigquery_data_transfer_config.location` 既定が US で、`asia-northeast1` の dataset と不整合 | `infra/terraform/modules/monitoring/main.tf` に `location = var.region`、root から `region` を渡す |
| `Failed to find a valid credential. The field 'version_info' or 'service_account_name' must be specified.` | DTS は `version_info` (user OAuth) か `service_account_name` (SA 代理) のいずれか必須、provider 既定はどちらも付けない | DTS に `service_account_name = var.service_accounts.dataform.email`、`infra/terraform/modules/data/main.tf` に `dataform_mlops_editor` を追加 |
| `terraform apply` が `Error: Post "https://.../api": dial tcp: lookup`/ `connect: connection refused` | `kubernetes` / `helm` provider が cluster 未作成時点で activate されようとしている | 初回のみ `-target=module.iam -target=module.data -target=module.gke` で cluster を先行作成 (`infra/terraform/environments/dev/provider.tf` の 2 段階 apply コメント参照)。2 回目以降の `make deploy-all` は全体 apply でよい |
| `ImagePullBackOff` on `deployment/search-api` after `kubectl set image` | Artifact Registry への push 未完了の tag を deploy に回した (Cloud Build のレース) | `scripts/deploy/api_gke.py` は Cloud Build 完了を待ってから `kubectl set image` を叩くように組み立て済。自前で並列化しない |
| `make tf-plan` で `ONCALL_EMAIL is required` (backslash 改行で trailing whitespace 混入) | shell の line continuation 破壊 | setting.yaml が既定値を持つ + Make を 1 行で叩く |
| GKE Deployment `/healthz` probe が持続的に失敗 | `readiness_probe` が encoder 未ロード時に 503 を返す正しい挙動。起動直後のみ ~30-60s 503 を許容 | `deployment.yaml` の `startupProbe.failureThreshold` を余裕持たせる (現状 30 回 × 10s = 5 分) |
| Meilisearch Cloud Run の `meili-master-key` secret 未投入で search-api が 5xx | Terraform は secret container まで作成。値自体は STEP 10 で Secret Manager へ投入し、GKE 側は External Secrets Operator が自動同期する | `env/secret/README.md` と `kubectl -n search get externalsecret meili-master-key` で状態確認 |

## 2. 個別 STEP 詳細 (PDCA から外れた手動操作が必要な場合)

`make deploy-all` で完結しない場面 — ローカルだけで API を立てたい (GCP 認証ゼロ) / 1 step だけ手動で叩きたい / CI を有効化したい — の参考用。フローは旧 STEP 1–17 をそのまま残してある。

**STEP 1–6** まではローカル完結 (GCP 認証不要)。お試し / CI 目的はここで終了。
**STEP 7–8** は PDCA 周回開始前の 1 回だけ (※ `make deploy-all` 内部でも実行されるので、deploy-all を回す予定なら個別実行は不要)。
**STEP 9–17** は `make deploy-all` のサブステップを分解したもの + ステージング疎通の中身。

### STEP 1: 前提ツールをインストール

| ツール | 用途 | 最低バージョン |
|---|---|---|
| Python | ランタイム | 3.12 |
| [`uv`](https://docs.astral.sh/uv/) | パッケージ / ワークスペース管理 | 0.4 |
| [`terraform`](https://developer.hashicorp.com/terraform/install) | IaC (STEP 8 以降) | 1.9 |
| [`gcloud`](https://cloud.google.com/sdk/docs/install) CLI | GCP 操作 (STEP 7 以降) | 最新 |
| `make` | ワークフロー一括実行 | 3.8 |
| `jq` | `make ops-check-retrain` の JSON パース | 任意 |
| `docker` | ローカル Dockerfile ビルド確認 | 任意 |

Ubuntu / WSL のインストール例:

```bash
# uv
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env    # PATH を反映 (新ターミナルで不要)

# Terraform (HashiCorp 公式 APT)
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform

# gcloud CLI
sudo apt install google-cloud-cli

# jq (任意)
sudo apt install jq
```

### STEP 2: ツール到達確認

```bash
make doctor
```

全行 `OK` で次へ。`MISS` が出たら STEP 1 に戻る。`python3` のバージョンは `pyproject.toml` の `requires-python = ">=3.12"` を満たせば何でもよい (3.12 / 3.13 / 3.14 いずれも可)。

`WARN VIRTUAL_ENV=... は project 外` が出たら、外側で別の venv を `source` している。uv は repo 直下の `.venv/` を使うため活動中の venv は無視される (害はないが混乱の元)。`deactivate` してから以降の `make` を叩くのを推奨。

### STEP 3: Python 依存を入れる

```bash
make sync
```

uv workspace (`common` / `app` / `jobs`) と `dev` グループ (ruff / mypy / pytest / httpx) を一括導入。`.venv/` が root に作られる (`uv` が自動管理するので手動で activate 不要 — `make` 系は常に `uv run` 経由で解決する)。

**所要時間**: 初回 ~2-3 分 (`sentence-transformers` が引く `torch` ~2GB ほか ML deps の download + 展開)。再実行時はキャッシュヒットで数秒。

### STEP 4: 静的検証 + テスト

```bash
make check
```

ruff / ruff-format / mypy strict / pytest を順に実行。CI と同じ。全緑で次へ。

### STEP 5: 合成データで学習 smoke

```bash
make train-smoke-persist
```

`/tmp/hybrid-search-cloud-smoke-model.txt` に LightGBM LambdaRank モデルが書き出される。BQ/GCS は触らない。

### STEP 6: API をローカル起動 (GCP 認証ゼロ)

```bash
make api-dev
```

uvicorn が **フォアグラウンドで** `http://127.0.0.1:8000` に bind する (`ENABLE_SEARCH=false` 既定なので `/search` は 503 を返す)。別ターミナルで疎通:

```bash
curl -sSf http://127.0.0.1:8000/healthz
```

/search を local で叩きたい場合は、後続の `make api-dev-hybrid` を使う。
確認後 **Ctrl+C で api-dev を停止** してから STEP 7 へ。GCP にデプロイしない場合はここで終わり。

### STEP 6b: ハイブリッド検索を local で起動 (推奨)

`make api-dev` は疎通確認専用。**実際に hybrid 検索 (`/search`) まで local で動かす標準手順は `make api-dev-hybrid`** とする。

```bash
make api-dev-hybrid
```

このターゲットは `scripts/setup/local_hybrid.py` を呼び、以下をまとめて処理する:

1. `env/config/setting.yaml` から **非秘密値** を読む
2. `env/secret/credential.yaml` から **ローカル用秘密値** を読む
3. `meili_master_key` が local に無ければ **Secret Manager** (`meili_master_key_secret_id`) から取得する
4. `/tmp/hybrid-search-cloud-smoke-model.txt` が無ければ synthetic LightGBM model を作る

- `/search` が 200

### STEP 9: Terraform apply (インフラ構築)

> **PDCA 標準フローでは `make deploy-all` がここを step 4-5 として内包する**。本 STEP は手動で plan → apply を分けたいとき用。

```bash
make tf-init                                                      # バケット存在 + provider (google / time / kubernetes / helm) を preflight
make tf-plan                                                      # GITHUB_REPO / ONCALL_EMAIL は env/config/setting.yaml が既定値
terraform -chdir=infra/terraform/environments/dev apply tfplan    # 保存済み plan を apply
```

別 GitHub repo / 別 oncall を試す場合のみ env で上書き:

```bash
make tf-plan GITHUB_REPO=other/repo ONCALL_EMAIL=oncall@example.com
```

ポイント:
- `GITHUB_REPO` は **GitHub リポジトリのパス**。WIF の `attribute.repository` 制約に焼き込まれる — 間違えると CI が auth できなくなる。setting.yaml の `github_repo` が canonical。
- `ONCALL_EMAIL` は monitoring alert policy の通知先。`@` を含まないと `variables.tf` + `scripts/setup/tf_plan.py` の validation で reject される。setting.yaml の `oncall_email` が canonical。plan 段階では通知送信されないので、練習用途なら自身の address でよい。
- `terraform -chdir=... apply` を `-var` 無しで直接叩くと対話プロンプトで vars を再入力させられ、`yes` 確認と混同しやすい。**必ず plan を `-out=tfplan` で保存して `apply tfplan` する** (Makefile の `tf-plan` が `-out=tfplan` を付けている)。
- オフラインで syntax チェックだけしたい場合は `make tf-validate` (バックエンド接続なし)。
- `make tf-plan` を **backslash 改行で書かない** (貼り付け時に trailing whitespace が混入すると line continuation が壊れる)。**1 行で叩く** のが安全。
- **2 段階 apply が必要な場面**: 完全ゼロから (GKE Autopilot cluster 未作成で) 一発 apply すると `kubernetes` / `helm` provider が cluster 未作成時点で activate されて失敗することがある。その場合は一度 `terraform apply -target=module.iam -target=module.data -target=module.gke -auto-approve` で cluster を先行作成してから、`terraform apply tfplan` を再度回す。`provider.tf` にも同じ手順がコメントで載っている。
- 初回 apply の所要時間は ~8-12 分 (GKE Autopilot cluster 作成 + KServe Helm chart apply + `time_sleep.wait_for_log_metric_indexing` 90s 含む)。
- 初回は search-api Deployment が placeholder image (例: `gcr.io/cloudrun/hello`) で起動 (Artifact Registry が空のため)。real image は `make deploy-api` で差し替える (PDCA 標準フローでは `deploy-all` が step 7 として連続実行)。
- encoder / reranker の KServe InferenceService も初回は `storageUri = "gs://.../placeholder"` のまま PENDING。Vertex AI Pipelines (`make ops-train-now` や pipeline schedule) が Model Registry に `production` alias を付けた後、**別途** `make deploy-kserve-models` で storageUri を `kubectl patch` する。
- 過去に踏んだ初回 apply のハマりは **§1.6 の表にまとめて記載**。再発時はそちらを参照。

### STEP 10: Secret Manager に値を投入

Terraform は secret container (`meili-master-key`, `search-api-iap-oauth-client-secret` on GCP Secret Manager) まで作る。本 STEP は値の投入を扱い、GKE 側の K8s Secret は External Secrets Operator が同期する。

Phase 7 では **Meilisearch は Cloud Run のまま**、**search-api は GKE** に移行したので、マスターキーは 2 ヶ所から参照される:

- **Cloud Run `meili-search`**: `--set-secrets MEILI_MASTER_KEY=meili-master-key:latest` で Secret Manager を直接参照
- **GKE `search/search-api` Pod**: External Secrets Operator が `search/meili-master-key` を自動生成し、それを Pod env から参照
- **IAP を有効化する場合**: `search-api-iap-oauth-client-secret` を Secret Manager に投入すると、`search/search-api-iap-oauth-client-secret` に自動同期される

責務分担:

- **Secret Manager**: 本番の正本。実値を持つ
- **`env/config/setting.yaml`**: secret そのものは置かず、`meili_master_key_secret_id` のような **参照情報だけ** を置く
- **`env/secret/credential.yaml`**: local 開発だけで使う override。必要な場合のみ実値を置く

```bash
export MEILI_KEY="<Meilisearch のマスターキー>"

# (1) GCP Secret Manager に投入 (meili-search Cloud Run 用)
echo -n "$MEILI_KEY" | \
  gcloud secrets versions add meili-master-key --data-file=-

# (2) sync 状態を確認 (External Secrets Operator 経由)
make kube-creds
kubectl -n search get externalsecret meili-master-key
kubectl -n search get secret meili-master-key

# (3) IAP を有効化する場合のみ: OAuth client secret を投入
echo -n "<oauth-client-secret>" | \
  gcloud secrets versions add search-api-iap-oauth-client-secret --data-file=-
kubectl -n search get externalsecret search-api-iap-oauth-client-secret
```

マスターキーは任意の文字列 (16 文字以上推奨)。`terraform apply` と `make apply-manifests` 後に値を投入し、ExternalSecret の同期を確認してから `make deploy-api` で search-api Pod を rollout すること。手順詳細は `env/secret/README.md`。

キーを変更する場合は Meilisearch のデータベースを初期化する必要がある (meili-data GCS バケットを消去 → STEP 16b で再投入)。

### STEP 15: 初回 embedding バッチ実行 (Vertex AI Pipelines)

Phase 7 では embedding batch も Vertex AI Pipelines (`property-search-embed`) 経由。`make ops-pipeline-run` で 1 回回して `feature_mart.property_embeddings` を埋める。初回は全件 encode するので物件数 × ~50ms/件。

```bash
# Vertex AI Pipelines (property-search-embed) を手動 submit
make ops-pipeline-run TARGET=embed

# 結果確認 (property_id の件数 = raw.properties 件数、embedding 次元 = 768)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT COUNT(*) AS n, AVG(ARRAY_LENGTH(embedding)) AS dim
  FROM `mlops-dev-a.feature_mart.property_embeddings`'
```

`dim = 768` / `n > 0` が期待値。`dim` が異なる場合は encoder の checkpoint (multilingual-e5-base) が取り違っている — pipeline パラメータの `encoder_model` を確認する。

**所要時間**: pipeline 起動オーバーヘッド ~1-2 分 + 物件数 × ~50ms/件。50 物件なら ~2 分、1,000 物件 ~3 分、10,000 物件 ~10 分。進捗は `Vertex AI Console → Pipelines` で確認できる (make target は submit のみで wait しない)。

### STEP 16: VECTOR INDEX 作成 (手動 SQL)

`google_bigquery_vector_index` リソースは 2026-04 時点の Terraform google provider では未対応なので **BQ コンソールまたは bq CLI で手動作成**。小データでは INDEX 無しでも `VECTOR_SEARCH(...)` は brute-force で動くが、物件 10,000 件超で必須:

```bash
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  CREATE OR REPLACE VECTOR INDEX property_embeddings_idx
  ON `mlops-dev-a.feature_mart.property_embeddings`(embedding)
  OPTIONS(
    index_type = "IVF",
    distance_type = "COSINE"
  )
'
```

作成には数分かかる (物件 10k で ~2-5 分、100k で ~10-20 分)。進捗は `INFORMATION_SCHEMA.VECTOR_INDEXES`:

```bash
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT index_name, coverage_percentage, last_refresh_time
  FROM `mlops-dev-a.feature_mart.INFORMATION_SCHEMA.VECTOR_INDEXES`'
```

`coverage_percentage` が 100 に到達したら疎通確認へ。`property_embeddings` を更新するたびに BQ が incremental refresh する (手動再実行不要)。

### STEP 16b: Meilisearch index 同期 (手動初期化)

### STEP 17: `/search` `/feedback` ステージング疎通 (完了ゲート)

Phase 7 疎通条件の 5 項目を逐一確認する。**rerank 無し** で `final_rank == lexical_rank` が返ることが合格ライン (booster はまだ load しない):

```bash
# search-api の ConfigMap を更新して rolling restart
# Phase 7 は GKE Deployment なので `kubectl set env` で env を注入し、Pod を restart する。
# 所要時間 ~1-2 分: ConfigMap / env 差し替え + rolling restart + encoder 層ロード。
# encoder 本体は KServe InferenceService 側で常駐しているので、search-api の再起動自体は軽い。
kubectl -n search set env deployment/search-api \
  ENABLE_SEARCH=true \
  KSERVE_ENCODER_URL="http://property-encoder.kserve-inference.svc.cluster.local/v1/models/property-encoder:predict" \
  KSERVE_RERANKER_URL="http://property-reranker.kserve-inference.svc.cluster.local/v2/models/property-reranker/infer"
kubectl -n search rollout status deployment/search-api

# 1) /search 200 応答 + 上位 20 件返却
URL=$(make -s ops-api-url)
TOKEN=$(gcloud auth print-identity-token)
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"赤羽駅徒歩10分 ペット可","filters":{"max_rent":150000},"top_k":20}' \
  | jq '.results | length'
# → 20 (上限)

# 2-3) final_rank == lexical_rank / score=null (rerank 未配線)
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"新宿駅","top_k":5}' \
  | jq '.results[] | {property_id, lexical_rank, final_rank, score}'
# → すべての行で lexical_rank == final_rank、score は null

# 4) ranking_log に 1 リクエストあたり候補 100 件分の行が流れる
# (Pub/Sub → BQ Subscription 経由。反映まで数分)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT request_id, COUNT(*) AS candidates
  FROM `mlops-dev-a.mlops.ranking_log`
  WHERE ts >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 10 MINUTE)
  GROUP BY request_id
  ORDER BY candidates DESC LIMIT 5'
# → candidates = 100 (pool sizeと一致)

# 5) /feedback publish 疎通
REQ_ID=$(curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"品川駅","top_k":1}' | jq -r .request_id)
PROP_ID=$(curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"品川駅","top_k":1}' | jq -r '.results[0].property_id')
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/feedback" -H 'content-type: application/json' \
  -d "{\"request_id\":\"$REQ_ID\",\"property_id\":\"$PROP_ID\",\"action\":\"click\"}"
# → {"accepted": true}

# feedback_events への反映 (数分)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT COUNT(*) AS n
  FROM `mlops-dev-a.mlops.feedback_events`
  WHERE ts >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 10 MINUTE)'
# → n >= 1
```

5 項目すべて PASS でステージング疎通完了。以降は Phase 5+ の rerank 有効化設定 (`/search` で `score` を返す経路) に進む。トラブル時は §4.7 を参照。

### 3.5 シークレットローテーション

Terraform は secret container + IAM のみ管理。値のローテは make 対象外 (秘密値を Make の引数に取らない):

# GCP Secret Manager

## 4. インシデント対応

### 4.1 search-api 5xx > 1% アラート発火

1. ログ確認: Cloud Logging で `resource.type="k8s_container" resource.labels.namespace_name="search" resource.labels.container_name="search-api" severity>=ERROR`
2. 直近デプロイ確認: `kubectl -n search rollout history deployment/search-api`
3. ロールバック: §5 ロールバック早見表の「search-api イメージ」を実行
4. Pub/Sub publish エラーなら API は 200 を返すが `ranking_log` / `feedback_events` 行数が減る — `make ops-search-volume` と `bq query ... mlops.ranking_log` で追跡
5. KServe 側の失敗なら `kubectl -n kserve-inference logs -l component=predictor --tail=200` で encoder / reranker predictor のログを見る

### 4.2 p95 レイテンシ > 500ms

- `jsonPayload.latency_ms` が大きい request_id を特定
- 原因切り分け: KServe predictor が詰まっているか (`kubectl -n kserve-inference top pods`) / search-api CPU 飽和か (`kubectl -n search top pods`) / BQ VECTOR_SEARCH が遅いか (`INFORMATION_SCHEMA.JOBS_BY_PROJECT` で query 時間)
- search-api CPU 飽和なら HPA が自動で Pod 増。超えても治らない場合は `deployment.yaml` の `limits` を一時拡張 (コスト影響あり、復旧後戻す)
- KServe predictor `minReplicas=1` を 0 に落とさない (コールドスタートで model load に数秒〜十数秒)

### 4.3 スキュー検出 FAIL (`status = 'FAIL'`)

- どの特徴量か: `make ops-skew-latest` で `feature_name` をチェック
- 推論側ソース (呼び出し元) が変わっていないか → API アクセスログと照合
- 学習分布が古い場合は `make ops-check-retrain` → 条件成立で自動キック / 手動なら `make ops-train-now`
- **しきい値を引き上げる前に必ず原因を特定**

### 4.4 再学習が走らない / 走り過ぎる

- `make ops-check-retrain` を叩き `reasons` を確認
- 閾値は `app/services/retrain_policy.py` の `NEW_FEEDBACK_ROWS_THRESHOLD=10_000` / `NDCG_DEGRADATION=0.03` / `STALE_DAYS=7`
- Cloud Scheduler: `gcloud scheduler jobs describe check-retrain-daily --location asia-northeast1`
- Eventarc 配送: Cloud Logging で `resource.type="eventarc.googleapis.com/Trigger"` を絞る
- Vertex AI Pipelines の submit 失敗: `Vertex AI Console → Pipelines` で FAILED run の error を確認

### 4.5 Dataform assertion 失敗

- `feature_mart_assertions.property_features_quality` の行を目視
- NULL 率・範囲外はだいたい生データ (`raw.properties`) の upstream 問題
- `property_features_freshness` 失敗 = 過去 2 日分のデータ欠損 → ETL 側を確認

### 4.6 ranker booster ロード失敗 (rerank 有効化時)

- `/livez` が 200 だが `/search` の `score` が null 固定: KServe `property-reranker` InferenceService の `storageUri` が placeholder のまま or `production` alias 未付与 — `make ops-promote-reranker VERSION=vN APPLY=1` → `make deploy-kserve-models` → `kubectl -n kserve-inference get inferenceservice property-reranker` で READY=True を確認
- rerank を有効化する前は `/search` が `score=null` / `final_rank == lexical_rank` を返すのが正 (Phase 4/5 系と同じ挙動)
- `mlops.training_runs` に `finished_at IS NOT NULL` の行が無い場合はそもそも学習ができていない → `make ops-train-now`

### 4.7 `/search` が 503 / 200 空配列

- `/readyz` → `search_enabled: false` の場合: `ENABLE_SEARCH=true` と `KSERVE_ENCODER_URL` / `KSERVE_RERANKER_URL` が Pod env に入っているか確認 (`kubectl -n search get deployment/search-api -o yaml | grep -A1 env`)
- `/readyz` → `search_enabled: true` なのに `/search` が 503: KServe encoder predictor が READY になっていない (`kubectl -n kserve-inference get inferenceservice`)、あるいは cluster-local DNS 解決失敗 (NetworkPolicy が search → kserve-inference を block していないか確認 — `infra/manifests/policies/{search-api-networkpolicy,kserve-networkpolicy}.yaml`)
- `/search` が 200 だが `results = []`: (a) `property_embeddings` が空 (STEP 15 を再実行) / (b) VECTOR INDEX が未作成で brute-force でも top_k=0 を返した (サンプルデータが少なすぎる) / (c) フィルタで全部落ちた (`max_rent` 等を外して再実行)
- `ranking_log` に行が流れない: Pub/Sub topic `ranking-log` と BQ Subscription が Terraform 側で作成済みか確認。`gcloud pubsub subscriptions list --filter="topic ~ ranking-log"` で Subscription の `bigquery_config.table` が `mlops.ranking_log` を指しているか

### 4.8 VECTOR INDEX が効かない / coverage が 100% 未満のまま

- `INFORMATION_SCHEMA.VECTOR_INDEXES` で `coverage_percentage` を確認。数時間経っても 100% に満たない場合は `property_embeddings` が絶え間なく更新されている可能性 (Vertex AI Pipeline `property-search-embed` スケジュールを一時停止し、INDEX 作成を完了させる)
- `coverage_percentage = 0` のまま: INDEX 作成クエリ自体が失敗している。`INFORMATION_SCHEMA.JOBS` で DDL ジョブのエラーを確認
- 応急: `VECTOR_SEARCH` は INDEX 無しでも動く (brute-force)。物件 10,000 件未満なら体感差は小さい

### 4.9 KServe InferenceService が READY にならない

`kubectl -n kserve-inference get inferenceservice` で `READY=False` が続く場合:

```bash
# (1) predictor Pod の状態
kubectl -n kserve-inference get pods -l serving.kserve.io/inferenceservice=property-encoder
kubectl -n kserve-inference describe inferenceservice property-encoder

# (2) storage access (Model Registry artifact GCS への Workload Identity)
kubectl -n kserve-inference logs -l serving.kserve.io/inferenceservice=property-encoder \
  -c storage-initializer --tail=100

# (3) predictor ランタイムログ
kubectl -n kserve-inference logs -l serving.kserve.io/inferenceservice=property-encoder \
  -c kserve-container --tail=200
```

典型原因:
- `storageUri` が placeholder のまま (Model Registry に production alias 未付与) → `make deploy-kserve-models`
- KSA (`kserve-inference/encoder` / `reranker`) が Workload Identity で GCS 読み取り権限を持っていない → `infra/terraform/modules/iam/main.tf` で `roles/storage.objectViewer` on `*-models` バケットが付いているか確認
- multilingual-e5 の image pull が重い → 初回のみ ~3-5 分待つ。`describe` で `ImagePullBackOff` が続くならレジストリ / ネットワーク側を疑う
