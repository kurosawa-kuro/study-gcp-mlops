# 04_検証.md — Phase 7 (GKE + KServe) 検証ガイド

> 注記: 「**何を見れば 'OK' とみなしてよいか**」を集約したドキュメント。`docs/05_運用.md` (PDCA / 個別 STEP / 定常運用) とは性質が異なる:
>
> - `05_運用.md` = **やる** (`make deploy-all` を回す手順、定常監視、インシデント対応)
> - `04_検証.md` = **OK か判定する** (ローカル / GCP 検証ゲート、退化検知、recovery)
>
> 順序関係: **本書 → `05_運用.md`**。変更後は本書の §2 (ローカル) → §3 (GCP ゲート、特に G3 = 3 種 all non-zero) を全部 PASS させる。検証ゲートを通った run のみが `05_運用.md §3` の定常運用に進む資格を持つ。途中でエラーが出たら §4 (一過性エラー) か `05_運用.md §1.6` (恒久対処済ハマり) に該当するか判定する。
>

## 1. 検証の考え方

PDCA を回す前 / 回しきった後に「何を見れば 'OK' とみなしてよいか」の checklist。**ローカルで早く落とせる失敗はローカルで落とし、cluster 依存は GCP 側で確認する**。

階層別の責務:

| 層 | 何が確認できる | 所要時間 | 認証 |
|---|---|---|---|
| ローカル (§2) | import / boot / 静的検証 / image 構成 | 1-5 分 | ADC のみ (任意) |
| GCP 検証ゲート (§3) | 3 種が all non-zero、NDCG@10、Gateway TLS、KServe Workload Identity | 3 分 (deploy 後) | フル GCP |
| 退化検知 (§5) | 性能ベースライン (build 時間、image size 等) との乖離 | 数分 | ローカル + GCP |

ローカルで弾けるはずの失敗が GCP に持ち込まれると、deploy-all を 1 回ぶん (~5 分) と Cloud Build コストが無駄になる → **§2 を先に通す**。

## 2. ローカル検証 (GCP 認証ゼロでも通せる範囲)

| # | 種類 | コマンド | OK 基準 | 落ちたら見る場所 |
|---|---|---|---|---|
| L1 | 静的検証 | `make check` (ruff / fmt / mypy / pytest) | `449 passed` | 直前の編集ファイル。新規 module は `scripts/ci/layers.py::DIRECTORY_RULES` に追記したか確認 |
| L1' | レイヤ境界 | `make check-layers` | `check-layers: OK (51 files clean)` | Port が adapter / SDK を import していないか (`scripts/ci/layers.py`) |
| L2 | image build | `DOCKER_BUILDKIT=1 docker build -f infra/run/services/search_api/Dockerfile -t local-test/search-api:dev .` | image size **~700MB** (3GB を超えていたら `[ml]` extra が混入している) | `pyproject.toml` の `[project.optional-dependencies]` から `lightgbm` / `sentence-transformers` / `kfp` が漏れて base に戻っていないか |
| L3 | image 起動 (import 検証) | `docker run --rm --entrypoint=python local-test/search-api:dev -c "from app.main import app; import scripts._common; import scripts.ops.destroy_check"` | エラーなく完了 | `Dockerfile` で `scripts/` の COPY が抜けていると `ModuleNotFoundError: scripts` で落ちる (Phase 7 Run 5 の事故ケース、現在は対処済) |
| L4 | container 起動 (`/livez`) | `docker run --rm -d -p 18080:8080 -e ENABLE_SEARCH=false -e PROJECT_ID=mlops-dev-a -e REGION=asia-northeast1 -e GOOGLE_APPLICATION_CREDENTIALS=/adc/adc.json -v /tmp/adc.json:/adc/adc.json:ro local-test/search-api:dev` → `curl localhost:18080/livez` | `{"status":"ok"} HTTP 200` | ADC を `chmod 644 /tmp/adc.json` で読めるようにしてから mount (uid 1001 の `app` user) |
| L5 | ハイブリッド検索 3 種 (lexical / semantic / rerank) のローカル統合 | (実施しない) | — | **ローカル単体では不可** — Meilisearch / BigQuery / KServe いずれも GCP / cluster 依存。`make ops-search-components` で実機検証する (§3 G3 参照) |

L5 が「ローカルだけで完結しない」のは設計上の制約。Phase 6 (Cloud Run 単体) は in-process reranker でローカル統合できたが、Phase 7 は serving 層を KServe に移譲した結果、reranker が cluster-side HTTP 呼び出しに変わった。意図的なトレードオフ:

- **ローカルで早く回せるもの**: import / boot / 静的検証
- **実機でしか回せないもの**: 3 種が all non-zero、NDCG@10、Gateway TLS、KServe Workload Identity

## 3. GCP 検証サイクル (PDCA = `05_運用.md §1.2` + 検証ゲート)

`make deploy-all` 完了後 → `make run-all-core` の各 step が以下のゲートになる:

| # | step | コマンド | OK 基準 | 失敗時のメンタル分岐 |
|---|---|---|---|---|
| G1 | レイヤ境界 (再確認) | `make check-layers` | OK | (ローカルで弾けるはず — §2 L1' 参照) |
| G2 | seed | `make seed-test` | `5 properties materialised` | BQ table がまだ deletion_protection で守られているか、SA に `bigquery.dataEditor` が無い |
| G2' | meili sync | `make sync-meili` | `meili.tasks` API が 200 | Meilisearch Cloud Run の `meili-master-key` 未投入。`05_運用.md §1.6` 参照 |
| G3 | **3 種コンポーネント (load-bearing)** | `make ops-search-components` | `[component-check] hits lexical=N1 semantic=N2 rerank=N3` で **全て non-zero** | `lexical=0` → Meilisearch index 未同期 (G2' 確認)。`semantic=0` → BQ `property_embeddings` が空 / VECTOR INDEX 未作成 (`05_運用.md §2 STEP 16`)。`rerank=0` → KServe `property-reranker` が `production` alias 未設定、または `make deploy-kserve-models` 未実行 |
| G4 | retrain pipeline | `make ops-train-now` → `ops-pipeline-run` | `PIPELINE_STATE_SUCCEEDED` (~5-10 分) | Vertex AI Pipeline 履歴 (`make ops-vertex-pipeline-status`) で個別 step エラーを確認 |
| G5 | feedback / ranking smoke | `make ops-feedback` / `ops-ranking` | `feedback_accepted: true`、ranking JSON に `lexical_rank` / `semantic_rank` / `score` が並ぶ | search-api pod log (`kubectl logs -n search deployment/search-api --tail=50`) |
| G6 | 精度 | `make ops-accuracy-report` | `ndcg_at_10 >= 1.0` (5 件 seed では満点) | seed が想定通り入ったか (G2)、ranking 関数の degradation はないか |

**G3 が最重要ゲート**: ハイブリッド検索の 3 構成要素 (BM25 / BQ VECTOR_SEARCH / LightGBM 再ランク) が全て寄与している証拠。ここが落ちたら**他のメトリクスがどれだけ良くてもその run は使えない** (片肺飛行)。

## 4. 検証中に出やすい一過性エラーと recovery

PDCA で実測したエラー (恒久対処済ではなく、**遭遇したらリトライ手順を踏む**もの)。コード修正済の永続的ハマりは `05_運用.md §1.6` を参照。

| 症状 | 発生フェーズ | 一次原因 | recovery |
|---|---|---|---|
| `Error waiting to create Service: ... container failed to start and listen on PORT=8080` (Meilisearch Cloud Run) | `terraform apply` 中 | 既存 service が部分作成状態 (cold-start race / image pull 失敗) | `gcloud run services delete meili-search --region=asia-northeast1 --quiet` で消してから `make deploy-all` 再実行 (Terraform が冪等で再作成) |
| `Error trying to delete bucket mlops-dev-a-pipeline-root containing objects without 'force_destroy' set to true` | `make destroy-all` 終盤 | run-all 中の Vertex Pipeline が `gs://...-pipeline-root/runs/<job-id>/` に artifact を書き込んだ後、destroy-all stage [3/6] の bucket wipe 後に新規 artifact が来た | `gcloud storage rm -r gs://mlops-dev-a-pipeline-root/runs/` → `make destroy-all` 再実行 |
| `URLError: Connection reset by peer` on `make ops-livez` | `make run-all-core` 開始直後 | Gateway PROGRAMMED 完了前 / managed cert 発行中 | 60 秒待機 → `kubectl get gateway -n search` で `PROGRAMMED: True` を確認してから再実行 |
| Pod が `CrashLoopBackOff: ModuleNotFoundError: No module named 'scripts'` | rollout 直後 | search-api Dockerfile が `COPY scripts/` を欠いている (Phase 7 Run 5 で実害発生) | `infra/run/services/search_api/Dockerfile` の `COPY scripts/ scripts/` と runtime stage の `COPY --from=builder /src/scripts /home/app/scripts` が両方あるか確認 (現在の HEAD は対処済) |

**運用方針**: いずれも「同じコマンドをもう一度叩けば通る」もの。手動修復ループに入ったら `destroy-all` でクリーンスレートに戻すのが PDCA の trade-off。

## 5. 性能ベースライン (実測値、退化検知用)

最適化済 PDCA 1 サイクルの実測値 (2026-04-29、`mlops-dev-a` / `asia-northeast1`、WSL2):

| 指標 | ベースライン | 退化検知の目安 |
|---|---|---|
| `make destroy-all` (state 空) | <1s | — (空に対する no-op) |
| `make destroy-all` (フル稼働状態から) | ~10 分 | 15 分超 → namespace finalizer 詰まりを `kubectl get servicenetworkendpointgroups -A` で確認 |
| `make deploy-all` 全体 | **~5 分** (kaniko cache hit + 軽量 image) | 10 分超 → kaniko cache が効いていない (`gcr.io/kaniko-project/executor:v1.20.0` と `--cache=true` が `cloudbuild.yaml` に居るか) |
| Cloud Build (search-api、kaniko cache hit) | **~3 分** | 6 分超 → image に重い依存 (torch / kfp) が混入していないか `docker history` で確認 |
| kubectl rollout (search-api) | **~1 分** | 3 分超 → image pull が遅い (image size が大きい)。`docker manifest inspect <image>` で `sizeSum` を確認 |
| search-api image | **~705MB** (compressed) | 1.5GB 超 → `pyproject.toml` の base `dependencies` に `[ml]` / `[pipelines]` が混入していないか |
| `make run-all-core` 全体 | ~3 分 | — (実機 API 呼び出しがほとんどなので GCP latency 依存) |
| pytest (`make test`) | 449 passed in ~5s | 失敗時は新規依存 import を `pyproject.toml` に追加するか、test 側で skip マーカー |

**ベースライン退化を疑うべき編集**:
- `pyproject.toml` の `dependencies` を増やした (特に `torch` / `tensorflow` / `kfp` 系)
- `Dockerfile` のレイヤ順を変えた (`COPY app/` を上に動かすと cache hit 率が崩れる)
- `cloudbuild.yaml` から kaniko を docker builder に戻した

## 6. 検証依頼プロンプト例 (Claude Code 向け)

「どの粒度で検証してほしいか」を明示すると、過不足のない作業範囲で動かせる。コピペで使える例:

### 6.1 ローカルだけ (1-5 分、GCP 認証不要)

```
ローカル検証だけ依頼。L1+L2 (make check + image build)
```
→ `make check-layers` → `make check` → `docker build` を回し、image size と test 結果を報告。

```
ローカル起動チェック依頼 (L4 まで)
```
→ 上に加えて `docker run` で `/livez` 200 確認まで。

### 6.2 単発 GCP 検証 (cluster は既に稼働中)

```
deploy 済みの cluster で 3 種 (lexical/semantic/rerank) チェック依頼。NDCG も
```
→ `make ops-search-components` + `make ops-accuracy-report` を叩いて結果報告。失敗時は §3 G3 の分岐表に従って原因切り分け。

```
ops-search-components が rerank=0 になる原因切り分け依頼
```
→ KServe `production` alias / `make deploy-kserve-models` 状況 / pod log の順で診断。

### 6.3 フル PDCA (~30 分、cluster 全消し→再構築→検証→消す)

```
destroy-all, deploy-all, run-all, destroy-all の PDCA サイクル検証依頼
```
→ 4 step を順に実行、各 phase の elapsed と 3 種ゲート (G3) 結果を報告。

```
deploy-all + run-all-core だけ依頼 (destroy はしない、cluster 残す)
```
→ コストが残るため明示要求時のみ。run-all-core 完了後に「destroy しますか?」と確認。

### 6.4 変更後の影響確認 (退化検知)

```
Dockerfile の COPY 順を変えた。L1+L2+G3 で退化検知依頼。
image size と build 時間が §5 のベースラインと乖離していないか確認
```
→ ローカル build → ベースライン照合 (705MB / kaniko cache hit ~3 分等) → 必要なら deploy して G3 まで。

```
pyproject.toml の dependencies を増やした。重い依存が search-api image に
混入していないか確認依頼
```
→ ローカル build → `docker run --entrypoint=python ... -c "import torch, lightgbm, kfp"` が全て ImportError であることを検証 (§5 の退化検知項目)。

### 6.5 一過性エラー recovery

```
make deploy-all が Meilisearch Cloud Run health check で失敗。recovery 依頼
```
→ §4 の一次手順 (`gcloud run services delete meili-search` → `make deploy-all` 再実行)。

```
make destroy-all が pipeline-root bucket force_destroy エラー。クリーンアップ依頼
```
→ `gcloud storage rm -r gs://mlops-dev-a-pipeline-root/runs/` → `make destroy-all` 再実行。

### 6.6 計測 / ベースライン更新

```
今回の最適化で deploy-all 時間がどれだけ縮んだか計測依頼。
§5 のベースラインを実測値で更新
```
→ destroy → deploy → 各 phase elapsed 抽出 → 表を更新する PR を起こす。
