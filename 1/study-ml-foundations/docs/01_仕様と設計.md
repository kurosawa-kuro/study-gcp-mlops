## 0. 位置付け (Phase 1: ML 基礎の最少装備)

Phase 1 は親 [`README.md` §4 コード化方針](../../../README.md) における **コード化対象の最末端 = 最少装備** の phase。**ML 基礎 (preprocess / training / evaluation / artifact 保存) に集中** し、API・設計パターン・検索ドメイン・GCP 系は持ち込まない。

題材は **California Housing (回帰)**。Phase 3 以降の不動産ハイブリッド検索とは別ドメイン (Phase 1-2 = ML 基礎、Phase 3 以降で検索ドメインへ展開)。

## 1. Phase 1 の守備範囲

### 含むもの

- データ投入 (sklearn California Housing → PostgreSQL)
- 前処理 (欠損補完 / 外れ値キャップ / 対数変換)
- 特徴量エンジニアリング (BedroomRatio / RoomsPerPerson)
- 学習 (LightGBM 回帰)
- 評価 (RMSE / R² を numpy で自前実装)
- アーティファクト保存 (`model.lgb` / `metrics.json` / `params.yaml` / `runs/{run_id}/`)
- 実験ログ (`runs/{run_id}/` + git commit hash)
- 構造化ロギング

### 含まないもの (Phase 2 以降 / Phase 7 完成版の責務)

- 推論 API / FastAPI / DI / lifespan → **Phase 2** で導入 (コード化)
- Port/Adapter 構造 → **Phase 2** で導入 (コード化)
- ハイブリッド検索 (Meilisearch / multilingual-e5 / RRF / LightGBM LambdaRank) → **Phase 3** で導入 (コード化)
- GCP マネージドサービス (Cloud Run / BigQuery / GCS / Secret Manager / Terraform / WIF / 軽量 retrain trigger) → **Phase 4** で導入 (コード化、Phase 3 からの足し算)
- Vertex AI 本番 MLOps 基盤 (Pipelines / Endpoint / Model Registry / Monitoring / Feature Store / Vector Search) → **Phase 5 (論理 Phase)** で資料化 (実装は Phase 7 完成版コード)
- Cloud Composer / PMLE 追加技術 (BQML / Dataflow / TreeSHAP / SLO + burn-rate) → **Phase 6 (論理 Phase)** で資料化 (実装は Phase 7 完成版コード)
- GKE / KServe → **Phase 7 (canonical = 教材コード完成版)** で本実装

## 2. アーキテクチャ (Phase 1 単独)

```text
PostgreSQL
  → pipeline/data_job        (sklearn → PostgreSQL 投入)
  → pipeline/training_job    (LightGBM 学習)
      → ml/data/preprocess
      → ml/data/feature_engineering
      → ml/training
      → ml/evaluation
      → ml/registry/artifacts/{run_id}
  → pipeline/evaluation_job  (評価単発)
```

**Key design decisions**:

- **Docker Compose 前提** — seed / train は `docker compose` 経由で実行
- **Repository pattern** — `PostgresRepository` (SQLAlchemy + psycopg)
- **No scikit-learn for metrics** — RMSE / R² は numpy で自前実装 (`ml/evaluation/metrics/regression.py`)
- **Run ID** — `YYYYMMDD_HHMMSS_{6桁UUID}` でモデルにバージョン付与
- **`latest` シンボリックリンク** — `ml/registry/artifacts/latest` で最新参照

---

## 3. モジュール責務

- `ml/common/` — Phase 1 内で閉じる共通基盤 (設定 / ロギング / schema / run_id)。Phase 2 とは import 共有しない (教材としての phase 独立性を優先)
- `ml/data/` — `loaders` (PostgreSQL 取得) / `preprocess` / `feature_engineering`
- `ml/training/` — LightGBM 学習
- `ml/evaluation/` — RMSE / R² 算出、metrics.json 保存
- `ml/registry/` — 学習成果物の保存先管理
- `pipeline/` — `data_job` / `training_job` / `evaluation_job` の 3 ジョブ。**推論系ジョブは Phase 2 に移管済**

---

## 4. 設定 (pydantic-settings)

`ml/common/config/base.py` (`BaseAppSettings`) を利用し、優先順位:

1. 環境変数
2. `env/secret/credential.yaml`
3. `env/config/setting.yaml`
4. コード上の既定値

主要キー: `postgres_host` / `postgres_port` / `postgres_db` / `postgres_user` / `postgres_password` / `model_dir`

---

## 5. 不変ルール

- 題材は California Housing 回帰 (変えない)
- 学習器は LightGBM 回帰 (変えない)
- 評価は RMSE / R² (変えない)
- 成果物は `run_id` / `model.lgb` / `metrics.json` を残す
- API や設計パターンは持ち込まない (Phase 2 以降の責務)
- Phase 2 とコード共有 import は行わず、phase 独立を優先
