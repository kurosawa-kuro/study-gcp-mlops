# フルリファクタリング実施計画 (Phase 1 / `study-ml-foundations`)

現状コードは 4 トップレベル (`app/` + `common/` + `jobs/` + `pipelines/`) にそれぞれ独立した `src/<pkg>/` を持つ多重ソースルート構成になっている。一方ドキュメント (`docs/01_仕様と設計.md` / `docs/03_実装カタログ.md` / `docs/04_運用.md`) はいずれも `src/{ml,api,share}/` + `tests/{api,ml}/` + ルート直下の `Dockerfile.{api,trainer}` という単一ソースルート構成を「現状」と記載しており、**コードとドキュメントが食い違っている**。本ロードマップは docs 側で既に権威化されている構造にコードを合わせる整理であり、設計テーゼ (LightGBM 回帰 / 前処理 / 特徴量エンジニアリング / W&B / FastAPI lifespan / PostgreSQL) を一切変えない。

## 0. 背景 (なぜ動かすか)

### 現状の構造と docs の記載のずれ

このずれにより以下が発生している:

1. **新規参加者が docs を読んでコードを探せない** — docs の「`ml/pipeline/main.py`」を見て実ファイルを `src/ml/pipeline/main.py` で探しても存在せず、実際は `pipelines/src/pipelines/housing_prices/main.py` にある
2. **Phase 2 (`2/study-hybrid-search-local`) との構造対比が線形にならない** — Phase 2 も `app/` + `common/` + `jobs/` + `pipelines/` + `ml/{embed,train,sync}/` の多層構成 (hybrid-search 固有の複雑さを持つ)。教育教材として Phase 1 → Phase 2 の差分を示す際、「Phase 1 の単純な `src/{ml,api,share}/` が Phase 2 で `app/` + `common/` + `pipelines/` + `ml/{embed,train,sync}/` に細分化される」という語り方ができなくなっている
3. **`common` / `app` / `training` / `pipelines` というトップレベル Python 名前空間が一般名詞と衝突** — `import app` や `import common` は IDE 補完や 3rd party パッケージ (`app` は典型的な変数名) との衝突が起きやすい。`api` / `share` / `ml.trainer` / `ml.pipeline` のほうが衝突しない
4. **`.dockerignore` / `docs/04_運用.md` が既に新構造前提で書かれている** — `.dockerignore:7` の `src/ml/wandb/` 参照、`docs/04_運用.md:78` の「offline モードで `src/ml/wandb/` 配下にローカル保存」記述は新構造 (`src/ml/`) を前提にしている。W&B offline run の実際の保存先は現状 `jobs/src/training/wandb/` で、`scripts/clean.sh` がこれをハードコードしているため、docs 通りの保存先に揃える意味でも移行が必要

**やらないこと** (非スコープ):

- **設計テーゼを変えない** (`01_仕様と設計.md` §依存の方向 / 前処理 / 特徴量エンジニアリング / LightGBM パラメータ / モデルバージョニング / API エンドポイント / W&B offline 動作 は維持)
- **非負制約は変えない** (Python 3.10+ / Docker Compose / PostgreSQL 16 / `env/config/setting.yaml` + `env/secret/credential.yaml` 分割 / `scripts/core.sh::load_credentials` による shell export → compose 補間)
- **Python 依存関係 / requirements の形を変えない** — Phase 1 は `pip install` を Dockerfile 内部に直書きしている。Phase 3 の `uv` / Phase 2 の `requirements.txt` 一括管理といった別 phase の慣行は引き込まない
- **`common.config.BaseAppSettings` の潜在的 `_ROOT` 計算バグの修正はしない** — 現状 `common/src/common/config.py::_ROOT = Path(__file__).resolve().parents[2]` は解決すると `<repo>/common/` を指しており、`env/config/setting.yaml` / `env/secret/credential.yaml` が実際には読まれていない (ファイルが存在しないパスを見ているため silent no-op)。実設定は環境変数と field default で決定している。移行後の `src/share/config.py` では `parents[2]` が `<repo>/` を指して YAML が読まれるようになる **挙動変化がある**。§4 で対処を詳述
- **SQLite → PostgreSQL の docs 齟齬は触らない** — `docs/04_運用.md:18` の「SQLite に格納する」等の記述は実コード (PostgreSQL) と食い違っているが、本ロードマップはフォルダ構成揃えのスコープのみ。docs 本文の整合性修正は Commit #5 のついでに可能な範囲に限定 (新構造記載への置換は実施、DB 種別の記述は触らない)
- **Makefile target の rename はしない** — `make seed` / `make train` / `make serve` / `make test` / `make all` / `make down` / `make clean` はそのまま。内部 `docker compose` コマンドの service 名も変えない
- **Dockerfile の base image / 依存パッケージは変えない** — `python:3.10-slim` + `lightgbm` + `libgomp1` 等の構成は維持。COPY 対象パスと `PYTHONPATH` / `CMD` のみ更新
- **新規テスト追加はしない** — 移動に伴う import 書き換えのみ。既存 `app/tests/api/test_api.py` / `jobs/tests/test_trainer.py` / `jobs/tests/test_evaluation.py` / `pipelines/tests/test_pipeline.py` / `pipelines/tests/test_preprocess.py` を `tests/api/` / `tests/ml/` へ移動するだけ

## 5. 次のアクション提案

**Commit #1 (足場作り) から着手することを推奨する。** 破壊ゼロで既存動作に影響せず、後続 Commit のベースになる (ディレクトリと `__init__.py` と pytest 配線の 3 点のみ)。
