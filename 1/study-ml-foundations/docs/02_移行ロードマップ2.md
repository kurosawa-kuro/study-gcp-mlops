# フルリファクタリング実施計画 (Phase 2 / `study-hybrid-search-local`)

**実施済 (2026-04-21)** — 本ロードマップの Commit #1〜#5 は全て適用済。新規作業時は [`docs/03_実装カタログ.md`](03_実装カタログ.md) の現行構造を正とする。本ファイルは設計意図・移行経緯の履歴ドキュメントとして保持。

---

`jobs/src/jobs/` 配下に 6 サブパッケージ + `jobs/src/training/` に ML ユーティリティ 2 ファイル、という二重構造になっている現状を、ML コア (embedding / training / Meilisearch sync) だけ `ml/{embed,train,sync}/` に切り出して Phase 3 (`study-hybrid-search-cloud`) と構造 parity を取る計画。本ロードマップは `02_移行ロードマップ.md` の決定的仕様 (Meilisearch + ME5 + LightGBM LambdaRank 3 段 / KPI 定義 / DB スキーマ) に **従属** する整理であり、スコープを変えるものではない。

## 0. 背景 (なぜ動かすか)

現状 `jobs/` 配下は **2 つの独立したパッケージ** を同居させている:

| 実体 | パス | 役割 |
|---|---|---|
| `jobs/src/jobs/` (6 サブパッケージ) | `embeddings/`, `indexing/`, `features/`, `evaluation/`, `maintenance/`, `training/` | バッチ orchestration (Makefile `ops-*` から `docker-compose exec -T api python -m jobs.*` で起動) |
| `jobs/src/training/` (トップレベル名前空間) | `lgbm_trainer.py`, `training_dataset_builder.py` | LightGBM LambdaRank 学習ユーティリティ (Makefile `training-fit` / `training-generate` から `python -m training.*` で起動) |

このまま肥大化すると以下が崩れる:

1. **`training.*` ルート名前空間が `jobs.training.*` と衝突**: `jobs/src/jobs/training/run_weekly_retraining.py` (`import から training.lgbm_trainer import train_model`) と `jobs/src/training/lgbm_trainer.py` がトップレベル `training` パッケージを共有しており、将来 `jobs.training.*` を import するコードが増えたときに参照解決が曖昧になる
2. **Phase 3 との構造 parity がない**: Phase 3 は `ml/{embed,train,sync}/` で embedding / training / Meilisearch 同期を分離している。Phase 2 → Phase 3 の継承関係を線形に比較できるよう、ML コアの位置関係だけは揃えておくと教材としての見通しが良くなる
3. **Make target 名の整合性**: `search-sync` / `embeddings-generate` / `training-generate` / `training-fit` / `training-fit-safe` / `retrain-weekly` は Phase 3/4 で整理済の `ops-*` 接頭命名規則と揃っていない。CLAUDE.md の「Phase 3/4 と命名揃い」方針を ML コア系にも拡張する余地がある
4. **AST 境界テスト (`scripts/checks/layers.py::_rules_for_stage(4)`) の RULES が 2 系統 (`jobs` / `trainers`) に分裂**: 将来 embed と train の依存境界を片方だけ絞りたいときに書き分けがしづらい

**やらないこと** (非スコープ):

- 設計テーゼ (`01_仕様と設計.md` §1 / §3 / §4) は変えない — Meilisearch BM25 + ME5 cosine + LightGBM LambdaRank の 3 段構成 / `me5_score` を特徴量化する方針 / `group = request_id` / NDCG@10 / 特徴量 10 列 (`common.schema.feature_schema`) は維持
- 非負制約 (Python 3.11+ / **pip + ルート `requirements.txt`** / PostgreSQL 16 / Meilisearch v1.7 / Redis / `env/config/setting.yaml` + `env/secret/credential.yaml` 分割 / `scripts/setup/compose.sh` 経由の docker compose) は一切変えない
- **`ml/*/Dockerfile` は作らない** — Phase 3 と違い CI workflow も Cloud Run Job も無いため、単一 `app/Dockerfile` に `COPY ml ./ml` と PYTHONPATH 拡張を加えるだけで完結する
- **`ml/*/requirements.txt` / `ml/*/pyproject.toml` も作らない** — 非負制約 (ルート `requirements.txt` / `requirements-dev.txt` 一括管理) 維持。Phase 3 の uv workspace 概念は引き込まない
- **Cloud Build / Cloud Run / BigQuery / Vertex AI / GitHub Actions** は一切関与しない — クラウド化は Phase 3 以降 (`3/study-hybrid-search-cloud`, `4/study-hybrid-search-vertex`) の担当
- **RRF 導入は Phase 3** — 本 Phase では `me5_score` を LightGBM 特徴量に入れる方式を維持 (CLAUDE.md「紛らわしい点」記載方針)
- `jobs/src/jobs/features/`, `evaluation/`, `maintenance/` は **移動しない** — Phase 3 に対応概念が無いため parity を取る意味が薄く、無理に `ml/` 配下に入れると意味的な切り分けが曖昧になる。これらは `jobs/` に残す
- feature parity invariant — `common/src/common/schema/feature_schema.py::FEATURE_COLS_RANKER` や `pipelines/services/ranking/feature_service.py` の推論側特徴量生成は動かさない
