## 設計思想の不変性

- Phase 間のコードは原則共有しない (教材としての独立性を優先)
- ただし **設計思想 (Port/Adapter / 依存方向 / `core → ports ← adapters`) は一貫** させ、**adapter 実装だけ差し替える**
- 実行方式の段差 (Docker Compose → uv + クラウド → Vertex AI → GKE) は、同じ設計思想を保ったまま実行基盤だけを段階的に差し替える学習設計

## 共通非負制約

### 全 Phase 共通

- 教材対象外技術は導入しない (親 [`README.md` §1](../../README.md))
- 設計思想 (Port/Adapter / 依存方向 / `core → ports ← adapters`) を破壊しない
- Phase 間のコード共有や広域リファクタは原則行わない

### Phase 3 / 4 / 7 (ハイブリッド検索系コード化対象)

- ハイブリッド検索中核 5 要素を維持 (詳細は Phase 7 docs/01 §2.3)
- 検索品質改善はこの基本構成を前提に行う
- 置換・削減・無効化は事前の明示合意を必要とする

### Phase 4 (GCP MLOps 土台 = 独立コード)

- **Phase 7 完成版からの引き算ではなく、Phase 3 からの足し算で構築**
- 扱うのは Cloud Run / Cloud Run Jobs / BigQuery / GCS / Secret Manager / Cloud Logging / Artifact Registry / Cloud Build / Terraform / WIF / IAM / 軽量 retrain trigger まで
- **Vertex AI Pipelines / Feature Store / Vector Search / Cloud Composer / GKE / KServe / PMLE 追加技術は扱わない** (Phase 7 完成版に集約、Phase 5 / 6 で分解説明)

### Phase 5 / 6 (論理 Phase = コード保守なし)

- 個別コードを持たない。Phase 7 完成版コードを基準にして技術境界を資料化する
- 各 Phase スライド・ドキュメントでは Phase 7 配下の参照先を必ず明示する

### Phase 7 (到達ゴール = canonical 起点)

- Phase 5 / 6 の技術要素も Phase 7 配下に集約
- Cloud Composer 本線 + Vertex AI Pipelines + Feature Store + Vertex Vector Search + Model Registry + BigQuery + Meilisearch + GKE + KServe + PMLE 4 技術を本 phase で本実装
- ハイブリッド検索中核コードの挙動は絶対に変えない
- default feature flag では Phase 5 相当の挙動を維持 (新技術は opt-in)
- `make check` / parity invariant / Port/Adapter 境界検知 / WIF は追加コードも含めて継続して PASS させる

## 教材対象外 (全 Phase 禁止)
