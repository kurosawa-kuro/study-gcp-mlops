# 教育コード化についての仕様変更（改訂版）

## 1. 結論

本プロジェクトは、当初のように **Phase 1〜7 すべてを個別コードとして完全実装・保守する教材** ではなく、以下の方針に変更する。

> **コード化対象は Phase 1 / 2 / 3 / 4 / 7 に限定する。**
> **Phase 5 / 6 は、Phase 7 完成版コードを基準にした技術習得用の論理 Phase として扱う。**

これにより、教材としての理解しやすさと、コード保守コストのバランスを取る。

---

## 2. 仕様変更の背景

Phase 5 / 6 をすべて独立コードとして実装すると、以下の問題が出る。

| 問題         | 内容                                                                                |
| ---------- | --------------------------------------------------------------------------------- |
| 保守コストが高すぎる | Vertex / Composer / Feature Store / Vector Search の差分を複数 Phase で維持する必要がある |
| 差分説明が複雑になる | 「何を追加したか」より「何を削ったか」の説明が中心になりやすい                                                   |
| 教材として重くなる  | 受講者・チームメンバーが各 Phase の差分追跡に疲弊する                                                    |
| 完成形が見えにくい  | 中間 Phase が増えるほど、最終的な理想形コードへの導線がぼやける                                               |
| 実務コストに合わない | 個人学習・チーム内共有としては過剰実装になる                                                            |

一方で、Phase 4（GCP MLOps の土台）は他 Phase と性質が異なり、**Phase 5 / 6 / 7 の前提**として独立コード化する価値が高い。Phase 7 からの引き算ではなく、Phase 3 からの足し算（local → cloud）として自然にコード化できるためである。

そのため、教育資料としては **Phase 5 / 6 の技術習得主眼をスライド・ドキュメントで整理し、Phase 4 まではコードを持つ** 方針に変更する。

* KServe

# 12. 判断

この仕様変更は妥当。

理由は、

1. **Phase 4 は GCP MLOps の土台として独立コード化する価値が高い**（Phase 5 / 6 / 7 の前提となるクラウド基盤を、Phase 3 からの足し算で自然に学べる）
2. **Phase 5 / 6 の中間コードを作るよりも、Phase 7 の完成版を正本にして技術境界を説明した方が、教材としても実務資料としても強い**

特に今回の教材は、単なるコード写経ではなく、

```text
ML基礎
App設計
Local検索
GCP化（土台）
Vertex MLOps
PMLE技術
Composer orchestration
GKE / KServe serving
```

を段階的に理解させることが目的。

そのため、**土台までは独立コード、上位レイヤーは完成版コードを分解して理解する教材**として設計するのが最も合理的。
