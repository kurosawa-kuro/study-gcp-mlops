>
> 教育資料側もこの改訂に追従し、Phase 5 / 6 の教材は「Phase 7 完成版コードを参照しながら技術境界を理解する」構成へ作り直す。

- 既存の教育資料が、最新の Phase 仕様およびコード実態と乖離している

- コード読解を主目的にせず、**新技術の理解と学習導線の再設計** を優先する
- 実作業の前段として、現状調査と改修タスクの棚卸しを先に行う

補足:
- `README.md` は Phase 1〜8 の全体像と非負制約の把握に使う
- Phase 1〜6 は、仕様書側の更新がコードより先行している可能性があるため、**各 Phase の `README.md` / `CLAUDE.md` / `docs/` と実コードの突合が必要**
- Phase 7 は最終 Phase として、比較的コードと仕様が一致している前提で確認を始める

## 3. 教育資料の作成方針

### 3.1 主眼

教育資料の目的は **新しい技術を吸収してもらうこと** であり、コード詳細の完全理解そのものではない。

### 3.2 記載してよい内容

- ML の重要要素の該当ファイル・該当行の抜粋
- モデル定義 / 学習 / 評価 / 推論の要点整理
- パイプラインと ML・推論の起動関係のフロー
- モデル格納場所や成果物配置の説明
- 精度評価の見方と確認ポイント
- サンプルアプリの挙動説明
- 必要に応じたコード全体の要約説明

教育資料では、次のような **具体テーマ** を扱ってよい。

- **ML 基礎機能**
  - `LightGBM` の役割、回帰 / ランキングでの使い分け
  - `scikit-learn` の位置づけ
    - Phase 1 では主にデータ取得や周辺用途
    - 学習器の主役は `LightGBM`
  - 前処理の基本
    - 欠損値補完
    - 外れ値処理
    - 対数変換
  - 特徴量エンジニアリングの基本
  - 学習・評価・推論の違い
  - `RMSE` / `R²` / `NDCG` など指標の読み方
  - モデル成果物、metrics、run 履歴の見方

- **アプリケーション / 設計パターン**
  - `FastAPI` の役割
  - `lifespan` による初期化
  - DI の考え方
  - `Port/Adapter` の依存方向
  - `Repository pattern`
  - `core -> ports <- adapters` の責務分離
  - job 分離
    - seed
    - train
    - batch serving
  - API と pipeline の役割の違い

- **検索 / ランキングの基本要素**
  - lexical search と semantic search の違い
  - `Meilisearch` の役割
    - `Elasticsearch` より導入しやすく、学習用ローカル構成として扱いやすいこと
  - `multilingual-e5` embedding の役割
  - `RRF` の役割
  - `LightGBM LambdaRank` の役割
  - cache (`Redis`) の使いどころ
  - `/search` のリクエストから再ランクまでの流れ

- **GCP 基礎機能**
  - `Cloud Run` の役割
  - `GCS` の役割
  - `BigQuery` の役割
  - `Cloud Logging` / `Cloud Monitoring` の役割
  - `Secret Manager` の役割
    - Meilisearch master key をどう扱うか
  - `Workload Identity Federation` の役割
  - `Terraform` によるリソース管理の考え方

- **Vertex AI 基礎機能**
  - `Vertex AI Pipelines` の役割
  - `Model Registry` の役割
  - `Endpoint` の役割
  - `Model Monitoring` の役割
  - Phase 4 の GCP 機能から、どこが Vertex 管理に置き換わるか

- **PMLE 追加技術の基礎要素**
  - `BQML`
  - `Dataflow`
  - `Monitoring SLO`
  - Explainable AI の基本的な見せ方
  - 「default 挙動は Phase 5 のまま、追加機能は opt-in」の考え方

- **GKE / KServe 基礎機能**
  - `GKE Autopilot` の役割
  - `KServe` の役割
  - `Gateway API` の役割
  - Kubernetes 上での serving 差し替えの考え方
  - Phase 6 から継承するものと、Phase 7 で差し替わるものの整理

- **再現性 / 成果物管理**
  - `run_id` の意味
  - モデル保存先と latest の見方
  - `metrics.json` や評価レポートの追い方
  - ローカル保存から GCS / Vertex 管理への移行
  - Phase 間で成果物を共有しない理由

- **テスト / 品質保証**
  - `pytest` の役割
  - unit test / integration test の違い
  - `make test` と `make check` の違い
  - lint / format / type check の意味
  - 「default 挙動を壊していない」ことをどう確認するか

- **設定 / シークレット管理**
  - `pydantic-settings` の役割
  - `setting.yaml` と `credential.yaml` の分離
  - 環境変数優先の考え方
  - ローカル設定とクラウド設定の違い
  - Secret をコードに埋め込まない理由

- **運用 / 監視の基礎**
  - ログを見る場所
  - metrics をどこで確認するか
  - 学習ジョブ / 推論サービス / バッチ処理の監視観点
  - `Cloud Monitoring` と `SLO` の違い
  - 障害時にどこから調べるかの導線

- **コスト / リソース意識**
  - ローカル実行とマネージドサービスのコスト差
  - 常時課金されやすい構成要素
  - 学習目的で一時的に有効化する機能と常設機能の違い

- **学習比較に使ってよい切り口**
  - 前 Phase と比べて増えた技術
  - 置き換わった adapter / service
  - 変わらない中核ロジック
  - ローカル実装からクラウド実装への責務移動
  - 「学ぶ対象」と「詳細説明を省いてよい対象」の線引き

- **学習者向けに補助してよい観点**
  - 各 Phase で新しく増えるコンポーネントの一覧
  - 前 Phase から何が増え、何が変わらないかの比較
  - ローカル実行とクラウド実行の違い
  - 実行コマンドと成果物確認ポイント
  - 「この Phase で理解すべきこと / 深追いしなくてよいこと」の切り分け

## 6. 横断的な不変ルール (改訂版)

## 8. 差分洗い出しの観点

教育資料の改修対象は、少なくとも以下の観点で分類する。

- **削除**: 既に採用しない技術、古い構成、誤った説明
- **更新**: 用語、構成図、実行手順、成果物配置、アーキテクチャ説明
- **追加**: 新技術の説明、図、比較、学習順序、注意点
- **保留**: Phase 正本とコードの差分確認後に判断すべき項目

特に Phase 1〜6 は、仕様更新が先行している可能性を前提に、**「README に書かれている」ことだけで教材内容を確定しない**。必ず Phase 配下資料と実コードの突合結果を残す。

## 9. Phase別の調査・修正タスク

### Phase 1

- ML 基礎教材として不要なアプリ説明が混入していないか確認
- preprocess / 学習 / 評価 / artifact 出力の導線を整理
- `runs/{run_id}/` など成果物の見せ方を確認

### Phase 2

- FastAPI / DI / Port/Adapter / job 分離の説明が最新構成と一致するか確認
- Phase 1 との差分が学習者に明確に伝わる構成にする

### Phase 3

- ローカルハイブリッド検索の全体像を、lexical / semantic / rerank の3段で説明できているか確認
- Meilisearch / multilingual-e5 / LambdaRank / Redis の役割分担を明確化

### Phase 4 (コード化対象、Phase 3 からの足し算)

- **Phase 3 からの足し算で構築する** ことを明示 (Phase 7 完成版からの引き算ではない)
- GCP 移行で増える管理対象を整理 (Cloud Run / Cloud Run Jobs / GCS / BigQuery / Secret Manager / Terraform / WIF / IAM / 軽量 retrain trigger)
- `Secret Manager` を必須学習項目として明示
- Cloud Run / BigQuery / GCS / Terraform / WIF の接続関係を図示できる形にする
- **Vertex AI / Cloud Composer / GKE / KServe / PMLE 追加技術は扱わない** ことを明示 (これらは Phase 5 / 6 / 7 の主題)

### Phase 5 (論理 Phase、Phase 7 完成版を参照)

- **個別コードを持たない論理 Phase** であることを明示
- Vertex AI MLOps 化の技術境界 (Phase 4 → Phase 7 の橋渡し) を説明
- Endpoint / Pipelines / Model Registry / Monitoring / Feature Store / Vertex Vector Search の役割分担を整理
- Phase 7 配下のどのコード (`pipeline/`, `infra/terraform/modules/{vertex,vector_search}/`) を参照するかを必ず明示

### Phase 6 (論理 Phase、Phase 7 完成版を参照)

- **個別コードを持たない論理 Phase** であることを明示
- PMLE の追加技術 + Composer 本線化を「Phase 5 default を壊さず opt-in で学ぶ」前提で整理
- BQML / Dataflow / SLO / Explainable AI / Cloud Composer DAG を、本流と副経路の区別つきで説明
- Phase 7 配下のどのコード (`pipeline/dags/` ほか) を参照するかを必ず明示

### Phase 7 (コード化対象、canonical 起点 / 教材コード完成版)

- **Phase 5 / 6 の技術要素も Phase 7 配下に集約**していることを明示 (Cloud Composer + Vertex AI 一式 + GKE + KServe + PMLE 4 技術)
- 中核ハイブリッド検索 (Meilisearch BM25 + Vertex Vector Search + ME5 + RRF + LightGBM LambdaRank) と GKE/KServe による serving 層を分離して説明
