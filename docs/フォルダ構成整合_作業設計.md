**軸**: `4/study-hybrid-search-vertex/`
**対象**: `3/study-hybrid-search-cloud/` / `2/study-hybrid-search-local/` / `1/study-ml-foundations/`
**作成日**: 2026-04-21
**スコープ**: フォルダ構成のみ（コードロジックの書き換えは含まない。import パスと packaging 変更は付随作業として含む）

---

## 1. 目的と原則

**目的**: 4 phase のフォルダ構成をできるだけ揃え、Phase を跨いだ読者（教材視聴者 / Claude Code / レビュアー）が「同じ概念は同じ場所」を期待できるようにする。

**原則**:
1. **できるだけ**（"as much as possible"）— 強制的な uniformity ではなく、**概念が共通する箇所**を揃える
2. **Phase 4 は不変**（軸、作業対象外）
3. **Phase 固有の設計意図は保持**（Phase 1 の Docker Compose 教育用途 / Phase 2 の Local 検証用途）
4. **コード修正は最小限**（フォルダ移動に伴う import path 修正だけ。ロジック変更はしない）
5. **doc 追従を必須扱い**（README / CLAUDE.md / 01_仕様と設計.md / 03_実装カタログ.md 等のパス参照）

## 5. Phase 別作業設計

### 5.1 Phase 4（軸、作業対象外）

**作業なし**。基準として不動。

### 5.2 Phase 3（すでに 95% 整合）

**作業なし**を推奨。`pipelines/` / `functions/` / `infra/modules/vertex/` / `jobs/containers/` / `tests/functions/` はすべて Phase 4 固有概念のため、空ディレクトリを作っても意味がない。

**唯一の検討事項**: Phase 3 にも `jobs/containers/` を作って Dockerfile を移すか？
- 現状 Phase 3 は `Dockerfile` 相当を `cloudbuild.*.yaml` 経由で build している。
- Phase 4 は `jobs/containers/{encoder,reranker,trainer}/` に Dockerfile を置いている。
- Phase 3 に `jobs/containers/trainer/` を作って既存 training-job の Dockerfile を移す余地はあるが、現状 Cloud Build が期待するパスが変わるためリスクあり。
- **推奨**: 変更なし。

### 5.3 Phase 2（部分整合）

#### やること（Tier A + Tier C）

**A-1. `CLAUDE.md` 新規作成**（必須）
- 現状: なし
- 目標: Phase 3/4 と同じ書式で、非負制約 / 参照リポジトリ / 開発コマンドを記載
- 作業: `CLAUDE.md` を Phase 2 ルートに新規作成
- 参考: Phase 3 の `CLAUDE.md` の章構成（「最初に読むもの」「設計テーゼ」「非負制約」「ドキュメント衝突時の権威順位」「開発コマンド」「リポジトリ状態」）
- コスト: S（数時間）

**C-1. `scripts/` を lifecycle 別に再分類**
- 現状:
  ```
  scripts/
  ├── ops/ (多数のチェックスクリプト)
  ├── check_layers.py (top-level)
  ├── compose.sh
  └── ... (その他)
  ```
- 目標:
  ```
  scripts/
  ├── checks/
  │   └── layers.py              ← 現 scripts/check_layers.py
  ├── ops/                       そのまま
  │   ├── health_check.py
  │   ├── search_check.py
  │   ├── feedback_check.py
  │   ├── ranking_check.py
  │   ├── ranking_check_verbose.py
  │   ├── training_label_seed.py
  │   └── training_fit_safe.py

  ```
- 作業: git mv 3 ファイル + Makefile の参照パス更新（`scripts/check_layers.py` / `./scripts/compose.sh`）
- コスト: S（30 分）

#### やらないこと（Tier B、明示的 skip）

**B-1. `src/` workspace 化**
- 理由: Phase 2 は「Local 検証 + 教材素材」という位置付けで、uv workspace の境界保証（AST 検査）は要件でない
- 代替: 現在の `src/` ディレクトリ構造は README / 03_実装カタログ / 教材すべてに深く引用されている。再編は教材全体の更新を巻き込む

**B-2. pip → uv 移行**
- 理由: Phase 2 の `make sync` はすでに `pip install -r requirements-dev.txt` に揃え済み。ツール差異は target 名では隠蔽済
- 代替: 教材で「pip でも uv でも同じコマンドで動く」と強調する方が ROI 高

### 5.4 Phase 1（整合対象最小）

#### やること（Tier A）

**A-1. 既に揃っている項目の維持**
- `CLAUDE.md` ✓ / `env/{config,secret}/` ✓ / `docs/教育資料/` ✓
