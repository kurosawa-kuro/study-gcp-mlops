### 2.0 作業姿勢

移行作業は、差分を意識しつつ **コード修正の前進速度を優先** する。

コード差分が大きい場合は一つまえのphaseのコードを大幅にコピーして引き算修正して各phaseのコードの差を抑えなさい

- まずは各 Phase のコードと文書を現行スコープへ揃える
- E2E 検証や CI/CD 上での総合動作確認は後段へ回す
- ローカルで即確認できる単体テスト、lint、限定的な smoke は許容する
- 既存の広範な lint / test 負債があっても、移行差分に直接関係しないものは一旦切り分ける
- 重要なのは「その Phase の学習主眼に合う形へ構造を戻すこと」であり、最初から完全検証を狙わない

# 02_移行ロードマップ

本ドキュメントは、**Phase 7 を最新・最終の基準実装として、Phase 6 → Phase 1 へ引き算で整合させるための全体作業計画書** である。  
対象は **教育資料だけでなく、各 Phase の実装コード・README・運用ドキュメントも含む**。

## 1. 目的

現状の課題は、各 Phase の仕様説明、教育資料、実装コードの進化タイミングがずれており、**Phase 間で「何が増えたのか / 何がまだ無いのか」が逆算しにくい** 点にある。

今回の移行方針は、Phase 1 から順に足し上げるのではなく、**Phase 7 を完成形の参照点として固定し、そこから不要要素を引いて各 Phase の学習スコープへ落とし戻す** 方式を採る。

狙い:

- Phase 間の差分を一貫した軸で説明できるようにする
- 教育資料と実装コードの対応関係を明確にする
- 各 Phase の「学ぶ対象」と「まだ存在しない対象」を明示する
- 古い技術選定や削除済み機能が下位 Phase に残る問題を防ぐ

## 2. 基本方針



### 2.1 基準 Phase

- **Phase 7 (`7/study-hybrid-search-gke/`) を最新・最終の基準実装とする**
- ただし Phase 7 は **serving 層差し替えに特化した Optional / Draft** であり、すべての技術を「最上位完成版」として押し広げるのではない
- 基準として使うのは、主に次の 3 点
  - ディレクトリ整理方針
  - Port/Adapter / DI / composition root の整理状態
  - Phase 6 までで確立した不動産ハイブリッド検索の中核構成

### 2.2 引き算の考え方

各 Phase は、Phase 7 から次を引いて定義する。

- その Phase ではまだ導入しないクラウド機能
- その Phase ではまだ導入しない serving 機構
- その Phase ではまだ導入しない PMLE 追加技術
- その Phase の学習主眼をぼかす補助機能

つまり、

- Phase 7 = Phase 6 + GKE/KServe serving 差し替え
- Phase 6 = Phase 5 + PMLE 追加技術
- Phase 5 = Phase 4 + Vertex 標準 MLOps
- Phase 4 = Phase 3 + GCP 実行基盤
- Phase 3 = Phase 2 + 不動産ハイブリッド検索
- Phase 2 = Phase 1 + App / Pipeline / Port-Adapter
- Phase 1 = ML 基礎だけを残した最小構成

## 3. 参照順序

本計画書を実施する際の参照順序は以下とする。

1. 各 Phase 配下の `README.md` / `CLAUDE.md` / `docs/`
2. ルート [`README.md`](../README.md)
3. ルート `docs/` 配下の横断ハブ

ルート `docs/` は横断管理のための補助資料であり、**各 Phase の実態確認は必ず Phase 配下を正本として扱う**。

## 4. 非負制約

今回の移行作業でも、以下は崩さない。

- Phase 3〜7 の中核は **LightGBM + multilingual-e5 + Meilisearch**
- Phase 6 の `/search` default 挙動は **Phase 5 と一致**
- Phase 7 は **Phase 6 の学習基盤・データ基盤を継承し、serving 層のみ差し替える**
- `W&B` / `Looker Studio` / `Doppler` は教材対象外
- `Agent Builder` / `Vector Search` は削除済み機能として扱い、教育資料や説明から除外する
- `Secret Manager` は Phase 4 の必須習得要素として維持する

## 7. Phase別の修正方針

### Phase 7

- 基準実装として扱う
- 「何を残し、何を差し替えたか」の説明を最優先で整理する
- 実装、README、教育資料の差を最初に収束させる

### Phase 6

- Phase 7 から **serving 差し替え要素を除いた状態** に揃える
- PMLE 追加技術は残すが、削除済みの `Agent Builder` / `Vector Search` は除外する
- default 挙動が Phase 5 と一致する説明へ統一する

### Phase 5

- Phase 6 から PMLE 追加技術を除いた、**Vertex 標準 MLOps の中核** に揃える
- `Pipelines` / `Endpoint` / `Model Registry` / `Monitoring` を主題に絞る

### Phase 4

- Phase 5 から Vertex 専用要素を除いた、**GCP serverless 実行基盤** に揃える
- `Secret Manager` を必須項目として強調する

### Phase 3

- Phase 4 からクラウド要素を除いた、**ローカルハイブリッド検索** に揃える
- `Meilisearch` 採用理由として「`Elasticsearch` より導入しやすい」を明記する

### Phase 2

- Phase 3 から検索ドメイン要素を除き、**App / Pipeline / Port-Adapter の導入教材** として揃える
- California Housing 題材に閉じる

### Phase 1

- Phase 2 から API / DI / Port-Adapter を除き、**ML 基礎の最小構成** として揃える
- 成果物管理は `run_id` / `metrics.json` / model 保存に限定する

## 8. 実作業の順序

実作業は **Phase 7 → Phase 6 → Phase 5 → Phase 4 → Phase 3 → Phase 2 → Phase 1** の順に進める。

理由:

- 最新実装に最も近いところから正本化した方が、引き算基準がぶれない
- 下位 Phase へ進むほど「残すべき最小構成」が見えやすくなる
- 教育資料とコードの差分を同じ順序で吸収できる

## 9. 各 Phase で行う確認

各 Phase ごとに、少なくとも次を確認する。

1. README の主張と実装コードが一致しているか
2. `docs/01_仕様と設計.md` / `03_実装カタログ.md` / `04_運用.md` の説明が一致しているか
3. `docs/教育資料/01_スライド.md` / `05_図解.md` がその Phase の学習主眼に合っているか
4. 削除済み技術や対象外技術が残っていないか
5. 次の Phase との接続説明が自然か
