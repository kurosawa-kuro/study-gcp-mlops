## 2. 実装ルール

### 2.1 必須ルール
1. **コード差分が大きい場合は、一つ前のPhaseのコードを大幅にコピーしてから引き算修正せよ** できるだけこの方針で作業してほしい。修正ファイル数が３，４個なら直接修正許可。なので、現実的にすべて一つ前のphaseのまるごとコピーから作業が始まるはず
   - 各Phase間のコード差分を**最小化**することが目的
   - スクラッチで書き直さない
2. **Phase間のコード変更箇所が一目で分かる構造を維持せよ**
3. **作業の主役は「コード」**。ドキュメントは整合のための従属作業。

### 2.2 例外ルール（盲目的コピーしない箇所）
> **盲目的なコピー＋引き算だけでは成立しないPhase差分は、微調整を行うこと。**

各Phaseには固有の学習ポイント・実行方式があるため、引き算後に以下を考慮して調整:
- そのPhase固有のテーマ（下表）に整合しているか
- 実行方式（Docker Compose / uv / Cloud / Vertex / GKE）の差異
- 主要技術スタックの差異

## 4. 不変制約（引き算しても残すこと）

以下はPhase 3〜7で**絶対に削除してはならない**:

- **ハイブリッド検索中核**: LightGBM + multilingual-e5 + Meilisearch
- **`/search` default 挙動**（Phase 6でPMLE技術を引く際も、default挙動はPhase 5と同等を維持）
- **Port/Adapter の設計思想**（Phase 2以降で一貫）

## 5. 作業順序

1. **Phase 7のコードを基準として確定**（現状コードのレビュー）
2. **Phase 6 を Phase 7 から派生**:
   - Phase 7 を丸ごとコピーベースとし、上表の「引き算対象」に該当するコード/設定/Terraform/manifestを削除
   - Phase 6 固有要件（PMLE 8技術 default opt-in）への微調整
3. **Phase 5 を Phase 6 から派生**:
   - 同様にPMLE追加技術の adapter / 副経路 / 追加エンドポイント を引き算
4. **Phase 4 を Phase 5 から派生**:
   - Vertex AI関連（Pipelines / Endpoint / Model Registry / Monitoring）を引き算
   - Cloud Run / GCS / BigQuery / Secret Manager は残す
5. **Phase 3 を Phase 4 から派生**:
   - GCPマネージド（Cloud Run / GCS / BigQuery / Secret Manager / Terraform / WIF）を引き算
   - ローカル実行（Docker Compose）へ調整
6. **Phase 2 を Phase 3 から派生**:
   - ハイブリッド検索一式（Meilisearch / multilingual-e5 / LambdaRank / Redis）を引き算
   - 単純な回帰モデル + FastAPI構成へ調整
7. **Phase 1 を Phase 2 から派生**:
   - App層 / Pipeline / Port-Adapter を引き算
   - ML基礎（学習・評価・保存）のみに集約
