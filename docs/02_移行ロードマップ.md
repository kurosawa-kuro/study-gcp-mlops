## 2. 実装ルール

### 🔴 2.1 【最重要】コード修正の3ステップ手順（厳守）

> **この3ステップを必ず守ること。スクラッチ修正・部分修正は禁止。**
> **理由: トークン消費が少なく、Phase間の最終的なズレが最小化されるため。**

#### ✅ STEP 1: 現Phaseコードを `old` としてバックアップ
```
例: Phase 6 を修正する場合
  6/study-hybrid-search-pmle/  →  6/study-hybrid-search-pmle_old/
```
- 修正対象Phaseのコードを **まるごと退避** する
- ディレクトリ名末尾に `_old` を付与してリネーム or コピー

#### ✅ STEP 2: 一つ前のPhaseコードを丸ごとコピー → 引き算修正

```
例: Phase 6 を修正する場合
  7/study-hybrid-search-gke/   →  6/study-hybrid-search-pmle/  (実コードのみコピー)
  その後、Phase 6 の特徴に合わせて引き算修正
```

- **§1.3 の対応関係に従い、一つ前のPhase（より進んだPhase）のコードをそのままコピー**
- コピー後、当該Phaseの特徴（§3）に合わせて **引き算でコード修正**
- スクラッチで書き直さない / 部分マージしない

##### ⚠️ コピー時は大容量ビルド/生成物を必ず除外（除外しないと現実的に不可能）

ビルド成果物・キャッシュ・モデル/データ等を含めたまま `cp -r` するとサイズ爆発でコピーが完了しない。**実コードのみ**を再複製する。

除外対象（最低限）:

```
.venv/  __pycache__/  *.pyc
.terraform/  .terraform.lock.hcl は残す
.mypy_cache/  .ruff_cache/  .pytest_cache/
node_modules/  dist/  build/  *.egg-info/
runs/  models/  artifacts/  data/  *.pkl  *.parquet  *.csv （シードデータを除く）
.DS_Store  *.log
```

推奨コピー手順（rsync 例）:

```bash
rsync -a \
  --exclude='.venv/' \
  --exclude='__pycache__/' \
  --exclude='*.pyc' \
  --exclude='.terraform/' \
  --exclude='.mypy_cache/' \
  --exclude='.ruff_cache/' \
  --exclude='.pytest_cache/' \
  --exclude='node_modules/' \
  --exclude='dist/' \
  --exclude='build/' \
  --exclude='*.egg-info/' \
  --exclude='runs/' \
  --exclude='models/' \
  --exclude='artifacts/' \
  --exclude='*.pkl' \
  7/study-hybrid-search-gke/  6/study-hybrid-search-pmle/
```

各 Phase の `.gitignore` / `.dockerignore` も合わせて参照し、コミット対象外＝コピー対象外を原則とする。

##### 作業計画

作業計画を新しい phase のフォルダの `02_移行ロードマップ.md` に書き込んでから、コード引き算修正を行う。

#### ✅ STEP 3: `old` バックアップを削除
```
例: Phase 6 修正完了後
  6/study-hybrid-search-pmle_old/  →  削除
```
- 引き算修正が完了し、動作確認できたら `_old` バックアップを削除
- 中間状態を残さない

#### この3ステップ手順を採用する理由

| 理由 | 効果 |
|---|---|
| トークン消費が少ない | 既存コードの差分追跡ではなく、上位Phaseの完成形をベースにできる |
| Phase間の最終的なズレが少ない | Phase 7を起点に一貫した派生で「引き算」のみが累積する |
| 引き算の意図が明確 | 「何を消したか」だけが差分になり、教材としての説明性が高い |

## 5. 作業順序

各Phaseの修正時は **必ず §2.1 の3ステップ手順（バックアップ → 上位Phaseコピー → 引き算 → バックアップ削除）** を踏むこと。

1. **Phase 7のコードを基準として確定**（現状コードのレビュー）
2. **Phase 6 を Phase 7 から派生**:
   - §2.1 の手順に従い、Phase 6 を `_old` 退避 → Phase 7 を Phase 6 ディレクトリへ丸ごとコピー → 引き算 → `_old` 削除
   - 引き算対象: GKE/KServe 等（§1.3）
   - Phase 6 固有要件（PMLE 8技術 default opt-in）への微調整
3. **Phase 5 を Phase 6 から派生**: 同手順、PMLE追加技術を引き算
4. **Phase 4 を Phase 5 から派生**: 同手順、Vertex AI関連を引き算（Cloud Run / GCS / BigQuery / Secret Manager は残す）
5. **Phase 3 を Phase 4 から派生**: 同手順、GCPマネージドを引き算、ローカル実行（Docker Compose）へ調整
6. **Phase 2 を Phase 3 から派生**: 同手順、ハイブリッド検索一式を引き算、回帰モデル+FastAPI構成へ調整
7. **Phase 1 を Phase 2 から派生**: 同手順、App層/Pipeline/Port-Adapterを引き算、ML基礎のみに集約
