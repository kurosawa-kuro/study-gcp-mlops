## フォルダ・ファイル名の必須仕様

### 1. 目的

- この構成は実運用最適化より、学習理解と MLOps 技術名の可視化を優先する
- レイヤ責務を名前で即判別できることを最重要要件とする

### 2. 命名ルール（必須）

- ディレクトリ名は小文字スネークケースのみ
- Python ファイル名は小文字スネークケースのみ
- ジョブのエントリーポイントは `pipeline/*_job/main.py` に統一
- API エントリーポイントは `app/main.py` に統一
- 評価・学習・推論・登録など MLOps 固有語を省略しない

### 3. ルート直下の必須ディレクトリ

- `app`
- `ml`
- `pipeline`
- `infra`
- `docs`
- `scripts`
- `tests`

上記は必須。別名（例: `applications`, `core`, `batch`, `qa`）への置換は禁止。

### 4. `ml` 配下の必須責務ディレクトリ

- `data`
- `training`
- `evaluation`
- `registry`
- `serving`
- `common`

上記は必須。`trainer` と `training` の混在、`eval` と `evaluation` の混在は禁止。

### 5. `pipeline` 配下の必須ジョブ名

- `data_job`
- `training_job`
- `evaluation_job`
- `batch_serving_job`
- `workflow`

ジョブごとのエントリーポイントは `main.py` に固定し、`run.py` `entry.py` などの別名は使わない。

### 6. `tests` 配下の必須区分

- `unit`
- `integration`
- `e2e`

テストはこの3区分に配置し、`api/` `ml/` などの責務別トップレベルを増やさない。

### 7. 禁止事項

- 旧構成（例: `app/src`, `common/src`, `jobs/src`, `pipelines/src`）の再導入
- 同義語ディレクトリの追加（例: `train`, `eval`, `serve`, `ops`）
- レイヤ責務をまたぐ配置（例: 学習コードを `app/` に置く）

### 8. 例外運用

- 一時互換レイヤを置く場合でも、最終形は本仕様に収束させる
- 例外を許可する場合は `docs/decisions/` に理由と期限を記録する

### 9. レビュー観点（チェックリスト）

- ルート必須ディレクトリが揃っているか
- `ml` と `pipeline` の必須サブディレクトリが揃っているか
- `pipeline/*_job/main.py` 形式になっているか
- 命名が小文字スネークケースに統一されているか
- 旧構成パスが新規追加されていないか

   
## 要素ごとの整理（目的・役割付き）

| 要素          | 目的                   | 役割                                                | Cloud Run Job / Service | 理由                                    |
| ----------- | -------------------- | ------------------------------------------------- | ----------------------- | ------------------------------------- |
| infra       | MLOps全体が動作する土台を用意する  | クラウド基盤、DB、Storage、IAM、Scheduler、実行環境、ネットワーク等を提供する | どちらでもない                 | 基盤そのもの。Cloud Run の上位にある土台  IaC Teraform           |
| data        | 学習・評価・推論で使えるデータを整備する | データ収集、投入、加工、前処理、特徴量生成、学習/推論データ整備を行う               | Job                     | データ収集・加工・特徴量生成は「実行して終わる」処理            |
| training    | モデルを作成し、再学習できる状態を保つ  | 学習ジョブの実行、再学習、実験、パラメータ変更によるモデル生成を行う                | Job                     | 学習は典型的なバッチ実行                          |
| evaluation  | モデル品質を判定し、本番採用可否を決める | 指標計算、比較、閾値判定、採用判定、レポート生成を行う                       | Job                     | 評価・比較・採用判定も実行完了型                      |
| registry    | 採用モデルと成果物を一元管理する     | モデル成果物保存、バージョン管理、採用モデル記録、メタデータ管理を行う               | どちらでもない                 | モデル保存先・管理先であり、実行主体ではない                |
| serving     | モデルを実際の業務利用に供する      | オンライン推論API、バッチ推論、推論結果返却、推論結果出力を行う                 | Service または Job         | オンライン推論は Service、定期一括推論は Job          |
| application | ユーザーや業務システムに機能を提供する  | UI、業務API、フロントエンド、BFF、利用者接点を担う                     | Service                 | UI/API はリクエスト待受型                      |
| monitoring  | 障害・劣化・異常を継続監視する      | ジョブ失敗、API異常、性能劣化、データドリフト、ログ・メトリクス監視を行う            | 原則どちらでもない               | 監視基盤は横断機能。補助的に Job や Service を使うことはある |

## 補足すると、各要素の見方はこうです

### infra

* **目的**: 全体を成立させるための土台
* **役割**: 実行環境やデータ保管先、権限管理を用意する
* **性質**: パイプラインの処理対象ではなく、パイプラインを載せる基盤

### data

* **目的**: モデルが扱える形にデータを整える
* **役割**: 収集・加工・前処理・特徴量化
* **性質**: Jobで順次実行しやすい工程

### training

* **目的**: モデルを生成・更新する
* **役割**: 学習、再学習、実験実行
* **性質**: バッチ処理そのもの

### evaluation

* **目的**: 学習済みモデルを使ってよいか判定する
* **役割**: 比較、評価、採用可否判断
* **性質**: 複数の評価機能を意図的に束ねた責務単位

### registry

* **目的**: どのモデルを正式採用したかを管理する
* **役割**: 保存、版管理、採用履歴管理
* **性質**: Jobではなく管理領域

### serving

* **目的**: モデルを利用可能な形で提供する
* **役割**: 推論APIまたはバッチ推論提供
* **性質**: 提供形態により Job / Service が分かれる

### application

* **目的**: 利用者や業務システムに価値として見せる
* **役割**: UI/API/業務導線の提供
* **性質**: 常駐型の入口

### monitoring

* **目的**: 全体の安定運用を維持する
* **役割**: 障害検知、劣化検知、アラート
* **性質**: 全体横断の監視レイヤー
