# フォルダ・ファイル

この文書は、`study-gcp-mlops` を教材として横断的に読むための構造索引です。  
方針は単純で、**Phase 7 時点の最大集合を基準にし、各 Phase はそこから必要なものだけを持つ**とみなします。

詳細な命名規約は [conventions.md](/home/ubuntu/repos/study-gcp-mlops/docs/conventions.md) を正本とします。  
この文書は「どこに何があるか」、`conventions.md` は「なぜその名前か」を扱います。

## ルート構成

```text
.
├── 1/ ... 7/                各学習 phase
├── .github/                 ルート共通の agent / skill / GitHub 設定
├── docs/                    ルート共通 docs
├── mk/                      共通 Make 補助

```

## 最大集合

教材としての標準形は次です。  
各 Phase はこの集合から不要なものが抜けるだけ、という読み方をします。

```text
<phase>/
├── README.md
├── CLAUDE.md
├── Makefile
├── pyproject.toml
├── app/                    # API / UI / serving
├── common/                 # 共通設定・logging
├── ml/                     # 機能別 ML 実装
├── pipeline/               # <verb>_job + workflow
├── scripts/                # make から呼ぶ CLI
├── infra/                  # terraform / run / manifests
├── env/                    # config / secret
├── monitoring/             # 監視 SQL
├── docs/                   # 01/03/04 を中核に置く
├── tests/                  # unit / integration / e2e
├── tools/
├── logs/                   # 生成物
└── dist/                   # 生成物
```

### 共通フォルダの意味

| フォルダ | 役割 |
| --- | --- |
| `app/` | API / UI / serving 層。Phase 7 では DI 全面整備済 (詳細下記) |
| `common/` | 共通設定・logging・shared utility (Phase 2/3 のみ root 直下、Phase 4-7 は `ml/common/` に集約) |
| `ml/` | 機能別 ML 実装 |
| `pipeline/` | pipeline / job orchestration |
| `scripts/` | `make` の公開 API を支える CLI |
| `infra/` | Terraform / Cloud Run / GKE / manifest / Cloud Build 設定 |
| `env/` | `config` と `secret` |
| `monitoring/` | BigQuery SQL や監視補助 |
| `docs/` | 仕様・実装・運用・教材 |
| `tests/` | `unit` `integration` `e2e` |
| `logs/` | monitor 出力などの生成物 |
| `dist/` | compile / export 済み生成物 |

#### `app/` の内訳 (Phase 7 で確立した DI 構造)

| 配下パス | 役割 |
| --- | --- |
| `app/main.py` | FastAPI entrypoint。`create_app()` 内で `lifespan` を生やし `ContainerBuilder.build()` を呼ぶ |
| `app/composition_root.py` | `Container` (frozen dataclass) + `ContainerBuilder`。DI 配線の単一箇所 |
| `app/container/{infra,search,ml}.py` | DI builder 分割 (Pub/Sub-BQ / search path / ML optional path) |
| `app/container/_optional_adapter.py` | `enable_xxx` flag 付き optional adapter 構築の共通 helper |
| `app/observability.py` | `Observability` frozen dataclass — service_name / logger_factory / `expose_prometheus()` を束ねる |
| `app/settings/api.py` | `ApiSettings` (pydantic-settings) |
| `app/domain/` | `Candidate` / `SearchFilters` / `SearchInput` / `SearchOutput` 等 (Pydantic schema とは別) |
| `app/api/handlers/` | 1 endpoint 1 file (search / rag / feedback / retrain / model / health / ui) |
| `app/api/mappers/` | Pydantic ↔ domain 変換 |
| `app/api/middleware/` | `RequestLoggingMiddleware` (request_id 発行) |
| `app/api/dependencies.py` | `get_container` / `get_search_service` 等 FastAPI Depends 解決 |
| `app/services/protocols/` | Port (Protocol、1 Protocol 1 file) |
| `app/services/adapters/` | 具象実装 (KServe / Meilisearch / Pub/Sub / BQ / Gemini / BQML 等) |
| `app/services/fakes/` | production noop / in-memory 実装 (test stub は `tests/fakes/` 別) |
| `app/services/{search,feedback,rag,model_metrics}_service.py` | Service 層 (Port のみ import、adapter 直 import 禁止) |
| `app/schemas/` | Pydantic request / response schema |
| `app/templates/` `app/static/` | operator UI (Jinja2 + 静的アセット、`/ui/` 配下に閉じ込め) |

## 標準命名

### `ml/`

`ml/` は **機能別** を正とします。

```text
ml/
├── common/
├── data/
├── training/
├── evaluation/
├── registry/
├── serving/
├── streaming/
└── experiment/
```

- `adapters` `ports` `core` は `ml/` 直下ではなく、必要なら各機能ディレクトリの内側で使うのが標準です

### `pipeline/`

`pipeline/` は **`<verb>_job`** を正とします。

```text
pipeline/
├── data_job/
├── training_job/
├── evaluation_job/
├── batch_serving_job/
└── workflow/
```

- `train_job` は `training_job` に寄せる
- `predict_job` は `batch_serving_job` に寄せる
- `seed_job` は `data_job` に寄せる
- Phase 3 のレイヤー構成は `pipeline/*_job/` の内側に閉じ込めるのが標準です

### `scripts/`

`scripts/` は Phase 4-7 で確立した構成を正本にします。

```text
scripts/
├── _common.py
├── setup/
├── deploy/
├── ops/
├── ci/
├── sql/
├── bqml/
├── dev/                    # deprecated wrapper
└── local/                  # deprecated wrapper
```

- `Make target = 公開 API`
- `scripts/setup` `scripts/deploy` `scripts/ops` = canonical
- `scripts/dev` `scripts/local` は移行期間中だけ残る互換 wrapper
- `core.py` は将来的に `_common.py` へ寄せる対象です

### `docs/`

`docs/` の基本番号は固定です。

```text
docs/
├── 01_仕様と設計.md
├── 02_移行ロードマップ.md
├── 03_実装カタログ.md
├── 04_運用.md
├── architecture/
├── decisions/
└── archive/
```

- `01` `03` `04` は全 Phase の中核 docs
- `02` は前 Phase からの差分説明が必要な Phase のみ
- 一時的な refactor plan は完了後 `archive/` へ移すのが標準です

### `tests/`

標準は次の三層です。

```text
tests/
├── unit/
├── integration/
└── e2e/
```

- package 直下の `app/tests` `common/tests` は教材としては例外形です
- 可能なら `tests/` 集約に寄せるのが標準です

## 各要素の役割と実装形態

旧 `docs/パイプラインとジョブ.md` を統合 (`docs/archive/パイプラインとジョブ.md` に履歴あり)。
Cloud Run Job / Service という実装媒体は Phase ごとに変わる (Phase 4-6 = Cloud Run、Phase 7 = GKE Deployment + KServe InferenceService + Vertex AI Pipelines) ため、列名は **「実装形態」** に一般化している。

| 要素 | 目的 | 役割 | 実装形態 (Phase 7 基準) | 性質 |
|---|---|---|---|---|
| `infra` | MLOps 全体が動作する土台を用意する | クラウド基盤、DB、Storage、IAM、Scheduler、ネットワーク等 | Terraform / Helm / k8s manifest (基盤) | パイプラインを「載せる」基盤 |
| `ml/data` + `pipeline/data_job` | 学習・評価・推論で使えるデータを整備する | データ収集、加工、前処理、特徴量生成 | Vertex AI Pipelines (Phase 5 まで Cloud Run Job)。Dataform で SQL 整備 | バッチ実行で完了する処理 |
| `ml/training` + `pipeline/training_job` | モデルを作成し、再学習できる状態を保つ | 学習ジョブ、再学習、実験トラッキング | Vertex AI Pipelines + Vertex Experiments | 典型的なバッチ |
| `ml/evaluation` + `pipeline/evaluation_job` | モデル品質を判定し、本番採用可否を決める | 指標計算、比較、閾値判定、レポート生成 | Vertex AI Pipelines (offline)、API `/model/metrics` (online 簡易) | 実行完了型 |
| `ml/registry` | 採用モデルと成果物を一元管理する | モデル成果物保存、バージョン管理、production alias 管理 | Vertex Model Registry + GCS | 管理領域 (実行主体ではない) |
| `ml/serving` | モデルを実際の業務利用に供する | オンライン推論 API、batch 推論、predictor 実装 | KServe InferenceService (encoder / reranker)、Vertex CPR (`/explain` Pod)、`pipeline/batch_serving_job` (batch) | online = Service 常駐、batch = Job |
| `app/` | ユーザや業務システムに機能を提供する | 業務 API (`/search` `/rag` `/feedback`)、operator UI (`/ui/`)、retrain orchestration | GKE Deployment (FastAPI) | リクエスト待受型 (常駐) |
| `monitoring/` + `infra/terraform/modules/{monitoring,slo}` | 障害・劣化・異常を継続監視する | ジョブ失敗、API 異常、性能劣化、データドリフト、ログ・メトリクス監視 | Cloud Monitoring + GMP + Scheduled Query (横断機能) | 補助的に Job / Service を使うことはあるが横断レイヤー |
| `pipeline/workflow` | 上記 job 群の orchestration | KFP compile / Cloud Function trigger / PipelineJobSchedule | Vertex AI Pipelines + Eventarc + Cloud Scheduler | 複数 job のつなぎ |
| `ml/streaming` (Phase 6 T2 追加) | リアルタイム集計 (CTR 等) | ranking_log → hourly window CTR 集計 | Apache Beam → Dataflow Flex Template Job | streaming 型 |

### 一言でまとめると

- **infra** = 土台
- **data** = 材料準備
- **training** = モデル作成
- **evaluation** = 品質判定
- **registry** = 採用モデル管理
- **serving** = 推論提供
- **app** = 利用者接点 (API + operator UI)
- **monitoring** = 全体監視
- **streaming** = リアルタイム集計 (Phase 6+)

### Phase ごとの実装形態の段差

| 要素 | Phase 1-3 | Phase 4 | Phase 5 | Phase 6 | Phase 7 |
|---|---|---|---|---|---|
| `app` (online API) | Docker Compose `uvicorn` | Cloud Run Service | 同左 | 同左 | **GKE Deployment** |
| `ml/serving` (推論) | local | local + Cloud Run Service | Vertex AI Endpoint | 同左 | **KServe InferenceService** + Vertex CPR (`/explain`) |
| `pipeline/*_job` | local script | Cloud Run Jobs | Vertex AI Pipelines | 同左 | 同左 (継承) |
| `ml/streaming` | — | — | — | **Dataflow Flex Template** (Phase 6 T2) | 同左 |
| `infra` | Docker Compose | Terraform + Cloud Run | + Vertex AI primitives | + BQML / Gemini / SLO | + GKE Autopilot + KServe Helm + manifests |

中核思想 (Port-Adapter / 機能別 dir / `<verb>_job` / hybrid 検索 3 段構成) は phase を跨いで不変、実装形態だけが段階的に差し替わる。

## 教材として見るときの読み替え

### 「存在しない」は欠陥ではなく差分

- Phase 1 に `app/` がないのは、まだ API を扱わないから
- Phase 5 以前に `scripts/bqml/` がないのは、まだ BQML を導入していないから
- Phase 7 に `infra/manifests/` があるのは、GKE / KServe を扱うから

### 今後そろえると効果が大きい項目

1. `pipeline/` の job 命名を `<verb>_job` に統一
2. `ml/` を機能別で統一し、`adapters` `ports` `core` は内側へ寄せる
3. 各 Phase の `docs/01_仕様と設計.md` `03_実装カタログ.md` `04_運用.md` を必須の基本セットとして維持
