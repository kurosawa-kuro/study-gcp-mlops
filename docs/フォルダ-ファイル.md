# フォルダ・ファイル

この文書は、`study-gcp-mlops` を教材として横断的に読むための構造索引です。  
方針は単純で、**Phase 7 時点の最大集合を基準にし、各 Phase はそこから必要なものだけを持つ**とみなします。

詳細な命名規約は [conventions.md](/home/ubuntu/repos/study-gcp-mlops/docs/conventions.md) を正本とします。  
この文書は「どこに何があるか」、`conventions.md` は「なぜその名前か」を扱います。

## ルート構成

```text
.
├── 1/ ... 7/                各学習 phase
├── .github/                 ルート共通の agent / skill / GitHub 設定
├── docs/                    ルート共通 docs
├── mk/                      共通 Make 補助
└── tools/                   ルート共通ツール
```

## まず見る順

1. 対象 phase の `README.md`
2. 対象 phase の `CLAUDE.md`
3. 対象 phase の `Makefile`
4. 対象 phase の `docs/01_仕様と設計.md`
5. 対象 phase の `docs/03_実装カタログ.md`
6. 対象 phase の `docs/04_運用.md`
7. 実装本体として `app/` `ml/` `pipeline/` `scripts/`

## 最大集合

教材としての標準形は次です。  
各 Phase はこの集合から不要なものが抜けるだけ、という読み方をします。

```text
<phase>/
├── README.md
├── CLAUDE.md
├── Makefile
├── pyproject.toml
├── app/                    # API / UI / serving
├── common/                 # 共通設定・logging
├── ml/                     # 機能別 ML 実装
├── pipeline/               # <verb>_job + workflow
├── scripts/                # make から呼ぶ CLI
├── infra/                  # terraform / run / manifests
├── env/                    # config / secret
├── monitoring/             # 監視 SQL
├── docs/                   # 01/03/04 を中核に置く
├── tests/                  # unit / integration / e2e
├── tools/
├── logs/                   # 生成物
└── dist/                   # 生成物
```

### 共通フォルダの意味

| フォルダ | 役割 |
| --- | --- |
| `app/` | API / UI / serving 層 |
| `common/` | 共通設定・logging・shared utility |
| `ml/` | 機能別 ML 実装 |
| `pipeline/` | pipeline / job orchestration |
| `scripts/` | `make` の公開 API を支える CLI |
| `infra/` | Terraform / Cloud Run / GKE / manifest |
| `env/` | `config` と `secret` |
| `monitoring/` | BigQuery SQL や監視補助 |
| `docs/` | 仕様・実装・運用・教材 |
| `tests/` | `unit` `integration` `e2e` |
| `logs/` | monitor 出力などの生成物 |
| `dist/` | compile / export 済み生成物 |

## 標準命名

### `ml/`

`ml/` は **機能別** を正とします。

```text
ml/
├── common/
├── data/
├── training/
├── evaluation/
├── registry/
├── serving/
├── streaming/
└── experiment/
```

- `wandb` は将来的には `experiment` に寄せる想定です
- `adapters` `ports` `core` は `ml/` 直下ではなく、必要なら各機能ディレクトリの内側で使うのが標準です

### `pipeline/`

`pipeline/` は **`<verb>_job`** を正とします。

```text
pipeline/
├── data_job/
├── training_job/
├── evaluation_job/
├── batch_serving_job/
└── workflow/
```

- `train_job` は `training_job` に寄せる
- `predict_job` は `batch_serving_job` に寄せる
- `seed_job` は `data_job` に寄せる
- Phase 3 のレイヤー構成は `pipeline/*_job/` の内側に閉じ込めるのが標準です

### `scripts/`

`scripts/` は Phase 4-7 で確立した構成を正本にします。

```text
scripts/
├── _common.py
├── setup/
├── deploy/
├── ops/
├── ci/
├── sql/
├── bqml/
├── enrichment/
├── dev/                    # deprecated wrapper
└── local/                  # deprecated wrapper
```

- `Make target = 公開 API`
- `scripts/setup` `scripts/deploy` `scripts/ops` = canonical
- `scripts/dev` `scripts/local` は移行期間中だけ残る互換 wrapper
- `core.py` は将来的に `_common.py` へ寄せる対象です

### `docs/`

`docs/` の基本番号は固定です。

```text
docs/
├── 01_仕様と設計.md
├── 02_移行ロードマップ.md
├── 03_実装カタログ.md
├── 04_運用.md
├── architecture/
├── decisions/
└── archive/
```

- `01` `03` `04` は全 Phase の中核 docs
- `02` は前 Phase からの差分説明が必要な Phase のみ
- 一時的な refactor plan は完了後 `archive/` へ移すのが標準です

### `tests/`

標準は次の三層です。

```text
tests/
├── unit/
├── integration/
└── e2e/
```

- package 直下の `app/tests` `common/tests` は教材としては例外形です
- 可能なら `tests/` 集約に寄せるのが標準です

## Phase ごとの差分

### Phase 1

パス: `1/study-ml-foundations`

特徴:
- `app/` と `common/` を持たない
- ML / pipeline 基礎を主題にしている
- `ml/wandb` と `scripts/core.py` という初期形が残る

主要フォルダ:

```text
1/study-ml-foundations/

├── pipeline/
├── scripts/
├── infra/
├── env/
├── docs/
├── tests/
├── artifacts/
└── logs/
```

注目点:
- `ml/common` `ml/data` `ml/training` `ml/evaluation` `ml/registry`
- `pipeline/data_job` `pipeline/training_job` `pipeline/evaluation_job` `pipeline/batch_serving_job`
- `scripts/core.py` は後続 phase の `_common.py` に相当する初期形


## 教材として見るときの読み替え

### 「存在しない」は欠陥ではなく差分

- Phase 1 に `app/` がないのは、まだ API を扱わないから
- Phase 5 以前に `scripts/bqml/` がないのは、まだ BQML を導入していないから
- Phase 7 に `infra/manifests/` があるのは、GKE / KServe を扱うから

### 今後そろえると効果が大きい項目

1. `pipeline/` の job 命名を `<verb>_job` に統一
2. `ml/` を機能別で統一し、`adapters` `ports` `core` は内側へ寄せる
3. 各 Phase の `docs/01_仕様と設計.md` `03_実装カタログ.md` `04_運用.md` を必須の基本セットとして維持
