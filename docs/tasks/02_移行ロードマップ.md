## 0. 最重要指示（READ FIRST）

> **本リポの「移行」には 2 つの軸がある**:
>
> 1. **Phase 派生メタルール** (引き算戦略) — Phase 7 を正本コードとして、後方 Phase を機械的に派生させるためのルール。**正本 = 親 [`README.md` §「基本戦略」](../README.md)**
> 2. **検索アプリの仕様追従ロードマップ** (Wave 1-3) — 最新仕様 (Vertex Vector Search / Feature Store / Composer) に Phase 7 のコードを段階的に追従させる Strangler 計画。**正本 = Phase 7 [`docs/02_移行ロードマップ.md`](../7/study-hybrid-search-gke/docs/02_移行ロードマップ.md)**
>
> どちらを参照すべきかは「Phase コードを派生したい」 vs 「Phase 7 のコードを最新仕様に追従させたい」の意図で分かれる。

## 1. 引き算戦略 (Phase 派生メタルール)

詳細は親 [`README.md` §「基本戦略」](../README.md)。要点:

- **Phase 7 (`7/study-hybrid-search-gke/`) を最終形・正本コードとする** (起点)
- Phase 7 のコードを起点に、後方 Phase (6 → 5 → 4 → 3 → 2 → 1) へ向かって **引き算で派生コードを生成する**
- 引き算する技術領域は親 README §「引き算の対応関係」表で定義
- **教材対象外技術 (Agent Builder / Vizier / Model Garden / Gemini RAG / W&B / Looker Studio / Doppler) の固有名は親 [`README.md` §1 教材対象外](../README.md) の 1 箇所にのみ記載**。それ以外の docs / コード / コメント / テスト / 履歴で言及禁止
- コード修正は **3 ステップ手順 (`_old` 退避 → 一つ前 Phase をコピー → 引き算修正)** を厳守 (詳細は親 [`README.md` §「基本戦略」](../README.md))

## 2. Wave 1-3 実装ロードマップ (Phase 7 の仕様追従)

Phase 7 のコードを最新仕様 (Vertex Vector Search / Feature Store / Cloud Composer / 実案件 reference architecture) へ追従させるための具体実装計画は **Phase 7 [`docs/02_移行ロードマップ.md`](../7/study-hybrid-search-gke/docs/02_移行ロードマップ.md) が正本**。

サマリ:

- **Wave 1** (検索アプリ層、ローカル完結) — PR-1 SemanticSearch Port + Vertex Vector Search adapter / PR-2 FeatureFetcher Port + FOS adapter / PR-3 VectorSearchWriter Port + embed pipeline 二重書き / PR-4 KServe Reranker × FOS opt-in。**M-Local 達成 (2026-05-01)**
- **Wave 2** (GCP インフラ層) — Vector Search Terraform module / FOS default 切替 / IAM / Composer 継承確認 / deploy 配線 / default flip
- **Wave 3** (docs 整合) — reference architecture / 実装カタログ / 運用 docs を Wave 1-2 完了に追従

詳細とチェックリストは Phase 7 docs/02 を参照。

## 3. 不変制約 (引き算しても残すこと、Phase 横断)

詳細は Phase 7 [`docs/01_仕様と設計.md` §2.3 不変ルール (Phase 3-7 共通)](../7/study-hybrid-search-gke/docs/01_仕様と設計.md):
