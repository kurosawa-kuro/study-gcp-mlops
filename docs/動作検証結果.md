## 事前準備: 競合ポート自動解放の仕組みを追加

host:5432 / host:8000 を他プロジェクトの docker コンテナ（例: `real-estate-postgres`）
が掴んでいると Phase 1 の postgres/api が起動できず、毎回手動で `docker stop` する
必要があった。検証速度を上げるため、`make` 経由で自動停止する仕組みを入れた。

- `scripts/core.py`:
  - 定数 `HOST_PORTS = (5432, 8000)` と `OWN_CONTAINER_PREFIX = "ml-pipeline-"` を追加
  - `free_host_ports()`: 指定ポートを publish している非プロジェクトコンテナを `docker stop`
  - `compose()` を拡張: `up` / `run` サブコマンド時に自動呼び出し
- `Makefile`: 標準ターゲット `free-ports` を追加（単体実行も可能）

これで `make seed` / `make train` / `make serve` は実行前に競合ポートを無条件で解放する。

## 背景

Phase 3 の `make verify-pipeline` は Docker サービス起動と host port (`8000`, `5432`, `5050`, `7700`, `6379`) を使用するため、
Phase 2 の `make serve` 稼働中に実行すると衝突リスクがある。

このため、`verify-pipeline` 単体で競合解消から検証まで自己完結できるように Make ターゲットを更新した。

## 変更内容（Phase 3）

変更ファイル: `3/study-hybrid-search-local/Makefile`

- `verify-pipeline` を以下の順で実行するよう更新
  1. `make up-clean`（`ports-free` で競合コンテナを停止してから compose 起動）
  2. `make ops-bootstrap`（migration + seed + sync + embed + train 準備）
  3. 既存の検証フロー（`check-layers`, `ops-*`, `eval-*`）

これにより、従来必要だった手動手順
(`phase2 停止 -> phase3 up -> phase3 ops-bootstrap -> phase3 verify-pipeline`) を
`make verify-pipeline` に集約した。
