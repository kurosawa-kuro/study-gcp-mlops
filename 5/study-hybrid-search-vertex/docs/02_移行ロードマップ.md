# 02_移行ロードマップ

本ドキュメントは Phase 5 (`study-hybrid-search-vertex`) の決定的仕様を管理する。Phase 5 は **Phase 6 / 7 の完成形から PMLE 追加技術と GKE serving 差分を引いた、Vertex 標準 MLOps 学習フェーズ** とする。

目的は、Phase 4 の GCP serverless 構成を土台に、**Vertex AI Pipelines / Endpoints / Model Registry / Feature Store + Feature Group + Feature Online Store(必須)/ Vertex Vector Search(必須)/ Model Monitoring** を実コードへ統合し、MLOps の標準責務をどこへ移すかを学ぶこと。Feature Store と Vertex Vector Search は training-serving skew 防止と semantic 正本ストアという本番 MLOps 中核の責務を担う。

## 0. 不変ルール

- 題材は不動産ハイブリッド検索のまま維持する
- 検索中核は `Meilisearch + Vertex Vector Search + multilingual-e5 + RRF + LightGBM LambdaRank`(Phase 4 で使っていた BigQuery `VECTOR_SEARCH` 経路は Phase 5 で Vertex Vector Search に置換)
- `/search` の既定挙動は検索 API として成立させる
- IaC は Terraform を正とする
- Phase 6 の PMLE 追加技術や Phase 7 の GKE serving 差分は持ち込まない

## 1. 学習主眼

Phase 5 で新しく吸収させるのは次の Vertex 標準機能。

| 技術 | 学ぶ観点 |
|---|---|
| Vertex AI Pipelines | 学習・埋め込みの DAG 管理 |
| Vertex AI Endpoints | encoder / reranker 推論の外出し |
| **Vertex Vector Search** | **ME5 ベクトルの正本ストア + serving 時の ANN 検索 (Phase 4 の BQ `VECTOR_SEARCH` を置換)** |
| Model Registry | モデル版管理と昇格 |
| **Vertex AI Feature Store + Feature Group + Feature Online Store(必須)** | **Phase 4 で作った BQ feature table / view を入力源に、training と serving で同一 feature を取り出す経路を確立する。training-serving skew を防ぐ本番 MLOps の中核部品** |
| Model Monitoring | 推論監視と drift 検知 |

## 2. 対象外

Phase 5 では以下を扱わない。

- BQML
- Dataflow
- Monitoring SLO
- Explainable AI
- GKE / KServe / Gateway API

加えて、親リポ `README.md` §1 / `docs/02_移行ロードマップ.md §1.4` の全 Phase 共通禁止技術 (Agent Builder / Discovery Engine / Gemini RAG / Model Garden / Vertex AI Vizier / W&B / Looker Studio / Doppler) は本 Phase でも一切扱わない。

> **Vertex Vector Search の扱い (2026-05-01 改定)**: 旧版では本 Phase でも扱わない技術として列挙していたが、実案件想定 (ME5 ベクトルは Vertex Vector Search に格納するのが定石) に合わせ、Phase 5 の semantic 検索の正本ストアとして採用する。ME5 ベクトルの永続化先と `/search` の semantic 経路は Vertex Vector Search に切替える(Phase 4 の BQ `VECTOR_SEARCH` 経路は据え置かない)。本変更は仕様 (本 docs / `01_仕様と設計.md` / `CLAUDE.md` / `README.md`) に先行反映され、実装 (`03_実装カタログ.md` / `04_運用.md` / コード) は別 PR で追従する。

## 3. 実装方針

- Phase 4 の検索コアは維持し、MLOps の実行責務だけを Vertex 標準へ移す
- `search-api` は重い推論を内部保持せず、Vertex Endpoint 呼び出しへ寄せる
- 学習オーケストレーションは Vertex AI Pipelines へ寄せる
- モデル管理は GCS 単独ではなく Model Registry を正本に含める
- 監視は Cloud Monitoring に加えて Model Monitoring を導入する

## 4. 修正方針

### 4.1 コード

- Phase 5 の役割を超える PMLE 追加機能は入れない
- `search-api` は Vertex Endpoint proxy として整理する
- Feature parity は維持する

### 4.2 文書

- README / CLAUDE / `docs/01_仕様と設計.md` は Vertex 標準 MLOps を主題に統一する
- Phase 6 以降の技術は「次 Phase で追加するもの」としてのみ触れる
- 教育資料は Pipelines / Endpoint / Registry / Monitoring を中心に整理する

### 4.3 検証方針

- 移行中は差分を意識しつつ、コード修正の速度を優先する
- E2E や CI/CD 上の総合検証は後段へ回す
- ローカルで回せる単体テスト、限定的な lint、局所的な確認を優先する
- 既存の lint / test 負債は、今回の移行差分と無関係なら別件として切り分ける
