# 01_仕様と設計

> 決定的仕様は `docs/02_移行ロードマップ.md` を正とする。本ファイルは Phase 5 の実装配置を説明する。

Phase 5 の主目的は、Phase 4 の不動産ハイブリッド検索を維持したまま、**Vertex 標準 MLOps の責務を実コードへ移す** こと。

## 0. 最重要ルール

- 不変なのは題材と検索中核
- 検索中核は `Meilisearch + Vertex Vector Search + multilingual-e5 + RRF + LightGBM LambdaRank`(Phase 4 の BQ `VECTOR_SEARCH` 経路は Phase 5 で Vertex Vector Search に置換)
- 変えるのは学習、推論配備、モデル管理、監視の責務分担、および ME5 ベクトルの格納先

## 1. Phase 5 で追加するもの

| 技術 | 配置 |
|---|---|
| Vertex AI Pipelines | `pipeline/` |
| Vertex AI Endpoints | `ml/serving/` と `app/` の endpoint client |
| **Vertex Vector Search** | **ME5 ベクトル検索の本番 serving index (ANN)。`pipeline/` の embed パイプラインが BigQuery 側 embedding テーブルから upsert、`app/` の semantic 検索 adapter から match endpoint を呼ぶ。embedding 生成履歴・メタデータの正本は BigQuery 側に残す** |
| Model Registry | 学習結果の登録経路 |
| **Vertex AI Feature Store + Feature Group + Feature Online Store(必須)** | **Phase 4 で作った BQ feature table / view を入力源に、`pipeline/` (training) と `app/` (serving) の双方が同一 feature を取り出す。`infra/terraform/modules/vertex/` で Feature Store / Feature Group を、Feature Online Store + FeatureView を定義** |
| Model Monitoring | 推論 drift 監視 |

## 2. 中核アーキテクチャ

1. Dataform で `property_features_daily` を生成する
2. Vertex AI Pipelines で埋め込み更新と学習を回す
3. `search-api` は Vertex Endpoint を呼びつつ検索をオーケストレーションする
4. Model Registry と `training_runs` でモデル履歴を持つ
5. Model Monitoring で drift を監視する

## 3. 主要コンポーネント

### 3.1 `search-api`

- `app/` が Cloud Run 上の API
- lexical retrieval は Meilisearch
- semantic retrieval は **Vertex Vector Search** の match endpoint (Phase 4 の BQ `VECTOR_SEARCH` 経路は撤去)
- rerank は Vertex Endpoint へ委譲する

### 3.2 `pipeline/`

- 埋め込み更新 pipeline
- 学習 pipeline
- Phase 4 の job 主導から Vertex AI Pipelines 主導へ移す

### 3.3 学習とモデル管理

- 学習本体は LightGBM LambdaRank
- モデル成果物は GCS と `training_runs` に残す
- あわせて Model Registry で版管理する

### 3.4 監視

- Cloud Monitoring に加えて Model Monitoring を使う
- feature parity は継続して守る

## 4. 実案件想定の reference architecture

Phase 5 が学習教材として狙っている **「Vertex 標準 MLOps を本番運用するときの理想的なハイブリッド検索構成」** を示す。本リポの実装はこの構成に Port/Adapter の差し替えで到達できることを設計上の目標にする。

```
[ユーザー]
   │
   │  検索キーワード入力
   ▼
[EC / 検索 API]
   │
   ├──────────────────────────────────────────┐
   │                                          │
   ▼ ① 系統: ベクトル候補取得                 ▼ ② 系統: 文字列一致候補取得
[multilingual-e5 でベクトル化]              [Elasticsearch + Redis 同義語辞書]
   │                                          │   (例: 「マンション」→「アパート / 共同住宅」
   ▼                                          │    のような query expansion 用辞書)
[Vertex AI Vector Search]                     │
   │                                          ▼
   │                                       [一致候補]
   └──────────────────────────────────────────┘
                  │
                  ▼
            [候補集合を統合 (RRF)]
                  │
                  ▼
            [LightGBM で再順位付け]
                  │
                  ▼
            [ユーザーに表示]
```

| レイヤー | 実案件想定 | 本リポの実装 (学習用 substitute) | 差し替え |
|---|---|---|---|
| Lexical retrieval | **Elasticsearch** | **Meilisearch** | `LexicalRetriever` Port の `MeilisearchAdapter` を `ElasticsearchAdapter` に差し替えれば移行可能 |
| Lexical 補助 (query expansion) | **Redis 同義語辞書** (「マンション」→「アパート / 共同住宅」のような言い換え) | Redis (結果キャッシュ + rate limit) | 同義語辞書は本リポでは未実装。Port 化して `SynonymExpander` adapter を後付けする想定 |
| Semantic encoder | **multilingual-e5** | multilingual-e5 ✓ | 同一 |
| Semantic vector store (serving index) | **Vertex AI Vector Search** (本番 serving index、source は BigQuery 側 embedding テーブル) | Vertex AI Vector Search ✓ (Phase 4 の BQ `VECTOR_SEARCH` から Phase 5 で置換) | 同一 |
| 候補融合 | **RRF** | RRF ✓ | 同一 |
| 再順位付け | **LightGBM (LambdaRank)** | LightGBM LambdaRank ✓ | 同一 |
| 特徴量管理 | **Vertex AI Feature Store + Feature Online Store** (training-serving skew 防止) | 同一 ✓ | 同一 |

**学習教材として Meilisearch を据え置く理由**: 親 [`README.md` §「検索エンジン(Phase 3 以降): Meilisearch」](../../../README.md) を参照。Elasticsearch は本番環境向けにチューニング項目が多く、学習関心事 (Port/Adapter / semantic 統合 / RRF / rerank / Feature Store) から外れるため、Phase 5 の実装は Meilisearch のまま据え置く。Port/Adapter で隔離してあり、`MeilisearchAdapter` → `ElasticsearchAdapter` への差し替えで上記 reference architecture に到達できる構造を維持する。

**Redis 同義語辞書を docs に明記する理由**: 本リポでは Redis を結果キャッシュ用途で使っているが、実案件のハイブリッド検索では query expansion 用の同義語辞書が lexical retrieval の精度を支える load-bearing なコンポーネントになる。実装はしないが、**設計の認識として欠かさない** ために本セクションで言及する。
