## 0. 位置付け (Phase 5: Vertex AI MLOps 化の技術境界 = 論理 Phase)

Phase 5 は **論理 Phase** (個別コード保守なし)。Phase 4 (GCP MLOps 土台) を出発点に、**Vertex AI 本番 MLOps 基盤への移行で何が増えるか** を Phase 7 完成版コードを参照しながら資料で理解する。

### 0.1 論理 Phase であることの意味

```text
Phase 5 は独立コードを保守しない。
Phase 7 完成版コードを基準に、Vertex AI MLOps 化の技術境界を理解するための論理 Phase とする。
```

Phase 4 → Phase 5 の主な技術境界:

- **Vertex AI Pipelines (KFP v2)** によるパイプライン管理 — Phase 4 の Cloud Run Jobs から発展
- **Vertex AI Endpoints** による推論配備 — Phase 4 の Cloud Run inference から発展
- **Vertex Vector Search** による ME5 ベクトル検索の本番 serving index — Phase 4 の BigQuery `VECTOR_SEARCH` から発展
- **Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store)** による training-serving skew 防止 — Phase 4 の BigQuery feature table / view を入力源として発展
- **Vertex Model Registry** による昇格運用 (`production` / `staging` alias) — Phase 4 の GCS から発展
- **Vertex AI Model Monitoring v2** による drift 監視 — Phase 4 の Cloud Monitoring から発展
- orchestration は Phase 4 の Cloud Scheduler + Eventarc + Cloud Function 軽量経路を継続 (Composer 本線化は Phase 6 論理境界 + Phase 7 本実装)

### 0.2 参照する Phase 7 完成版コード

Phase 5 で説明する技術要素はすべて Phase 7 配下に本実装されている:

## 1. Phase 4 → Phase 5 の差分 (技術境界の理解)

> **論理 Phase の説明方針**: 以下の差分は **Phase 7 完成版コードに本実装されている** ものを Phase 5 の技術境界として整理した表。Phase 5 配下に独立コードはない。

| 観点 | Phase 4 (GCP MLOps 土台、コード化) | Phase 5 で増える技術 (Phase 7 配下に本実装) |
|---|---|---|
| 学習 pipeline | Cloud Run Jobs | **Vertex AI Pipelines (KFP v2)** (`7/.../pipeline/`) |
| 推論配備 | Cloud Run inference | **Vertex AI Endpoints** (Phase 7 では KServe に差し替え済み) |
| ベクトル検索 | BigQuery `VECTOR_SEARCH` | **Vertex Vector Search** (ME5 ベクトル検索の本番 serving index、ANN。`pipeline/` の embed パイプラインが BigQuery 側 embedding テーブルから upsert、`app/` semantic adapter が match endpoint を呼ぶ。embedding 生成履歴・メタデータの正本は BigQuery 側に残す) |
| 特徴量管理 | BigQuery feature table / view | **Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store)** — Phase 4 の BQ feature table / view を入力源に、`pipeline/` (training) と `app/` (serving) の双方が同一 feature を取り出す。Online Store を使う実務では **Feature View が serving 接続点** |

中核 5 要素 (Meilisearch BM25 / multilingual-e5 / ベクトルストア (Phase 4 = BQ `VECTOR_SEARCH` / Phase 7 完成版 = Vertex Vector Search) / RRF / LightGBM LambdaRank) は Phase 3-7 で不変。**Phase 5 の技術境界として変わるのは ML 工程の責務分担と ME5 ベクトル / 特徴量の格納先**。

### 2.1 主要コンポーネント (Phase 7 完成版コード上で理解)

- **search-api** (`7/.../app/`): Phase 7 では GKE Deployment 上の API。lexical = Meilisearch、semantic = **Vertex Vector Search の match endpoint**、rerank = KServe (Phase 5 の論理 Phase 上では Vertex Endpoint と読み替え)、feature 取得 = Feature Online Store (Feature View 経由)

- **監視**: Cloud Monitoring + **Vertex Model Monitoring v2** (drift 監視)

## 3. Phase 5 で理解する重要技術 (Phase 7 配下に本実装)

### 3.1 Vertex AI Feature Store (training-serving skew 防止)

実務では **Feature View が serving 接続点**:

- `7/.../infra/terraform/modules/vertex/` で Feature Store / Feature Group + Feature Online Store + Feature View を定義
- `7/.../pipeline/` (training) は Feature Group の specs に従って materialize された特徴量を読む
- `7/.../app/` (serving) は Feature View 経由で online fetch (regional public endpoint)
- 入力源は Phase 4 で構築した BQ feature table / view (Phase 4 が前提)

### 3.2 Vertex Vector Search (ME5 ベクトル検索の本番 serving index)

2026-05-01 改定で禁止リストから外し、Phase 7 完成版コードで本番採用:

- ME5 で encode したクエリベクトルを `find_neighbors` API に投げて ANN 検索
- index への upsert は embed pipeline が BQ embedding テーブルから実行
- **embedding 生成履歴・メタデータの正本は BigQuery 側に残す** (data lake / serving index の二層構造)
- Phase 4 の BigQuery `VECTOR_SEARCH` 経路は Phase 5 の技術境界で Vertex Vector Search へ発展 (Phase 7 完成版コードで本実装)

## 4. 実案件想定の reference architecture

> 本セクションは **Phase 横断で参照される正本**。本リポの実装はこの構成に Port/Adapter の差し替えで到達できることを設計目標にする。親 README §「検索エンジン(Phase 3 以降): Meilisearch」と Phase 7 docs/01 §2.7 から本セクションが参照される。

```
[ユーザー]
   │
   │  検索キーワード入力
   ▼
[EC / 検索 API]
   │
   ├──────────────────────────────────────────┐
   │                                          │
   ▼ ① 系統: ベクトル候補取得                 ▼ ② 系統: 文字列一致候補取得
[multilingual-e5 でベクトル化]              [Elasticsearch + Redis 同義語辞書]
   │                                          │   (例: 「マンション」→「アパート / 共同住宅」
   ▼                                          │    のような query expansion 用辞書)
[Vertex AI Vector Search]                     │
   │                                          ▼
   │                                       [一致候補]
   └──────────────────────────────────────────┘
                  │
                  ▼
            [候補集合を統合 (RRF)]
                  │
                  ▼
            [Vertex AI Feature Store (Feature View) で fresh feature 取得]
                  │
                  ▼
            [LightGBM で再順位付け]
                  │
                  ▼
            [ユーザーに表示]
```

| レイヤー | 実案件想定 | 本リポの実装 (学習用 substitute) | 差し替え |
|---|---|---|---|
| Lexical retrieval | **Elasticsearch** | **Meilisearch** | `LexicalRetriever` Port の `MeilisearchAdapter` を `ElasticsearchAdapter` に差し替えれば移行可能 |
| Lexical 補助 (query expansion) | **Redis 同義語辞書** | Redis (結果キャッシュ + rate limit) | 同義語辞書は Port 化して `SynonymExpander` adapter 後付け想定 |
| Semantic encoder | **multilingual-e5** | multilingual-e5 ✓ | 同一 |
| Semantic vector store (serving index) | **Vertex AI Vector Search** | Vertex AI Vector Search ✓ (Phase 5+ で BQ から置換) | 同一 |
| 候補融合 | **RRF** | RRF ✓ | 同一 |
| 特徴量管理 | **Vertex AI Feature Store + Feature Online Store** | 同一 ✓ | 同一 |
| 再順位付け | **LightGBM (LambdaRank)** | LightGBM LambdaRank ✓ | 同一 |
| orchestration | **Cloud Composer / Managed Airflow** (Phase 6+ で本線昇格) | Cloud Scheduler + Eventarc + Cloud Function (Phase 5 軽量経路) → Composer (Phase 6+) | Phase 5 → 6 引き算境界 |

**Meilisearch を据え置く理由**: 親 [`README.md` §「検索エンジン(Phase 3 以降): Meilisearch」](../../../README.md) を参照。Elasticsearch は本番向けにチューニング項目が多く、学習関心事 (Port/Adapter / semantic 統合 / RRF / rerank / Feature Store) から外れるため、Phase 5 では Meilisearch を据え置く。

**Redis 同義語辞書を docs に明記する理由**: 本リポでは Redis を結果キャッシュ用途で使うが、実案件のハイブリッド検索では query expansion 用の同義語辞書が lexical retrieval の精度を支える load-bearing なコンポーネント。実装はしないが、設計の認識として欠かさない。
