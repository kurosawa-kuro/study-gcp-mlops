## 0. 位置付け (引き算戦略上の Phase 5)

Phase 5 は親 [`README.md` §4](../../../README.md) の引き算チェーンで **Phase 6 から PMLE 追加技術 (BQML / Dataflow / TreeSHAP / Monitoring SLO + burn-rate / Cloud Composer 本線昇格 / Composer-managed BQ monitoring query) を引き算した結果**。題材は Phase 4 から継承して不動産ハイブリッド検索を維持しつつ、**Vertex AI 本番 MLOps 基盤** を実コードへ移す phase。

**Phase 4 → Phase 5 の核心**: ML 工程 (学習 / 推論配備 / モデル管理 / 監視) を Vertex AI に寄せる + **Vertex Vector Search を ME5 ベクトル検索の本番 serving index に採用** + **Vertex AI Feature Store を training-serving skew 防止のため必須化**。orchestration は Phase 4 の Cloud Scheduler + Eventarc + Cloud Function 軽量経路をそのまま継続 (Composer 本線化は Phase 6、= Phase 5 → 6 引き算境界)。

## 1. Phase 4 → Phase 5 の差分

| 観点 | Phase 4 (継承元) | Phase 5 (追加 / 置換) |
|---|---|---|
| 学習 pipeline | Cloud Run Jobs | **Vertex AI Pipelines (KFP v2)** (`pipeline/`) |
| 推論配備 | Cloud Run inference | **Vertex AI Endpoints** (`ml/serving/` + `app/` の endpoint client) |
| ベクトル検索 | BigQuery `VECTOR_SEARCH` | **Vertex Vector Search** (ME5 ベクトル検索の本番 serving index、ANN。`pipeline/` の embed パイプラインが BigQuery 側 embedding テーブルから upsert、`app/` semantic adapter が match endpoint を呼ぶ。embedding 生成履歴・メタデータの正本は BigQuery 側に残す) |
| 特徴量管理 | BigQuery feature table / view | **Vertex AI Feature Store (Feature Group / Feature View / Feature Online Store) (必須)** — Phase 4 の BQ feature table / view を入力源に、`pipeline/` (training) と `app/` (serving) の双方が同一 feature を取り出す。Online Store を使う実務では **Feature View が serving 接続点** |
| モデル管理 | GCS | **Vertex Model Registry** (昇格運用 = `production` / `staging` alias) |
| 監視 | Cloud Monitoring | Cloud Monitoring + **Vertex Model Monitoring v2** |
| orchestration | Cloud Scheduler + Eventarc + Cloud Function (軽量) | **同じ軽量経路を継続** (Composer 本線化は Phase 6 で実施 = 引き算境界) |

中核 5 要素 (Meilisearch BM25 / multilingual-e5 / **Vertex Vector Search** (Phase 5+ で BQ から置換) / RRF / LightGBM LambdaRank) は Phase 3-7 で不変。**変えるのは ML 工程の責務分担と ME5 ベクトル / 特徴量の格納先**。

### 2.1 主要コンポーネント

- **search-api** (`app/`): Cloud Run 上の API。lexical = Meilisearch、semantic = **Vertex Vector Search の match endpoint**、rerank = Vertex Endpoint へ委譲、feature 取得 = Feature Online Store (Feature View 経由)
- **pipeline/**: Vertex AI Pipelines 主導 (Phase 4 の Cloud Run Jobs 主導から移行)。embed pipeline + train pipeline
- **学習・モデル管理**: LightGBM LambdaRank → GCS + **Vertex Model Registry** で版管理
- **監視**: Cloud Monitoring + **Vertex Model Monitoring v2** (drift 監視)

## 3. Phase 5 必須要素の詳細

### 3.1 Vertex AI Feature Store (training-serving skew 防止のため必須)

実務では **Feature View が serving 接続点**:

- `infra/terraform/modules/vertex/` で Feature Store / Feature Group + Feature Online Store + Feature View を定義
- `pipeline/` (training) は Feature Group の specs に従って materialize された特徴量を読む
- `app/` (serving) は Feature View 経由で online fetch (regional public endpoint)
- 入力源は Phase 4 で構築した BQ feature table / view

### 3.2 Vertex Vector Search (ME5 ベクトル検索の本番 serving index)

2026-05-01 改定で禁止リストから外し、Phase 5 で本番採用:

- ME5 で encode したクエリベクトルを `find_neighbors` API に投げて ANN 検索
- index への upsert は embed pipeline が BQ embedding テーブルから実行
- **embedding 生成履歴・メタデータの正本は BigQuery 側に残す** (data lake / serving index の二層構造)
- Phase 4 の BigQuery `VECTOR_SEARCH` 経路は Phase 5 で撤去

## 4. 実案件想定の reference architecture

> 本セクションは **Phase 横断で参照される正本**。本リポの実装はこの構成に Port/Adapter の差し替えで到達できることを設計目標にする。親 README §「検索エンジン(Phase 3 以降): Meilisearch」と Phase 7 docs/01 §2.7 から本セクションが参照される。

```
[ユーザー]
   │
   │  検索キーワード入力
   ▼
[EC / 検索 API]
   │
   ├──────────────────────────────────────────┐
   │                                          │
   ▼ ① 系統: ベクトル候補取得                 ▼ ② 系統: 文字列一致候補取得
[multilingual-e5 でベクトル化]              [Elasticsearch + Redis 同義語辞書]
   │                                          │   (例: 「マンション」→「アパート / 共同住宅」
   ▼                                          │    のような query expansion 用辞書)
[Vertex AI Vector Search]                     │
   │                                          ▼
   │                                       [一致候補]
   └──────────────────────────────────────────┘
                  │
                  ▼
            [候補集合を統合 (RRF)]
                  │
                  ▼
            [Vertex AI Feature Store (Feature View) で fresh feature 取得]
                  │
                  ▼
            [LightGBM で再順位付け]
                  │
                  ▼
            [ユーザーに表示]
```

| レイヤー | 実案件想定 | 本リポの実装 (学習用 substitute) | 差し替え |
|---|---|---|---|
| Lexical retrieval | **Elasticsearch** | **Meilisearch** | `LexicalRetriever` Port の `MeilisearchAdapter` を `ElasticsearchAdapter` に差し替えれば移行可能 |
| Lexical 補助 (query expansion) | **Redis 同義語辞書** | Redis (結果キャッシュ + rate limit) | 同義語辞書は Port 化して `SynonymExpander` adapter 後付け想定 |
| Semantic encoder | **multilingual-e5** | multilingual-e5 ✓ | 同一 |
| Semantic vector store (serving index) | **Vertex AI Vector Search** | Vertex AI Vector Search ✓ (Phase 5+ で BQ から置換) | 同一 |
| 候補融合 | **RRF** | RRF ✓ | 同一 |
| 特徴量管理 | **Vertex AI Feature Store + Feature Online Store** | 同一 ✓ | 同一 |
| 再順位付け | **LightGBM (LambdaRank)** | LightGBM LambdaRank ✓ | 同一 |
| orchestration | **Cloud Composer / Managed Airflow** (Phase 6+ で本線昇格) | Cloud Scheduler + Eventarc + Cloud Function (Phase 5 軽量経路) → Composer (Phase 6+) | Phase 5 → 6 引き算境界 |

**Meilisearch を据え置く理由**: 親 [`README.md` §「検索エンジン(Phase 3 以降): Meilisearch」](../../../README.md) を参照。Elasticsearch は本番向けにチューニング項目が多く、学習関心事 (Port/Adapter / semantic 統合 / RRF / rerank / Feature Store) から外れるため、Phase 5 では Meilisearch を据え置く。

**Redis 同義語辞書を docs に明記する理由**: 本リポでは Redis を結果キャッシュ用途で使うが、実案件のハイブリッド検索では query expansion 用の同義語辞書が lexical retrieval の精度を支える load-bearing なコンポーネント。実装はしないが、設計の認識として欠かさない。

## 6. 不変ルール

- 題材は不動産ハイブリッド検索 (Phase 3 から継承)
- 中核 5 要素 (Meilisearch BM25 + **Vertex Vector Search** + multilingual-e5 + RRF + LightGBM LambdaRank) — Phase 4 の BQ `VECTOR_SEARCH` は本 phase で Vertex Vector Search に置換
- Vertex AI Feature Store + Vertex Vector Search は Phase 5 必須要素 (Phase 6 / 7 へそのまま継承)
- orchestration は Phase 4 軽量経路を継続 (Composer 本線化は Phase 6)
- Feature parity は崩さない (6 ファイル同 PR 原則は Phase 6 / 7 で本格化)
- Terraform を正とする
- 設計思想 (Port/Adapter / `core → ports ← adapters` / DI) は Phase 2 から継承して不変
