# ナレーション要約 (Phase 4 / Vertex 編)

簡易版。詳細版は `../02_ナレーション台本.md` を参照。

---

## [Slide 00]

> Phase 3 の hybrid-search-cloud に Vertex AI レイヤを後付けした差分プロジェクトの教材です。

## [Slide 01]

> Phase 3 までは受講済前提で、Phase 3 → 4 の差分整理と Port/Adapter 境界の効果を学びます。

## [Slide 02]

> 題材は Phase 3 と同じ不動産検索、検索コアや学習ロジックは温存し、境界の外側だけ Vertex に差し替えます。

## [Slide 03]

> 学習オーケは Cloud Run Jobs から KFP 二本へ、rerank と encoder は lifespan から Endpoint 二本へ移ります。

## [Slide 04]

> Meilisearch と BigQuery VECTOR_SEARCH と Dataform と監視 SQL は、意図的に据え置きです。

## [Slide 05]

> Vertex 側は Pipelines、Feature Store、Endpoints、Registry、Monitoring の五コンポーネントが工程を分担します。

## [Slide 06]

> search-api は encoder と booster を外出しにしたプロキシになり、メモリは 4 ギガから 2 ギガに落ちます。

## [Slide 07]

> 追加した Port は EncoderClient と RerankerClient の二つで、services の ranking は書き換えずに済みます。

## [Slide 08]

> CPR サーバ二本は AIP 規約に従った薄い FastAPI で、推論ロジック本体は Phase 3 から import します。

## [Slide 09]

> KFP 二本は、embed が三段、train が四段構成です。

## [Slide 10]

> Registry は staging と production の二段エイリアスで、カナリア後に promote スクリプトで昇格します。

## [Slide 11]

> Vertex AI Experiments と BQ `training_runs` の二重書き込みで、SQL からのアドホック集計を壊さずに Vertex 側にも lineage を残します。

## [Slide 12]

> Feature Group は 6 ファイル目の parity invariant として、property 側 7 列を FEATURE_COLS_RANKER と一対一対応させます。

## [Slide 13]

> 再学習経路は三つあり、Eventarc の subscriber が Cloud Run Job から Cloud Function に移っています。

## [Slide 14]

> サービスアカウントは 5 個から 9 個に増え、Endpoint 実行と Pipeline 起動を厳密に分離します。

## [Slide 15]

> Model Monitoring v2 は日次ドリフト検知で、アラート時はベースライン hyperparameters の `train_pipeline` を Cloud Function 経由で submit します。

## [Slide 16]

> Cloud Run Jobs や旧 model_store 系は Phase 9 / 6 で dead code になり、削除しました。

## [Slide 17]

> Endpoint deploy は Terraform では書けず、VECTOR INDEX も手動 DDL、シークレットは Secret Manager に直接投入する運用です。

## [Slide 18]

> 全体アーキは Vertex 中心で、Dataform と BQ と Meilisearch が支える運用ループに見えます。

## [Slide 19]

> デモでは変わった箇所と変わっていない箇所を両方確認するのが狙いです。

## [Slide 20]

> 差分工事の思想、5 コンポーネント、KFP 移行、dual write、CPR と Endpoint、の五点を押さえます。

## [Slide 21]

> Phase 4 は境界を守って差し替える差分プロジェクトで、次は vertex_prediction と ranking を見比べることをおすすめします。
